# Task 30.20: Text Input + Voice Output Mode

**Epic:** 30 — ACP Content Blocks
**Status:** [ ]
**Depends on:** T30.01, T30.03
**Blocks:** T30.21

## Goal

Add a third interaction mode: user types on the keyboard, agent responds with both text (chat UI) and spoken audio (TTS via voice agent). Combines the text input path with voice output.

## Context

Epic 30 delivers two modes:
1. **Text only** — no voice agent, relay-based
2. **Voice only** — voice agent handles STT + TTS, relay dual-publishes to mobile

Mode 3 combines text input (keyboard → relay → ACP) with voice output (voice agent speaks the response). The relay dual-publish infrastructure (T30.01) already ensures the voice agent receives responses on `voice-acp`. The missing piece is a voice agent "unsolicited response listener" that feeds relay-originated responses to TTS.

## Deferred

**Why deferred:** Scope reduction decision during planning. Two modes are sufficient for launch. Mode 3 adds complexity: voice agent needs TTS-only posture (no VAD/STT), mobile needs a three-state mode selector, and the voice agent needs to handle responses it didn't initiate.

**Revisit when:** User demand or UX testing indicates text-input + voice-output is a priority interaction pattern.

## Implementation notes (for future reference)

The voice agent has `AgentSession.say(text | ReadableStream<string>)` built into the LiveKit SDK. The approach:
1. Voice agent listens for unsolicited `session/update` on `voice-acp` (messages not matching an active request)
2. Creates a `ReadableStream<string>`, pushes `agent_message_chunk` text deltas
3. Calls `session.say(stream, { allowInterruptions: true })`
4. Mobile needs a three-state mode selector: text-only, voice-only, text+voice
5. Voice agent needs a TTS-only posture: connected, TTS active, mic not published
