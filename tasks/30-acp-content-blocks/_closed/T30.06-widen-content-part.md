# Task 30.06: Widen ContentPart in acp-client

**Epic:** 30 — ACP Content Blocks
**Status:** [ ]
**Depends on:** none
**Blocks:** T30.08

## Goal

Widen the `ContentPart` type in `packages/acp-client/src/types.ts` from its current minimal shape (`{ type: string; text: string }`) to a discriminated union supporting all ACP content block types. This ensures the relay and any TypeScript consumer can type-check content blocks flowing through the ACP protocol.

## Context

**Current definition** at `packages/acp-client/src/types.ts:161-164`:
```typescript
export interface ContentPart {
  type: string;
  text: string;
}
```

This only works for text content. The ACP spec defines 5 content block types (text, image, audio, resource, resource_link) plus tool call content types (content wrapper, diff, terminal).

The relay forwards ACP events transparently — it doesn't inspect `ContentPart` fields. But widening the type ensures:
1. TypeScript consumers get type safety
2. Tests can construct typed content blocks
3. Future relay-side logic (e.g., chunking in T30.13) can inspect content types

## Implementation

### 1. Define content block types (`packages/acp-client/src/types.ts`)

Replace the flat `ContentPart` with a discriminated union:

```typescript
export interface TextContentPart {
  type: "text";
  text: string;
  annotations?: Record<string, unknown>;
}

export interface ImageContentPart {
  type: "image";
  data: string;       // base64
  mimeType: string;
  uri?: string;
  annotations?: Record<string, unknown>;
}

export interface AudioContentPart {
  type: "audio";
  data: string;       // base64
  mimeType: string;
  annotations?: Record<string, unknown>;
}

export interface ResourceContentPart {
  type: "resource";
  resource: {
    uri: string;
    mimeType?: string;
    text?: string;     // text resource
    blob?: string;     // blob resource (base64)
  };
  annotations?: Record<string, unknown>;
}

export interface ResourceLinkContentPart {
  type: "resource_link";
  uri: string;
  name: string;
  mimeType?: string;
  title?: string;
  description?: string;
  size?: number;
  annotations?: Record<string, unknown>;
}

export type ContentPart =
  | TextContentPart
  | ImageContentPart
  | AudioContentPart
  | ResourceContentPart
  | ResourceLinkContentPart;
```

### 2. Define tool call content types (`packages/acp-client/src/types.ts`)

```typescript
export interface ToolCallContentItem {
  type: "content";
  content: ContentPart;
}

export interface ToolCallDiffItem {
  type: "diff";
  path: string;
  oldText?: string;
  newText: string;
}

export interface ToolCallTerminalItem {
  type: "terminal";
  terminalId: string;
}

export type ToolCallContent =
  | ToolCallContentItem
  | ToolCallDiffItem
  | ToolCallTerminalItem;
```

### 3. Update SessionUpdateParams and related types

Ensure `tool_call` and `tool_call_update` session update types reference the new `ToolCallContent` type for their `content` field.

Add explicit types for `tool_call` and `tool_call_update` if they don't exist:
- `ToolCallUpdate` with `toolCallId`, `title`, `kind`, `status`, `content`, `locations`
- `ToolCallUpdateUpdate` with `toolCallId`, `status`, `content`

### 4. Fix downstream compilation (`packages/acp-client/`)

Any code that directly accesses `contentPart.text` without narrowing needs a type guard. Check:
- `client.ts` — likely forwards params without inspecting
- `relay-bridge.ts` — forwards transparently
- Tests — may construct `ContentPart` objects

### 5. Tests (`packages/acp-client/test/`)

- Existing tests still pass with the widened type
