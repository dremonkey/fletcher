# Task 30.01: Relay Dual-Publish in Voice Mode

**Epic:** 30 — ACP Content Blocks
**Status:** [ ]
**Depends on:** none
**Blocks:** T30.10, T30.13

## Goal

Make the relay publish ACP `session/update` events to mobile on the `acp` topic regardless of which mode originated the request. Today the relay routes exclusively: voice-acp requests → voice-acp responses, relay requests → relay responses. After this task, voice-mode responses go to BOTH voice-acp (voice agent) and acp (mobile), so mobile always has the full conversation transcript.

## Context

The relay's `onUpdate()` handler at `apps/relay/src/bridge/relay-bridge.ts:196-248` receives all `session/update` events from the ACP subprocess. Routing is determined by `activeRequestSource` (line 91):

```
if (this.activeRequestSource === "voice-acp") {
  this.forwardToVoiceAgent(msg);   // exclusive today
} else {
  this.forwardToMobile(msg);
}
```

The change: when `activeRequestSource === "voice-acp"`, also call `forwardToMobile(msg)`. Use a participant check to confirm mobile is connected (avoid publishing to an empty room).

**Voice mode detection:** Check if a non-relay, non-voice-agent participant exists in the room via `room.remoteParticipants`. The relay already iterates participants for relay discovery (`relay-llm.ts:119-143`). A mobile participant will have an identity that doesn't start with `relay-` or `voice-agent-`.

**Prompt result routing:** The same dual-publish applies to `session/prompt` results (lines 244-247 for updates, but also lines 613-618 for prompt results). When voice-acp prompt completes, forward the result to BOTH voice-acp and mobile.

## Implementation

### 1. Add dual-publish to `onUpdate()` (`apps/relay/src/bridge/relay-bridge.ts`)

In the `onUpdate()` method, change the exclusive routing to inclusive:

```typescript
if (this.activeRequestSource === "voice-acp") {
  this.forwardToVoiceAgent({ jsonrpc: "2.0", method: "session/update", params });
  // Dual-publish: mobile always gets content
  this.forwardToMobile({ jsonrpc: "2.0", method: "session/update", params });
} else {
  this.forwardToMobile({ jsonrpc: "2.0", method: "session/update", params });
}
```

### 2. Dual-publish voice-acp prompt results (`apps/relay/src/bridge/relay-bridge.ts`)

In `handleVoiceAcpMessage()`, after forwarding the prompt result to voice-acp (line ~720), also forward to mobile:

```typescript
this.forwardToVoiceAgent(resultMsg);
this.forwardToMobile(resultMsg);  // mobile gets the result too
```

Apply same logic to error responses from voice-acp prompts.

### 3. Add tests (`apps/relay/src/bridge/relay-bridge.spec.ts`)

Add test cases:
- Voice-acp prompt updates are published to BOTH voice-acp AND relay topics
- Voice-acp prompt results are published to BOTH topics
- Relay (text-mode) prompts still only publish to relay topic (no change)
- When no mobile participant connected, dual-publish degrades gracefully (no error)

## Not in scope

- Dual-publish in the reverse direction (mobile prompt → voice-acp). The voice agent doesn't need mobile-originated responses for TTS in the current two-mode design.
- Payload chunking for large messages (T30.13)
