# Task 30.02: Derive Status from ACP Tool Call

**Epic:** 30 — ACP Content Blocks
**Status:** [ ]
**Depends on:** none
**Blocks:** T30.10

## Goal

Replace the StatusBar's dependency on Ganglia's custom `StatusEvent` (from `ganglia-events` topic) with ACP-native `tool_call` events from the `acp` topic. The StatusBar should display tool execution status using the ACP `tool_call` fields: `kind`, `title`, and `status`.

## Context

**Current flow:** Voice agent's EventInterceptor emits `StatusEvent` on `ganglia-events` → mobile's `_processGangliaEvent()` at `livekit_service.dart:1127-1136` parses it → `_state.currentStatus` updates → StatusBar widget renders.

Ganglia `StatusAction` values: `thinking`, `searching_files`, `reading_file`, `writing_file`, `editing_file`, `web_search`, `executing_command`, `analyzing`.

**ACP tool_call** provides equivalent information:
- `kind`: `read`, `edit`, `delete`, `move`, `search`, `execute`, `think`, `fetch`, `other`
- `title`: "Reading configuration file", "Searching for imports", etc.
- `status`: `pending`, `in_progress`, `completed`, `failed`

The ACP `title` field is richer than Ganglia's generic action names — it describes the specific operation. The StatusBar should prefer `title` when available, falling back to a display label derived from `kind`.

**Existing parser:** `AcpUpdateParser` at `apps/mobile/lib/services/relay/acp_update_parser.dart` already handles `tool_call` and `tool_call_update` and emits `AcpToolCallUpdate`. The `RelayChatService` receives this as `RelayToolCallEvent`. This task adds status extraction from those existing events.

## Implementation

### 1. Extract status from AcpToolCallUpdate (`apps/mobile/lib/services/relay/acp_update_parser.dart`)

Ensure `AcpToolCallUpdate` carries `kind`, `title`, and `status` fields from the ACP `tool_call` event. These may already be parsed — verify and add if missing.

### 2. Map tool_call to StatusBar state (`apps/mobile/lib/services/livekit_service.dart` or appropriate service)

When `RelayToolCallEvent` arrives with a `tool_call`:
- Extract `title` and `kind`
- Set `_state.currentStatus` using `title` (human-readable) or map `kind` to a display label
- When `status === "completed"` or `status === "failed"`, clear status after a short delay (match current 5s auto-clear)
- `tool_call_update` with `status: "in_progress"` keeps the status visible

Kind → display label mapping:
```
read    → "Reading"
edit    → "Editing"
search  → "Searching"
execute → "Running"
think   → "Thinking"
fetch   → "Fetching"
delete  → "Deleting"
move    → "Moving"
other   → (use title)
```

### 3. Update StatusBar widget if needed (`apps/mobile/lib/widgets/`)

The StatusBar widget may need adjustment if it currently expects `StatusAction` enum values. Update to accept the new status shape (string-based kind + title).

### 4. Tests

- Parser test: `tool_call` event emits `AcpToolCallUpdate` with kind, title, status
- Status mapping test: kind → display label
- Auto-clear test: status clears after completion
