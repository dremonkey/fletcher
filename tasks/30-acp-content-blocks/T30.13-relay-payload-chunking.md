# Task 30.13: Relay Payload Chunking

**Epic:** 30 — ACP Content Blocks
**Status:** [ ]
**Depends on:** T30.01
**Blocks:** T30.14, T30.15

## Goal

Add payload chunking to the relay's `forwardToMobile()` and `forwardToVoiceAgent()` methods so that ACP content blocks exceeding the ~15KB LiveKit data channel limit (base64 images, large text resources) are split into reassemblable chunks. Port the proven chunking pattern from Ganglia's `EventInterceptor` to the relay. Add corresponding reassembly on the mobile side for the relay topic.

## Context

**Current behavior:** The relay silently drops payloads >15KB at `relay-bridge.ts:755,804`. This works today because ACP text content is small. Phase 4 introduces images (easily >15KB in base64) that need reliable delivery.

**Existing chunking pattern** in `event-interceptor.ts:107-148`:
- Max chunk size: 14KB (line 18)
- Payload serialized to JSON → base64 encoded
- Split into chunks with metadata: `{ type: "chunk", transferId, chunkIndex, totalChunks, data }`
- Receiver reassembles by `transferId`, triggers processing when all chunks arrive

**Mobile already has chunk reassembly** for ganglia-events at `livekit_service.dart:1087-1122`:
- `_handleChunk()` buffers chunks by `transferId`
- Concatenates base64 data when all chunks received
- Decodes and processes the complete message

The relay topic needs the same pattern. The relay chunks outbound messages; mobile reassembles inbound chunks on the relay topic.

## Implementation

### 1. Add chunking utility (`apps/relay/src/bridge/chunk.ts`)

Extract the chunking logic into a reusable function:

```typescript
interface ChunkOptions {
  maxChunkSize?: number;  // default 14_000 bytes
}

function chunkPayload(payload: string, options?: ChunkOptions): string[] | null;
// Returns null if payload fits in one message, or array of chunk JSON strings
```

Each chunk:
```json
{
  "type": "chunk",
  "transferId": "uuid",
  "chunkIndex": 0,
  "totalChunks": 3,
  "data": "base64..."
}
```

### 2. Integrate chunking into forwardToMobile (`apps/relay/src/bridge/relay-bridge.ts`)

In `forwardToMobile()`, replace the payload-size drop with chunking:

```typescript
const payload = JSON.stringify(msg);
if (payload.length > MAX_DATA_CHANNEL_SIZE) {
  const chunks = chunkPayload(payload);
  for (const chunk of chunks) {
    await this.roomManager.sendToRoom(roomName, JSON.parse(chunk));
  }
  return;
}
// Normal single-message publish
```

Apply same to `forwardToVoiceAgent()`.

### 3. Add reassembly on mobile (`apps/mobile/lib/services/relay/`)

Add chunk reassembly to `RelayChatService` or a shared utility. The pattern matches `_handleChunk()` in `livekit_service.dart`:
- Buffer chunks by `transferId`
- When all chunks received, concatenate, base64-decode, parse JSON
- Feed reassembled message to the normal ACP update handler

### 4. Stale transfer timeout

Add a timeout (10 seconds) for incomplete transfers. If not all chunks arrive within the window, discard the partial transfer and log a warning. This prevents memory leaks from dropped chunks.

### 5. Tests

**Relay-side:**
- Payload under limit → single message (no chunking)
- Payload over limit → chunked into correct number of pieces
- Each chunk has correct transferId, index, totalChunks
- Round-trip: chunk → reassemble → original payload

**Mobile-side:**
- Chunks arrive in order → reassembles correctly
- Chunks arrive out of order → reassembles correctly
- Incomplete transfer → times out and discards
- Single-message payload → passes through unchanged

## Not in scope

- Changing the data channel transport layer
- Compression (chunking alone is sufficient for Phase 4 image sizes)
