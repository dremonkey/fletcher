# T29.02 — Determine ACP Mode Flags

**Status:** Complete [x]
**EPIC:** [29-relay-claude-code](./EPIC.md)
**PRD Requirement:** FR2
**Depends on:** T29.01

## Objective
Identify the exact CLI flags required to launch Claude Code in ACP stdio mode and validate that the subprocess accepts JSON-RPC on stdin.

## Key Findings
- Claude Code CLI has **NO `--acp` flag**. GitHub issue #6686 requesting it was closed without implementation.
- Using `@zed-industries/claude-agent-acp` instead — a ready-made ACP adapter wrapping `@anthropic-ai/claude-agent-sdk`
- Binary: `claude-agent-acp` (no flags needed — it is a pure stdio ACP server)
- `ACP_COMMAND=claude-agent-acp ACP_ARGS=""`
- Auth: `ANTHROPIC_API_KEY` env var (NOT `CLAUDE_CODE_OAUTH_TOKEN`)
- `agentInfo.name` = `"@zed-industries/claude-agent-acp"`

## Details
- No flags needed: the `claude-agent-acp` binary is inherently an ACP stdio server
- Subprocess spawns cleanly with zero arguments
- Accepts JSON-RPC 2.0 `initialize` on stdin and returns valid `agentInfo` + `agentCapabilities`
- The `ACP_ARGS` env var should be empty (or unset) when using this backend

## Acceptance Criteria
- [x] ACP mode approach documented (no flags — dedicated adapter binary)
- [x] Subprocess spawns without error
- [x] Subprocess accepts a JSON-RPC `initialize` request on stdin and returns a valid response on stdout
- [x] Configuration recorded in EPIC and architecture doc

## Notes
- Original assumption was `claude --acp` or `claude code --acp` — both incorrect
- The Zed adapter removes the need for flag discovery entirely
- Pinned version: `@zed-industries/claude-agent-acp` v0.22.1
