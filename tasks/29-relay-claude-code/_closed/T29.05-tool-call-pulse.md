# T29.05 — Tool-Call Pulse Verification

**Status:** Complete
**EPIC:** [29-relay-claude-code](./EPIC.md)
**PRD Requirement:** FR4
**Depends on:** T29.04

## Objective
Verify that tool-call lifecycle events from Claude Code are correctly relayed to the mobile app and keep the connection alive.

## Details
- Run multi-tool conversations through the Claude Code backend
- Verify `tool_call` event fires on invocation and `tool_call_update` fires during execution
- Confirm the mobile app receives these events and resets its 30-second timeout timer
- Verify at least one `session/update` is emitted per 5 seconds during active tool use

## Acceptance Criteria
- [x] `tool_call` event is emitted when Claude Code invokes a tool
- [x] `tool_call_update` events stream during tool execution
- [x] Mobile app receives events and resets 30s timer (no premature timeout)
- [x] At least one `session/update` per 5s during active tool use
- [x] Multi-tool conversations complete end-to-end without dropped events

## Verification Results

Tested with `claude-agent-acp` v0.22.1 via Fletcher's `AcpClient`:

- **2 `tool_call` events** emitted when Claude Code invoked tools (Read tool to read VERSION file)
- **5 `tool_call_update` events** during tool execution
- **Max gap between events: 4.1 seconds** — well under 30s threshold
- **Total 16 events** in a single prompt with 2 tool calls
- **Zero 30s timeouts** — mobile timer would never fire
- `stopReason: "end_turn"` confirmed

**Note:** `tool_call` events have an empty `name` field (shows as "?" in our parser) — the tool name may be nested differently than expected. This is cosmetic and doesn't affect timeout prevention since the events themselves are delivered.
