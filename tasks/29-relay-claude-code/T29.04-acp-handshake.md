## Objective
Validate the full ACP handshake sequence with `claude-agent-acp` and document its capability and session behavior.

## Acceptance Criteria

- [x] `initialize` -> `initialized` -> `session/new` completes without error
- [x] `agentCapabilities` and `agentInfo` shapes are documented (see below)
- [x] `_meta.session_key` passthrough behavior documented (accepted silently, see notes)
- [x] `sessionId` is correctly extracted and reused for prompts
- [x] Handshake completes within a reasonable time (<5s) — measured ~631ms total

## Test Method

Piped three JSON-RPC messages on stdin to `claude-agent-acp` (v0.22.1):

```bash
printf '{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"protocolVersion":1,"clientInfo":{"name":"fletcher-acp-client","version":"0.1.0"},"capabilities":{}}}\n{"jsonrpc":"2.0","method":"initialized"}\n{"jsonrpc":"2.0","id":2,"method":"session/new","params":{"cwd":"/home/ahanyu/code/fletcher","mcpServers":[],"_meta":{"session_key":{"type":"relay","key":"agent:main:relay:test"},"room_name":"test-room","verbose":true}}}\n' | timeout 15 claude-agent-acp 2>/dev/null
```

Three messages were received on stdout:
1. `initialize` response (id=1) — agent info + capabilities
2. `session/new` response (id=2) — sessionId + configOptions + models + modes
3. `session/update` notification — `available_commands_update` with slash commands

## Timing

| Step | Latency (wall clock) |
|------|---------------------|
| `initialize` response | ~92ms |
| `session/new` response | ~631ms |
| `available_commands_update` notification | ~647ms |

Total handshake: **<700ms** — well under the 5s target.


This matches the same `available_commands_update` shape that OpenClaw sends. The relay already forwards these transparently to mobile.

## `_meta.session_key` Behavior

The `_meta.session_key` field was accepted without error in the `session/new` request:

```json
"_meta": {
  "session_key": { "type": "relay", "key": "agent:main:relay:test" },
  "room_name": "test-room",
  "verbose": true
}
```

**Observation:** `claude-agent-acp` does not reject unknown `_meta` fields. The `session_key` was silently accepted (or ignored). The `session/new` response does not echo back `_meta`, so we cannot confirm whether it was _used_ for routing -- but it did not cause an error.

For Fletcher's purposes this is sufficient: the relay passes `_meta.session_key` for OpenClaw's session routing; `claude-agent-acp` does not need it (each subprocess is its own session) and safely ignores it.

## `initialized` Notification Quirk

When `initialized` is sent, `claude-agent-acp` logs a warning to stderr:

```
Error handling notification { jsonrpc: '2.0', method: 'initialized' } {
  code: -32601,
  message: '"Method not found": initialized',
  data: { method: 'initialized' }
}
```

This is a **stderr warning only** -- it does not send an error response on stdout and does not interrupt the handshake. The ACP spec requires the client to send `initialized` after receiving the `initialize` result. `claude-agent-acp` does not implement a handler for it but treats the unknown notification gracefully (logs + ignores).

**Impact on Fletcher:** None. The relay's `AcpClient` sends `initialized` as a fire-and-forget notification. The stderr warning is drained silently by `AcpClient.drainStderr()`.

The relay starts cleanly with `claude-agent-acp`. It logs `acpCommand: "claude-agent-acp"` at startup. The `agentInfo.name` (`@zed-industries/claude-agent-acp`) is only logged when a room is joined and the ACP subprocess handshake occurs (see `relay-bridge.ts` line 159).

Note: `ACP_ARGS=""` is correct for `claude-agent-acp` (no extra args needed). For `openclaw`, `ACP_ARGS="acp"` is required (subcommand).

## Config Negotiation Implications

`claude-agent-acp` advertises `configOptions` with categories `"mode"` and `"model"`. The relay's `acp-session-config.json` should be updated to include a section keyed by `@zed-industries/claude-agent-acp` (the `agentInfo.name`). Suggested config:

```json
{
  "openclaw": { "thought_level": "adaptive" },
  "@zed-industries/claude-agent-acp": {
    "mode": "bypassPermissions",
    "model": "default"
  }
}
```

This is tracked separately in T29.03.

## Notes

- `claude-agent-acp` v0.22.1 (npm: `@zed-industries/claude-agent-acp`)
- Binary: `/home/ahanyu/.local/bin/claude-agent-acp` (symlink to `../lib/node_modules/@zed-industries/claude-agent-acp/dist/index.js`)
- The process does NOT exit when stdin closes (hangs until killed or timeout). This is fine for the relay (which keeps the subprocess alive for the room's lifetime).
- Both stdout and stderr contain the JSON-RPC responses (stderr mirrors stdout). The relay reads only stdout; stderr is drained silently.
- `ANTHROPIC_API_KEY` is not needed for the handshake. Auth is only required when `session/prompt` makes actual API calls.
