### BUG-002: Idle Timer Premature Start during TTS Playout (MEDIUM)

**First seen:** 18:40 UTC (Tester report)
**Frequency:** 100%
**Impact:** The agent's idle timeout timer (which triggers the "Sleep" cycle) appears to be starting as soon as the LLM finishes generating text, rather than waiting for the TTS engine to finish speaking. In long storytelling sessions, the agent could potentially "time out" and disconnect while it is still in the middle of talking to the user.

**Analysis:**
- Current implementation in `agent.ts` resets the idle timer on user activity (speech/text).
- It does not appear to pause or reset the timer based on the agent's own speech state (`AgentState.speaking`).
- If a response is long, the 5-minute idle window could be significantly consumed by the agent's own output.

**Proposed Fix:**
- Modify the `IdleTimeout` logic to either:
    1. Reset the timer whenever the agent starts speaking.
    2. Pause the timer during the `AgentState.speaking` state and resume only when it returns to `listening`.

**Task:** [`tasks/02-livekit-agent/032-idle-timer-tts-aware.md`](../../tasks/02-livekit-agent/032-idle-timer-tts-aware.md)

**Additional field observation (22:57–22:58 UTC):** Timer expired mid-response. Last user input at 22:57:46. Warning fired at 22:58:08 — while the agent was still speaking (playout ran 22:57:55–22:58:21). User missed the warning entirely. Warm-down fired at 22:58:16 (also during speech). Agent disconnected at 22:58:31, only 10s after finishing its response. Confirms the fix must reset the timer on `AgentState speaking → listening` so the idle window doesn't start until the agent goes quiet.

**Status:** Fixed — `resetIdleWithWarmDownRecovery()` added to `AgentStateChanged` handler on `speaking → listening` transition (`agent.ts`). Deployed 2026-03-10.

### BUG-001: TTS Race Condition on Agent Wake-up — First Response (MEDIUM)

**First seen:** 07:39 UTC (Verification attempt)
**Frequency:** High (depends on network timing)
**Impact:** When the agent wakes up from a sleep cycle, the first response (usually the bootstrap reply) still plays audio even if the client is set to `TTS: OFF`. Subsequent responses are correctly silent.

**Analysis:**
- **The Wake-Up:** User speaks, triggering a fresh agent process.
- **The Bootstrap:** Agent immediately sends a synthetic "Bootstrap" message to the Gateway in its `entry()` loop.
- **The Inference:** The Gateway starts generating the first response while the agent's local `ttsEnabled` is still its default (`true`).
- **The Re-Sync:** A few milliseconds later, the client (Flutter) sends the `tts-mode: off` command via the data channel.
- **The Result:** The first response's TTS stream is already created with audio enabled before the re-sync command is processed.

**RCA Logs:**
```
[07:39:15.650] INFO: Sending bootstrap message
[07:39:15.687] INFO: TTS mode changed (value: off)
```
The re-sync arrived 37ms *after* the bootstrap started.

**Root cause (confirmed):** `_pipelineReplyTaskImpl` (`SDK agent_activity.ts:1521`) captures `audioOutput` synchronously — before the first `await` — as part of the `session.generateReply()` call chain. On wake-up, `ctx.waitForParticipant()` resolves immediately (client was already in the room), so `generateReply()` fires and snapshots `audioOutput = non-null` before the client's `tts-mode:off` re-sync arrives 37ms later. Calling `session.output.setAudioEnabled(false)` after this point has no effect on the in-flight pipeline task.

**Task:** [`tasks/02-livekit-agent/033-bootstrap-tts-settle-window.md`](../../tasks/02-livekit-agent/033-bootstrap-tts-settle-window.md)

**Field verification (22:00–22:02 UTC):** First fix (settle window + unconditional `_sendTtsMode()`)
was deployed (container rebuilt at 21:57). Settle window correctly protected the BOOTSTRAP
pipeline — bootstrap produced 0 LLM chunks, no audio played. However, audio still played for
the first response because the **user text message pipeline** (the wake trigger) races separately.

**Second root cause (confirmed):** `ParticipantConnectedEvent` in `livekit_service.dart` called
`_flushPendingTextMessages()` BEFORE `_sendTtsMode()`. The queued text message arrived at the
agent at T+68ms; TTS mode arrived at T+89ms (21ms too late). The text message pipeline captured
`audioOutput = non-null` and audio played via Piper fallback.

**Fix (Change 3):** Reorder `ParticipantConnectedEvent` handler — send `_sendTtsMode()` before
`_flushPendingTextMessages()`. SCTP ordered delivery guarantees `tts-mode` arrives before
`text_message`, so the text message pipeline captures `audioOutput = null`.

**Third root cause (confirmed):** Concurrent s_1 (user text) + s_2 (bootstrap) pipelines caused `TranscriptManager` to finalize s_1's empty segment when s_2 started. s_1's LLM content arrived after its segment was deleted → silently dropped → no transcript visible.

**Fix (Change 4 — Agent bootstrap skip):** Added `userMessageReceivedBeforeBootstrap` flag in `agent.ts`. When a `text_message` arrives during the 200ms settle window, the flag is set. After the settle window, if the flag is true, bootstrap is skipped. This prevents the concurrent s_2 stream from deleting s_1's segment.

**Field verification (22:55–22:58 UTC):** Wake-up session (agent PID 764, room `fletcher-1773183319831`):
- TTS mode arrived at T+6ms (22:57:13.010) — well before settle window
- Bootstrap fired at T+200ms (22:57:13.216) — settle window correctly applied
- Bootstrap interrupted by user speech — no audio pipeline started
- Session completed normally, tester confirmed: no unwanted audio, transcript visible

**Status:** Verified Fixed — all three changes deployed and confirmed working. See task 033.

### BUG-004: Artifacts Clump Under Single Message After Agent Reconnect (HIGH)

**First seen:** ~20:36 UTC (screenshot, 2026-03-10)
**Frequency:** 100% — occurs on every sleep/wake cycle that produces artifacts
**Impact:** All artifacts from a new agent session appear stacked under the last message
of the *previous* session rather than under the message that produced each one. In the
observed case, 5 "Voice Degraded" artifacts spanning 12:18–12:33 (15 minutes, multiple
turns) all clumped under one old message. This is a regression of task 023
(Artifact–Message Association).

**Evidence:**
- Screenshot 1 (artifact panel): 5 "Voice Degraded" artifacts with timestamps
  12:18, 12:19, 12:25, 12:30, 12:33 — clearly from different turns.
- Screenshot 2 (chat transcript): all 5 `[ARTIFACT: Voice Degraded]` buttons stacked
  below a single agent message.

**Root Cause (confirmed):**

`_lastAgentSegmentId` in `livekit_service.dart` (line 50) is never cleared when the
agent disconnects. When the agent reconnects after a sleep cycle, `_lastAgentSegmentId`
still holds the segment ID from the **last message of the previous session**. New
artifacts arriving from the new session are stamped with this stale ID
(`livekit_service.dart:842–844`), associating them all with that one old message.

The transcript is preserved across sleep/wake cycles (Epic 20 design), so the old
segment ID *does* exist in the transcript — `ChatTranscript._groupArtifactsByMessage`
correctly places all stamped artifacts there, producing the observed clump.

Trigger chain:
1. Old session ends — last agent message has `segmentId = "seg-abc"` →
   `_lastAgentSegmentId = "seg-abc"`.
2. Agent disconnects — `_lastAgentSegmentId` is NOT cleared.
3. New agent session starts — first artifact arrives before any new transcript text.
4. Artifact stamped with `"seg-abc"` (the old segment ID).
5. All subsequent artifacts in the new session share the same stamp (until the new
   agent's transcript events update `_lastAgentSegmentId` at `livekit_service.dart:861`).
6. **Result:** every artifact from the new session groups under the old final message.

**Fix (one line):**

In `ParticipantDisconnectedEvent` handler (`livekit_service.dart:602`), when
`remaining == 0`, add:

```dart
_lastAgentSegmentId = null;  // Reset for new agent session (BUG-004)
```

This ensures artifacts arriving before the first new transcript event fall back to the
nearest prior agent message (the safe fallback in `_groupArtifactsByMessage`) rather
than being mis-stamped with the stale ID.

**Task:** [`tasks/07-ui-ux/038-artifact-clump-reconnect-regression.md`](../../tasks/07-ui-ux/038-artifact-clump-reconnect-regression.md)

**Status:** Closed — Fixed. `_lastAgentSegmentId = null` added in `ParticipantDisconnectedEvent` handler when `remaining == 0` (`livekit_service.dart`).

### BUG-007: Agent attempts sleep during "Thinking" state (MED)

**First seen:** 23:48 UTC (Tester report)
**Frequency:** Observed during field test
**Impact:** Agent could enter sleep mode or trigger idle countdown while an LLM response was still being composed/pondered.

**Root Cause (confirmed):**
The `AgentStateChanged` handler only called `resetIdleWithWarmDownRecovery()` on `speaking → listening`. When the agent entered `thinking`, the idle timer kept counting from the user's last speech event. If OpenClaw took 30–60s to respond, the 5-min clock could approach expiry during thinking, causing warning → warm-down → shutdown mid-response.

**Fix applied (`agent.ts`):**
Added a second `resetIdleWithWarmDownRecovery()` call in the `AgentStateChanged` handler when `ev.newState === 'thinking'`. The full idle window now restarts from the moment the brain receives the user's request. Deployed 2026-03-10.

**Status:** Fixed — timer now resets on `thinking` state entry. Deployed.

### BUG-006: Agent Sleep Timer starting too early — TTS cut off (MED)

**First seen:** 23:32 UTC (Tester report)
**Frequency:** Consistent
**Impact:** Agent attempted to sleep while TTS was actively playing, cutting the response off mid-sentence.

**Root Cause (confirmed):**
Same root cause as BUG-007. The idle timer started from the user's last speech event (`UserInputTranscribed`). **Fix applied:**

**Status:** Fixed. Deployed.

### BUG-008: No maxWait for slow/hung brain responses (MED)

**First seen:** Session note from tester (~16:48 UTC)
**Frequency:** Intermittent (depends on OpenClaw load)
**Impact:** When the brain (OpenClaw) takes an extremely long time or hangs indefinitely, the agent stays in `thinking` forever. No feedback to the user, no recovery path.

**Tester request:** Add a separate `maxWait` timeout. If the brain hasn't delivered any tokens after N seconds, cancel the pending request and surface an error artifact to the client so the user knows to retry.

**Analysis:**
- The `thinking` reset (BUG-007 fix) prevents false idle during slow brains, but doesn't bound the wait time.
- A hung OpenClaw stream can keep the agent locked in `thinking` indefinitely (BUG-005 documents a 45s+ case).
- A `FLETCHER_BRAIN_MAX_WAIT_MS` env var (e.g., 60000 = 60s) would trigger an abort + user-visible "Brain timed out — please retry" artifact if no tokens arrive within the window.
- Implementation requires either stream-level timeouts in `OpenClawChatStream` (ganglia) or a `setTimeout`-based abort in `agent.ts` triggered on `thinking` entry.

**Related task:** [`tasks/02-livekit-agent/039-brain-max-wait-timeout.md`](../../tasks/02-livekit-agent/039-brain-max-wait-timeout.md)

**Status:** Identified — task created.

### BUG-009: Network Handoff Reclaims Mic While User is Muted (MED)

**First seen:** ~17:00 UTC (Tester report, end-of-session)
**Frequency:** Every network handoff (WiFi→5G, BT device change, etc.)
**Impact:** When a network transition triggers an audio device change event, the app restarts the audio track unconditionally — even when the user is muted. This temporarily reclaims the microphone from the OS, blocking the Android keyboard dictation feature. Tester cannot use device-native STT to compose text messages while this is happening.

**Client log evidence:**
```
03-10 17:00:18 [Fletcher] Device change detected — debouncing (2s)
03-10 17:00:18 [Fletcher] Device change detected — debouncing (2s)
03-10 17:00:20 [Fletcher] Audio device changed — refreshing audio track
```
Also visible at 16:48:08 after app restart while connected (agent absent, presumably muted).

**Analysis:**
- `_onDeviceChange` fires on any audio device change (BT handoff, WiFi→5G ICE renegotiation, etc.).
- The handler debounces (2s) and calls `_refreshAudioTrack()` (`livekit_service.dart:998–1012`).
- `_refreshAudioTrack()` (`livekit_service.dart:1014–1038`) already has a guard at line 1029:
  `if (track != null && !_isMuted)` — so `restartTrack()` is NOT called when muted.
- HOWEVER: the function still runs its preamble (sets `_isRefreshingAudio = true`, waits 1 second,
  logs "Audio device changed") even when muted, because the early-exit for `_isMuted` is missing.
- This wastes the 1-second `_isRefreshingAudio` lock, dropping any device change that fires during
  that window; and there is no `_pendingDeviceChange` flag to defer the restart until unmute.
- Additionally, `RoomReconnectedEvent` at line 532 calls `_refreshAudioTrack()` with no mute
  pre-check, producing "Audio device changed — refreshing audio track" even when muted (confirmed
  in BUG-010 log at 16:53:47).

**Root cause (confirmed):** `_refreshAudioTrack()` lacks an early-exit for the muted state,
running wasteful work (1s delay + `_isRefreshingAudio` lock) without actually restarting the track.
`restartTrack()` itself is already gated by `!_isMuted` at line 1029 — the original theory that it
ran unconditionally is incorrect. The remaining fix is: early-exit on `_isMuted`, add
`_pendingDeviceChange` flag, clear the flag on unmute.

**Task:** [`tasks/03-flutter-app/040-muted-device-change-mic-guard.md`](../../tasks/03-flutter-app/040-muted-device-change-mic-guard.md)

**Status:** RCA COMPLETE — `restartTrack()` already blocked when muted; remaining fix is early-exit guard + `_pendingDeviceChange` deferred refresh in `_refreshAudioTrack()` and `toggleMute()`.

### BUG-010: Repeated ICE Drops After Agent Idle Disconnect — UI Stuck in Reconnecting State (HIGH)

**First seen:** ~16:53:45 UTC (tester screenshot)
**Frequency:** Observed twice in ~5 minutes; 16:50 and 16:53 sequences
**Impact:** After the agent goes idle (45s timeout), the LiveKit SDK starts cycling through ICE disconnects every 25-32 seconds. Each cycle shows `SDK reconnect attempt 1/10` → `SDK reconnecting...` → `SDK reconnected successfully`. The UI appears to get stuck in a "Reconnecting..." state even though the SDK recovers, forcing the user to force-quit and restart the app.

**Client log evidence — occurrence 1 (16:44, first occurrence):**

```
16:45:05-18 [user tries voice twice, no agent joins]
16:45:59 App backgrounded → Disconnecting  ← tester force-closes
```

**Occurrence 2 (16:50, duplicated attempt 1):**
```
16:50:44.978 SDK reconnect attempt 1/10 (next retry in 0ms)
16:50:45.708 SDK reconnect attempt 1/10 (next retry in 0ms)  ← duplicate!
  [No reconnect success logged; app recovered by 16:52:24 when next text was sent]
```

**Occurrence 3 (16:53, tester screenshot at 16:53:45):**
```

16:53:43 SDK reconnect attempt 1/10 → attempt 3/10
16:53:47 SDK reconnected successfully
16:53:47 Audio device changed — refreshing audio track
  -- 25 seconds later --
16:54:12 SDK reconnect attempt 1/10 → 3/10
16:54:16 SDK reconnected successfully
16:54:16 Audio device changed — refreshing audio track
  -- tester force-quits at 16:55:40 --
```

**Task:** [`tasks/09-connectivity/041-sdk-reconnect-loop-after-idle.md`](../../tasks/09-connectivity/041-sdk-reconnect-loop-after-idle.md)

**Status:** RCA COMPLETE

Root cause per symptom:
1. **Repeated ICE drops (25s cycle):** `RoomReconnectedEvent` handler calls `_refreshAudioTrack()` unconditionally (`livekit_service.dart:532`). `restartTrack()` inside it calls `getUserMedia()`, which fires an OS `devicechange` event on Android. `_onDeviceChange` debounces and calls `_refreshAudioTrack()` again — looping every ~25s.
2. **Duplicate `attempt 1/10` events:** SDK `Engine.handleReconnect()` (`engine.dart:932`) has no concurrent-call guard before emitting `EngineAttemptReconnectEvent`. Both subscriber and publisher `onConnectionState` callbacks call `handleReconnect()` near-simultaneously when the ICE transport fails, producing two `attempt=1` events before `reconnectAttempts` is incremented.
3. **UI stuck in Reconnecting:** `_updateAudioLevels()` has a sticky "Keep reconnecting state" branch (`livekit_service.dart:1123-1125`) that holds the status indefinitely. Additionally, `RoomReconnectingEvent` handler does not set `_reconnecting = true` (`livekit_service.dart:450`), so the audio-level loop cannot distinguish an active SDK-native reconnect from a recovered state.
4. **Dispatch fails during coincident ICE reconnect:** `toggleMute()` dispatches the agent without checking `_reconnecting` or `_state.status == reconnecting` (`livekit_service.dart:1203-1207`). Dispatch fired while the WebSocket was mid-reconnect; the agent either couldn't establish tracks or the dispatch call failed silently with no retry.

See task file for full code-grounded analysis and specific before/after fix code.
