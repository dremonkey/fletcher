### BUG-001: TTFT spike to 15.8s on second utterance (HIGH)

**First seen:** 07:19:04 UTC
**Frequency:** Once (1 of 2 utterances)
**Impact:** User waited ~16 seconds with cycling pondering messages before getting a response

**Log excerpt:**
```
voice-agent-1  | 2026-03-10T07:19:04.793Z ganglia:openclaw:client POST http://localhost:18789/v1/chat/completions msgCount=1 hasTools=false sessionKey=guest
voice-agent-1  | 2026-03-10T07:19:04.799Z ganglia:openclaw:client response 200 OK (fetchLatency=7ms)
voice-agent-1  | 2026-03-10T07:19:20.572Z ganglia:openclaw:client timing: fetchStart→firstChunk=15779ms
voice-agent-1  | [ttftMs]: 15784
```

**Analysis:** The first utterance ("I'm testing to see if it'll release the mic when I mute it.") had a normal 1,448ms TTFT. The second utterance ("It does not look like it's working as expected.") spiked to 15,784ms. The HTTP fetch itself completed in 7ms (200 OK), so the gateway connection was established quickly — the delay was in OpenClaw/gateway generating the first token of the response. The user saw 6 pondering status rotations ("Mulling it over...", "Shuffling neurons...", "Asking the magic 8-ball...", "Casting a spell...", "Summoning words...", "Consulting the runes...") while waiting.

**Root cause:** Gateway-side latency spike. Likely the LLM backend (behind OpenClaw) was slow to respond on this particular request. Could be model cold start, queue depth, or the conversation history being longer on the second turn.

**Related task:** `tasks/05-latency-optimization/` (if exists)

### BUG-002: ADM stopLocalRecording fix insufficient — mic still held on mute (HIGH)

**First seen:** 07:19:04 UTC (tester verbal report)
**Frequency:** Every mute cycle
**Impact:** When user mutes in Fletcher, Android keyboard voice typing still can't access mic

**Tester said:** "It does not look like it's working as expected."

**Client log excerpt:**
```
00:18:38.630 [Fletcher] Mute toggled: muted=true
00:18:38.636 FlutterWebRTCPlugin: trackDispose() track is null
00:18:38.649 [Fletcher] Android ADM recording stopped (mic released for OS)
00:18:38.649 [Fletcher] Mic stopped
```

**Analysis:** The recent fix (commit 139286d) added `stopLocalRecording` via Android AudioDeviceModule on mute. Client logs confirm the ADM recording stop fires correctly, but `trackDispose() track is null` shows the WebRTC track was already null when dispose was attempted. Despite the ADM stop, the OS still reports the mic as in use — likely because the WebRTC PeerConnection maintains the audio session/handle even when the track is stopped.

The mute→unmute cycle was tested twice during this session (00:18:38 and 00:19:06), both showing the same pattern: track null on dispose, ADM stop fires, but mic not actually released to OS.

**Root cause:** Stopping ADM recording is not sufficient. The WebRTC PeerConnection keeps the native `AudioRecord` open as long as an `RtpSender` exists for audio. `setMicrophoneEnabled(false)` only calls `mute()` which stops the track but leaves the sender in the PeerConnection.

**Fix (commit d2416ad):** Replace `setMicrophoneEnabled(false)` with `removePublishedTrack(sid)` which:
1. Removes the `RtpSender` from the PeerConnection via `pc.removeTrack(sender)`
2. Triggers SDP renegotiation via `engine.negotiate()`
3. WebRTC native layer sees no active audio sender and releases `AudioRecord`

On unmute, `setMicrophoneEnabled(true)` creates and publishes a fresh audio track.

**Verified:** 2026-03-10 00:37 UTC — mute toggle released mic, keyboard STT worked. Log excerpt:
```
00:37:50.563 [Fletcher] Mute toggled: muted=true
00:37:50.575 [Fletcher] Audio track unpublished (mic released for OS)
00:37:50.576 [Fletcher] Mic stopped
```

**Related:** This is a continuation of BUG-001 from 2026-03-09 session. Three fix iterations:
1. `setAndroidAudioConfiguration(media)` — insufficient (AudioRecord still held)
2. `ADM.stopLocalRecording()` — insufficient (PeerConnection overrides)
3. `removePublishedTrack()` — **works** (removes sender, triggers renegotiation)

**Status:** Fixed (verified).

### BUG-003: Bluetooth audio input change not detected (MEDIUM, TESTER REPORT)

**Frequency:** Once, not reproducible
**Impact:** Switching audio input to Bluetooth was not picked up by the app

**Tester said:** "Change of input to Bluetooth was not picked up. I can't seem to recreate this one."

**Analysis:** Intermittent failure of audio route detection on Bluetooth connection. This is a known problem area — see `tasks/09-connectivity/009-bluetooth-audio-route-recovery.md`. The Flutter WebRTC plugin relies on Android's `AudioDeviceCallback` or `onRouteChange` to detect BT transitions, and these events can be unreliable depending on BT connection timing and Android audio focus state. Without reproduction steps or log evidence, root cause is speculative.

### BUG-004: TTS OFF state lost on agent rejoin after idle disconnect (HIGH, TESTER REPORT)

**Frequency:** Frequent — every idle disconnect/rejoin cycle when TTS is off
**Impact:** Client header shows "TTS OFF" but agent plays TTS audio. User perceives a broken toggle.

**Tester said:** "TTS OFF is inconsistent. Client will display TTS OFF but TTS comes through. This one shows up pretty frequently. I think it has to do with the agent auto-disconnect feature we added. I suspect it forgets."

**Analysis:** Confirmed by code inspection. The `ParticipantConnectedEvent` handler in `livekit_service.dart:571-593` does NOT re-send the `tts-mode` event when a new agent joins. The TTS mode resync only fires on:
1. **Initial room connect** (`livekit_service.dart:430-432`)
2. **Room reconnect** (`livekit_service.dart:507-510`)

When the agent idle-disconnects and a new agent job is dispatched (triggered by user activity), the room stays connected — so neither path fires. The new agent defaults to `ttsEnabled=true`, but the client still shows "TTS OFF" from its local state.

**Root cause:** Missing `_sendTtsMode()` call in `ParticipantConnectedEvent` handler.

**Fix:** Add TTS mode resync to the `ParticipantConnectedEvent` handler at `livekit_service.dart:581`, right after `_flushPendingTextMessages()`:
```dart
// Re-sync TTS mode with new agent (BUG-004)
if (_textOnlyMode) {
  _sendTtsMode();
}
```

**Related:** Same as BUG-002 from 2026-03-09 session. The "Client-Side Re-Sync" approach (option 2 from that analysis) is the right fix.
