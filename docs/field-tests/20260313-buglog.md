## Detailed Entries

### BUG-011: Relay does not rejoin room after dropping from LiveKit (HIGH)

**First seen:** 2026-03-13
**Frequency:** 100% — relay never recovers once it leaves/drops from a room
**Impact:** Chat mode permanently broken until relay process is manually restarted. User sees "Relay not connected — try again" with no recovery path.

**Symptom:** Once the relay drops from the LiveKit room (idle timeout, network glitch, or LiveKit server restart), it never comes back. Chat messages fail silently from the user's perspective.

**Root cause — two independent gaps:**

#### Gap 1: No `RoomEvent.Disconnected` handler

`apps/relay/src/livekit/room-manager.ts:77-100` — `joinRoom()` only subscribes to `RoomEvent.DataReceived`. There is no handler for `RoomEvent.Disconnected`. When the underlying LiveKit connection drops (ICE failure, server restart, network issue), the relay has no idea — the `rooms` Map still holds a stale `RoomConnection` pointing at a dead `Room` object. Subsequent `sendToRoom()` calls publish into the void, and the catch in `relay-bridge.ts:251` swallows the error silently.

The `BridgeManager` also doesn't know the room died — `bridges` Map still has the entry, `hasRoom()` returns `true`, and the webhook handler's idempotency check (`webhook.ts:49`) skips the room because it thinks the relay is already there.

#### Gap 2: No rejoin trigger after idle disconnect

The architecture doc (`apps/relay/docs/architecture.md:139-152`) describes this lifecycle:

```
4. Relay disconnects from room (idle timeout)
5. User rejoins room → webhook triggers relay rejoin
```

The intended flow is: idle timeout removes the relay → user does something → user "rejoins" the room → LiveKit fires `participant_joined` webhook → relay auto-joins.

**But the user doesn't rejoin the room.** The mobile client stays connected to the LiveKit room the entire time — it never leaves and reconnects. So `participant_joined` never fires, and there is no trigger to bring the relay back. The mobile tries to send a chat message, `_hasRelayParticipant` returns `false` (`livekit_service.dart:1378`), and the user sees "Relay not connected — try again" with no recovery.

The mobile side has no mechanism to request a relay rejoin — it doesn't call `POST /relay/join` or signal the relay in any way. It passively checks if a relay participant exists and gives up if one doesn't.

**Reproducing:**
1. Connect to a room in chat mode
2. Wait for relay idle timeout (~30 min, or set `RELAY_IDLE_TIMEOUT_MS=10000` for quick repro)
3. Send a chat message after the relay has left
4. Observe: "Relay not connected" error, no recovery

Alternatively: kill the relay's LiveKit connection (e.g., restart LiveKit server) while the mobile client stays connected.

**Proposed fix — two parts:**

1. **RoomManager: `RoomEvent.Disconnected` handler** — When the room connection drops unexpectedly, clean up local state (`rooms` Map) and emit a callback/event so `BridgeManager` can attempt a rejoin with exponential backoff. This handles unexpected drops (network, server restart).

2. **Mobile: trigger relay rejoin on chat send** — When `_hasRelayParticipant` is `false` and the user sends a chat message, call `POST /relay/join` (via the relay's HTTP endpoint) to wake the relay back up, instead of just showing an error. This handles the designed idle-disconnect-then-rejoin flow. The endpoint already exists (`routes.ts:47`) but nothing calls it at runtime.

**Related:**
- BUG-007 (20260312): "Relay fails to auto-connect/reconnect after service restart" — same family; startup discovery handles restarts but not runtime drops
- TASK-059 (deferred teardown): Solved the `participant_left` → immediate teardown race, but doesn't help when the relay itself drops
- Architecture doc assumption (`architecture.md:152`): "The next participant join webhook restarts the cycle" — incorrect when the human stays connected

**Status:** FIXED — Added `RoomEvent.Disconnected` handler in RoomManager (cleans stale map entry, fires callback), BridgeManager wires disconnect → auto-rejoin with exponential backoff (1s/2s/4s, max 3 retries), and periodic room discovery timer (default 30s) as a safety net.
