### BUG-011: Relay does not rejoin room after dropping from LiveKit (HIGH)

**First seen:** 2026-03-13
**Frequency:** 100% — relay never recovers once it leaves/drops from a room
**Impact:** Chat mode permanently broken until relay process is manually restarted. User sees "Relay not connected — try again" with no recovery path.

**Symptom:** Once the relay drops from the LiveKit room (idle timeout, network glitch, or LiveKit server restart), it never comes back. Chat messages fail silently from the user's perspective.

**Root cause — two independent gaps:**

#### Gap 1: No `RoomEvent.Disconnected` handler

`apps/relay/src/livekit/room-manager.ts:77-100` — `joinRoom()` only subscribes to `RoomEvent.DataReceived`. There is no handler for `RoomEvent.Disconnected`. When the underlying LiveKit connection drops (ICE failure, server restart, network issue), the relay has no idea — the `rooms` Map still holds a stale `RoomConnection` pointing at a dead `Room` object. Subsequent `sendToRoom()` calls publish into the void, and the catch in `relay-bridge.ts:251` swallows the error silently.

The `BridgeManager` also doesn't know the room died — `bridges` Map still has the entry, `hasRoom()` returns `true`, and the webhook handler's idempotency check (`webhook.ts:49`) skips the room because it thinks the relay is already there.

#### Gap 2: No rejoin trigger after idle disconnect

The architecture doc (`apps/relay/docs/architecture.md:139-152`) describes this lifecycle:

```
4. Relay disconnects from room (idle timeout)
5. User rejoins room → webhook triggers relay rejoin
```

The intended flow is: idle timeout removes the relay → user does something → user "rejoins" the room → LiveKit fires `participant_joined` webhook → relay auto-joins.

**But the user doesn't rejoin the room.** The mobile client stays connected to the LiveKit room the entire time — it never leaves and reconnects. So `participant_joined` never fires, and there is no trigger to bring the relay back. The mobile tries to send a chat message, `_hasRelayParticipant` returns `false` (`livekit_service.dart:1378`), and the user sees "Relay not connected — try again" with no recovery.

The mobile side has no mechanism to request a relay rejoin — it doesn't call `POST /relay/join` or signal the relay in any way. It passively checks if a relay participant exists and gives up if one doesn't.

**Reproducing:**
1. Connect to a room in chat mode
2. Wait for relay idle timeout (~30 min, or set `RELAY_IDLE_TIMEOUT_MS=10000` for quick repro)
3. Send a chat message after the relay has left
4. Observe: "Relay not connected" error, no recovery

Alternatively: kill the relay's LiveKit connection (e.g., restart LiveKit server) while the mobile client stays connected.

**Proposed fix — two parts:**

1. **RoomManager: `RoomEvent.Disconnected` handler** — When the room connection drops unexpectedly, clean up local state (`rooms` Map) and emit a callback/event so `BridgeManager` can attempt a rejoin with exponential backoff. This handles unexpected drops (network, server restart).

2. **Mobile: trigger relay rejoin on chat send** — When `_hasRelayParticipant` is `false` and the user sends a chat message, call `POST /relay/join` (via the relay's HTTP endpoint) to wake the relay back up, instead of just showing an error. This handles the designed idle-disconnect-then-rejoin flow. The endpoint already exists (`routes.ts:47`) but nothing calls it at runtime.

**Related:**
- BUG-007 (20260312): "Relay fails to auto-connect/reconnect after service restart" — same family; startup discovery handles restarts but not runtime drops
- TASK-059 (deferred teardown): Solved the `participant_left` → immediate teardown race, but doesn't help when the relay itself drops
- Architecture doc assumption (`architecture.md:152`): "The next participant join webhook restarts the cycle" — incorrect when the human stays connected

**Status:** FIXED — Added `RoomEvent.Disconnected` handler in RoomManager (cleans stale map entry, fires callback), BridgeManager wires disconnect → auto-rejoin with exponential backoff (1s/2s/4s, max 3 retries), and periodic room discovery timer (default 30s) as a safety net.

### BUG-013: Voice mode times out — relay missing voice-acp handler (HIGH)

**First seen:** 22:19 UTC
**Frequency:** 100% — every voice prompt fails
**Impact:** Voice mode completely non-functional. User speaks, pondering phrases cycle for 120s, then pipeline error.

**Log excerpt:**
```
ganglia:relay:client → relay [voice-acp]: {"jsonrpc":"2.0","id":"req-relay_1-...","method":"session/prompt",...}
ganglia:relay:stream pondering: "Scanning the multiverse..." streamId=relay_1
... (pondering cycles for 120s) ...
ERROR: RelayChatStream error: Error: relay session/prompt timed out after 120000ms
ERROR: Pipeline error — source: "Pipeline" message: "Error: relay session/prompt timed out after 120000ms"
```

**Root cause:** `.env` has `GANGLIA_TYPE=relay`, so the voice-agent sends `session/prompt` on the `voice-acp` data channel topic. The relay process (PID 3085903, started at 15:17) was launched before the 064b commit that adds the `voice-acp` handler to `RelayBridge`. The relay ignores the messages (`← relay: ignoring topic=relay (want voice-acp)`) and the voice-agent times out after 120s.

Chat mode works because it uses the existing `relay` topic path.

**Fix:** Restart the relay process to pick up the 064b code. Not a code bug — deployment/restart gap.

**Status:** FIXED — relay restarted, voice mode working end-to-end through relay.

### BUG-015: Response transcripts missing when TTS is disabled (MED)

**First seen:** 2026-03-13 16:09 PDT
**Frequency:** 100% when TTS is off
**Symptom:** When TTS is disabled in GTS mode, response transcripts do not appear in the UI, even though the message is received and a response is composed.

**Impact:** User is "blind" to the assistant's response when voice output is off. The conversation flow is broken because the user can't see what the assistant said.

**Notes:** Tester reported: "when TTS is off no response transcript is visible even though it looks like you received the message and composed a response". Likely a suppression of the transcript event or data channel message when the audio path is disabled.

**Root cause:** No explicit suppression found via static analysis. Most likely a stream consumption difference: when TTS is off, the LLM text stream has a single consumer (no tee), which may cause event-loop starvation or backpressure preventing `onContent` from firing. Needs runtime debugging.

**Code analysis (2026-03-13 evening session):** Confirmed `onContent` fires unconditionally in both relay-stream.ts:176-179 and acp-stream.ts:133-136 — no TTS state check anywhere in the ganglia pipeline. The bug is upstream in the LiveKit SDK's stream consumption model (single-consumer vs tee'd path when TTS is disabled).

**Client log cross-reference (20260313-client-1634-1640.txt):** Room `timber-cairn` session shows two text messages sent (16:35:44, 16:37:56) with TTS toggled OFF at 16:36:50. Zero transcript events appeared in client log. Pondering status cycled for ~2 minutes without ever completing. However, this session may have been affected by the relay-not-found race (BUG-017) — backend logs for `timber-cairn` were not captured. Cannot conclusively isolate BUG-015 from this data alone.

### BUG-017: Agent tries RelayLLM before relay participant joins room (MED)

**First seen:** 23:34:42 UTC
**Frequency:** On initial room join — race between agent bootstrap and relay join
**Impact:** First agent utterance fails silently. Agent enters "thinking" state, triggers LLM inference, but relay participant hasn't joined yet (joins ~2s later). The bootstrap speech handle produces no response.

**Log excerpt:**
```
[23:34:42.444] Creating speech handle speech_id: "speech_a100257a-2e0"
[23:34:42.446] Task performLLMInference started
[23:34:42.452] Task performLLMInference done
[23:34:42.454] Unhandled promise rejection
  "RelayLLM: no relay-* participant found in room. Remote participants: device-BP3A.251105.015."
...
[23:34:44.761] Participant connected identity: "relay-zebes-rail"
```

**Root cause:** Agent bootstrap triggers `generateReply()` (agent.ts:662) immediately on room join. With `GANGLIA_TYPE=relay`, RelayLLM.chat() searches for a `relay-*` participant but the relay takes ~2s to join (webhook → room discovery → join). The bootstrap speech handle fails, and the error is caught as an unhandled promise rejection.

**Client impact from cross-reference (20260313-client-1634-1640.txt):** Client sees continuous pondering status rotations (every 3s) but never receives a response. Possible that subsequent interactions also fail if the agent state is not recovered after the initial bootstrap failure.

**Status:** NEW — needs investigation. May need a retry mechanism or deferred bootstrap when using relay backend.
