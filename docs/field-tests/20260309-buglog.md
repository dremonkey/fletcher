## Detailed Entries

### BUG-003: Mute Button Fails to Release Mic — Verification Failed (HIGH)

**First seen:** 07:18 UTC (Verification attempt)
**Frequency:** 100%
**Impact:** Even after toggling mute in the Fletcher TUI, the Android OS microphone resource remains locked. External apps (like the Android keyboard speech-to-text) are blocked from accessing the microphone.

**Analysis:**
- Confirmed that the `flutter_webrtc` layer is likely keeping the `AudioManager` in `MODE_IN_COMMUNICATION`.
- Simply stopping the track is insufficient to release the hardware lock on Android.
- This confirms that "Mute" is currently implemented as a software-level track stop rather than a hardware-level session release.

**Required Fix:**
- Implement the "Hard Unpublish" strategy: completely unpublish and dispose of the audio track when muted to force the WebRTC layer to release the audio session.
- Alternatively, use a platform channel to manually force `AudioManager.setMode(MODE_NORMAL)` when the mute state is active.

**Status:** Verification failed — moving to "Hard Unpublish" implementation.

### BUG-002: TTS State Persistence Loss on Agent Rejoin (HIGH)

**First seen:** 20:46 UTC (tester report)
**Frequency:** Every agent wake-up cycle (if TTS was previously toggled)
**Impact:** When the agent process releases (idle timeout) and a new agent job re-joins the room upon user activity, the `ttsEnabled` state is lost. The client TUI maintains the local "OFF" state in the header, but the new agent defaults to `TTS: ON`.

**Analysis:**
- The agent's `ttsEnabled` variable is local to the process memory.
- The "Sleep/Wake" (Idle Timeout) cycle results in a completely fresh agent process.
- The current `voice-agent/src/agent.ts` does not persist user preferences between jobs.
- The data channel `tts-mode` command only reaches the currently active agent.

**Possible approaches:**
1. **Gateway Persistence:** Anchor the `tts_enabled` state in the OpenClaw Gateway session metadata. The agent should fetch this state upon joining the room.
2. **Client-Side "Re-Sync":** Flutter app should re-broadcast the current `tts-mode` to the `ganglia-events` topic whenever a new participant (agent) joins the room.
3. **OpenResponses Persistence:** If using the OpenResponses API, include the preference in the `session` configuration so the Gateway can enforce silence server-side.

**Status:** Identified — fix required in agent initialization.

### BUG-001: Mute doesn't release OS mic — Android keyboard STT blocked (HIGH)

**First seen:** 22:27 UTC (tester report)
**Frequency:** Every occurrence (when agent is connected)
**Impact:** After muting (switching to text-input mode), Android keyboard speech-to-text cannot access the microphone. User cannot use voice typing while in text mode.

**Log excerpt:**
```
16:27:17.680 [Fletcher] Input mode toggled: TextInputMode.voiceFirst → TextInputMode.textInput
16:27:17.680 [Fletcher] Mute toggled: muted=true
16:27:17.691 [Fletcher] Mic stopped
```

**Analysis:**
- `toggleMute()` executes correctly — `setMicrophoneEnabled(false)` completes in ~11ms
- LiveKit SDK's `mute(stopOnMute: true)` calls both `mediaStreamTrack.stop()` and `mediaStream.dispose()`
- Despite track/stream cleanup, the OS mic resource is not freed
- Root cause: `flutter_webrtc` holds Android `AudioManager` in `MODE_IN_COMMUNICATION` while the WebRTC peer connection is active. This mode blocks Android's speech recognizer from accessing the mic, even after the audio track is stopped.
- The track is muted (not unpublished) — the publication stays active with a stopped track. The WebRTC audio sender/transceiver remains in the SDP.

**Prior fix:** Commit `dbb4859` added `await` to `setMicrophoneEnabled(false)` — this ensured the track was stopped before the UI showed the text field, but did not address the underlying `AudioManager` mode issue.

**Attempted fixes (this session):**
1. Reordered `_updateState()` to fire after `await setMicrophoneEnabled()` — no effect (confirmed mic stops in ~7ms, timing is not the issue)

**Possible approaches (not yet tried):**
1. Fully unpublish the audio track on mute (`removePublishedTrack`) instead of just muting — triggers SDP renegotiation and may release the audio session. Downside: heavier operation, ~100-200ms unmute latency.
2. Explicitly reset Android `AudioManager` mode via platform channel when muting.
3. Investigate whether `flutter_webrtc` exposes an API to release the audio session without dropping the peer connection.

**Root cause:** LiveKit SDK's `NativeAudioManagement` only sets Android `AudioManager` to `MODE_IN_COMMUNICATION` on room connect and clears it on disconnect — muting a track doesn't change the mode. `MODE_IN_COMMUNICATION` blocks Android's speech recognizer from accessing the mic. Fix: toggle `AudioManager` mode via `flutter_webrtc` `Helper.setAndroidAudioConfiguration()` on mute/unmute.

**Task:** [`tasks/livekit-flutter-sdk/005-android-audio-mode-mic-release.md`](../../tasks/livekit-flutter-sdk/005-android-audio-mode-mic-release.md)

**Status:** RCA Complete — fix plan ready.
