### BUG-011: Zombie streams cause pondering storm, segment ID explosion, and data loss risk (CRITICAL)

**First seen:** 19:41:04 UTC
**Frequency:** Every session with multiple turns (reproducible)
**Impact:** Client receives a firehose of conflicting "thinking" status events from 6+ zombie streams. Segment IDs climb from 37 to 70+ in seconds. Risk of premature finalization of the real stream's content.

**Log excerpt:**
```
[19:41:04.615] Pondering status published  segmentId=39 streamId=s_9  (latest stream)
[19:41:04.625] Pondering status published  segmentId=40 streamId=s_4  (zombie!)
[19:41:05.517] Pondering status published  segmentId=41 streamId=s_8  (zombie!)
[19:41:05.875] Pondering status published  segmentId=42 streamId=s_6  (zombie!)
...6 streams all cycling pondering phrases every 3s, segmentId 37→70+ in 15s
```

**Analysis:** The TranscriptManager treats every `onPondering(phrase, streamId)` call as a "new stream starting" — it finalizes the current active stream, creates a new segment, and sets the caller as active. But pondering phrases ROTATE every 3 seconds on the SAME stream (the ganglia pondering timer). Old streams (s_4, s_5, s_6) whose HTTP connections haven't closed keep rotating their pondering timers, and each rotation steals the active stream slot from the actual latest stream (s_9).

**Root cause:** `TranscriptManager.onPondering()` doesn't distinguish between:
1. **First pondering call from a truly new stream** (should finalize old, switch active)
2. **Pondering rotation on an existing stream** (should just update status if active, or be silently ignored if stale)

The existing `streamSegments.set(streamId, ...)` always overwrites the entry with a fresh segment, treating every call like a new stream.

**Fix applied:** 19:47 UTC
- Added `knownStreamIds: Set<string>` to track all ever-seen stream IDs
- `onPondering(phrase)` now checks `knownStreamIds` first:
  - Known + active stream → publish status rotation only (no new segment)
  - Known + stale/zombie → silently ignored
  - Unknown → truly new stream (finalize old, create segment, set active)
- `onContent()` now guards `streamId !== activeStreamId` → reject stale content
- Added 7 new BUG-011 tests (19 total, 69 expect() calls, all passing)
- Rebuilt and redeployed voice-agent container

### BUG-012: User Transcripts Not Forwarded to Data Channel (MEDIUM)

**First seen:** 19:54:12 UTC
**Frequency:** Constant
**Impact:** Client UI (Fletcher) shows no visual feedback of what the user is saying. The "black hole" effect makes the system feel unresponsive while speaking.

**Analysis:**
In `apps/voice-agent/src/agent.ts`, the `UserInputTranscribed` event listener only logs to the server console:
```typescript
session.on(voice.AgentSessionEventTypes.UserInputTranscribed, (ev) => {
  if (ev.isFinal) {
    logger.info({ transcript: ev.transcript }, 'User input (final)');
  }
});
```

**Root cause:** `outputOptions: { transcriptionEnabled: false }` in `agent.ts` disables the SDK's
`lk.transcription` text stream forwarding for BOTH agent AND user transcripts. They're gated behind
the same flag in `RoomIO` (line 322: `if (this.outputOptions.transcriptionEnabled)`). We disabled it
to fix BUG-010's agent transcript issues, but collateral-killed user STT forwarding.

**Fix applied:** 19:53 UTC
- **Voice agent:** `UserInputTranscribed` handler now publishes `user_transcript` events on the
  `ganglia-events` data channel (both interim and final, with `user_seg_N` segment IDs)
- **Flutter app:** Added `user_transcript` handler in `_processGangliaEvent` that calls
  `_upsertTranscript` with `TranscriptRole.user`
- Voice agent rebuilt & redeployed. Flutter app needs hot-restart to pick up client-side change.

### BUG-013: Transcript UI Stale When Panel Open (MEDIUM)

**First seen:** 21:44:00 UTC (reported by Andre)
**Frequency:** Constant while panel is mounted
**Impact:** Client UI fails to update transcription/response text in real-time if the transcript drawer is already open. Requires manual interaction (re-opening) to sync.

**Root cause:** `showTranscriptDrawer()` passed a snapshot of the transcript list at open time.
The `TranscriptDrawer` widget received this as a constructor parameter but never subscribed to
`LiveKitService` updates. Since `showModalBottomSheet` creates a separate route, parent rebuilds
in `ConversationScreen` don't propagate into the already-open drawer.

**Fix applied:**
- Changed `showTranscriptDrawer()` to accept `LiveKitService` instead of a static transcript list.
- `TranscriptDrawer` now subscribes to `LiveKitService` via `addListener`/`removeListener`.
- Reads `widget.service.state.transcript` in `build()` — always gets the latest data.
- Auto-scrolls to bottom (animated) when new entries arrive while drawer is open.

### BUG-015: WiFi→5G Network Switch Kills Session (HIGH/KNOWN)

**First seen:** 22:32:15 UTC
**Frequency:** Every WiFi→cellular transition
**Impact:** Session dies immediately. User must force-quit and reopen the app to recover.

**Log excerpt:**
```
22:31:48 user joins room RM_4dvSbnrY4SML (network: cellular)
22:32:15 participant closing reason=PEER_CONNECTION_DISCONNECTED isExpectedToResume=false
22:32:15 DTLS timeout on all 4 data channels
22:32:35 room closes via departure timeout (20s)
22:32:35 job ended status=JS_FAILED error="agent worker left the room"
```

**Analysis:** The WiFi→5G transition caused Tailscale to renegotiate its tunnel. During the
handoff (~seconds), the ICE connection died. LiveKit set `isExpectedToResume: false` — it gave
up on the client immediately rather than waiting for reconnection. The 20s departure timeout
then closed the room.

The session lasted only 27 seconds (22:31:48 → 22:32:15) before the network switch killed it.

**Fix applied:**
- **Server:** Added `room.departure_timeout: 120` to `livekit.yaml` (up from default 20s). Keeps the
  room alive for 2 minutes after the last non-agent participant leaves, giving the client time to
  complete the 5G handoff (~40-80s worst case) and reconnect.
- **Agent:** Added `ParticipantDisconnected`/`ParticipantConnected` room event listeners in
  `agent.ts` for observability during reconnection windows.
- **Why this is sufficient:** BUG-016 proved the agent session continues seamlessly when the client
  reconnects (DUPLICATE_IDENTITY). The only issue was timing — the 20s departure timeout was too
  short for WiFi→5G's "break before make" handoff. 120s provides comfortable margin.

**Related task:** [`tasks/09-connectivity/007-wifi-to-5g-ice-renegotiation.md`](../../tasks/09-connectivity/007-wifi-to-5g-ice-renegotiation.md)

### BUG-016: Diagnostics Shows "No Agent" After Successful Reconnect (MEDIUM)

**First seen:** 22:43:46 UTC
**Frequency:** After network transitions that trigger DUPLICATE_IDENTITY eviction
**Impact:** Diagnostics panel shows no agent present, but the agent IS responding — confusing/misleading UI.

**Log excerpt:**
```
22:43:46 removing duplicate participant user-1772488878301 (room RM_rVrBwx8veHGd)
22:43:46 participant closing reason=DUPLICATE_IDENTITY (old stale participant evicted)
22:43:46 DTLS timeout on old participant's data channels
```

**Analysis:** The 5G→WiFi transition was handled **gracefully** by LiveKit — the client
reconnected with the same identity, LiveKit evicted the stale participant via DUPLICATE_IDENTITY,
and the new participant joined the existing room where the agent was already present. The agent
continued working (responses were flowing).

However, the Flutter diagnostics panel didn't detect the existing agent after the client's own
reconnect. The `HealthService` likely caches participant state and doesn't re-query room
participants when the client reconnects to an existing room. It may only check on initial join.

**Proposed fix:** After a reconnect (DUPLICATE_IDENTITY or ICE reconnect), the Flutter client
should re-enumerate room participants and update the diagnostics panel.

### BUG-017: Bluetooth Headphone Removal Kills Peer Connection (HIGH/KNOWN)

**First seen:** 22:49:47 UTC
**Frequency:** Every BT audio device disconnect
**Impact:** WebRTC peer connection dies. Room closes after departure timeout. Diagnostics turn red.

**Log excerpt:**
```
22:49:47 participant closing reason=PEER_CONNECTION_DISCONNECTED isExpectedToResume=false
22:49:47 DTLS timeout on all 4 data channels
22:50:07 room closes via departure timeout (20s)
22:50:07 job ended status=JS_FAILED error="agent worker left the room"
```

**Analysis:** Removing BT headphones changes the audio route on Android. The WebRTC peer
connection interprets the audio device change as a fatal transport failure rather than a
recoverable route change. This is the same root cause as BUG-004 from the 0301 session.

Note: Room 3 (`RM_gyk9dvZ6CTYg`) was created at 22:50:30 with only the agent — the user never
reconnected. The room closed after 5 minutes via "empty timeout". The app was likely in a
broken state after the BT disconnection.

**Related:** BUG-004 (0301 session), [`tasks/09-connectivity/009-bluetooth-audio-route-recovery.md`](../../tasks/09-connectivity/009-bluetooth-audio-route-recovery.md)

### BUG-018: No TTS audio — ElevenLabs API returns 402 (payment_required) (HIGH)

**First seen:** 01:30:19 UTC (entire session)
**Frequency:** Every turn — no TTS audio ever played
**Impact:** Agent responds with text transcripts only. User hears acknowledgment chimes but never hears the agent's voice.

**Log excerpt:**
```
WARN: unexpected message received from elevenlabs tts
    data: { "audio": null, "isFinal": true, ... }
(92 occurrences across the session — every TTS context closes with null audio)
```

**Direct API test confirmed:**
```
HTTP 402
{"detail":{"type":"payment_required","code":"paid_plan_required",
"message":"Free users cannot use library voices via the API.
Please upgrade your subscription to use this voice."}}
```

**Analysis:** The ElevenLabs API key (`sk_7f02...`) is on a free-tier plan. Voice ID `XZUXLIpE3dqJ9aCZUj2R` is a library voice that requires a paid subscription. The WebSocket multi-stream endpoint accepts the connection (no auth error) but returns frames with `audio: null` and then `isFinal: true`. The LiveKit `@livekit/agents-plugin-elevenlabs` SDK treats this as a normal (empty) stream completion — no error is raised.

This is why server-side metrics looked healthy:
- TTS TTFB (257-514ms) reflects WebSocket message timing, not actual audio
- `performAudioForwarding` completes instantly (zero frames to forward)
- Chimes play fine (local audio files, not ElevenLabs)
- No ERROR-level logs anywhere

**Root cause:** ElevenLabs 402 payment_required — free plan cannot use library voices via API.

**Fix options:**
1. Upgrade ElevenLabs subscription
2. Switch to a voice available on the free tier
3. Switch TTS provider (Cartesia plugin already installed)

**Status:** OPEN — awaiting decision on fix approach
