### TEST-001: Network Handover ("Nose Hole" Air Gap)

**Started:** 22:37:00 PDT
**Description:** Testing stability during the transition from local WiFi to Cellular while outside. Focus is on whether the relay/ACP session survives the IP change or if the "Nose Hole" causes a disconnect.

**Finding:** The relay tore down the bridge immediately on `participant_left` webhook — which LiveKit fires on ICE drop, **not** after `departure_timeout` (120s). The mobile SDK reconnects within 5-30s, but by then the relay has already killed the ACP subprocess and left the room. **Fix applied (commit 9e387c8, TASK-059):**
Added deferred teardown to `BridgeManager`. On `participant_left`, the relay schedules a 120s grace period instead of tearing down immediately. If `participant_joined` fires before the timer expires (i.e., client reconnected), the teardown is cancelled and the bridge stays alive. Configurable via `RELAY_DEPARTURE_GRACE_MS` env var. 8 new tests, all green.

**Status:** FIXED ✅

### BUG-001: Chat message returns empty response — ACP schema mismatch (HIGH)

**First seen:** 21:07:19 UTC
**Frequency:** Every chat message
**Impact:** User sends message in chat mode, mobile UI shows empty response box — content is flowing end-to-end but not rendered

**Log excerpt (relay debug, showing OpenClaw's actual format):**
```
"params": {
  "sessionId": "31ed0f39-...",
  "update": {
    "sessionUpdate": "agent_message_chunk",
    "content": { "type": "text", "text": "Copy that. I" }
  }
}
```

**Analysis:**
The relay was written expecting `params.updates[]` (array with `kind: "content_chunk"` items), but OpenClaw's ACP protocol sends `params.update` (singular object with `sessionUpdate: "agent_message_chunk"`). The mobile client calls `params['updates']` → gets null → defaults to `[]` → no content rendered. OpenClaw was responding correctly throughout; the mismatch was in our relay and mobile assumptions.

**Root cause:** Schema mismatch on our side — the relay + mobile were written against a designed protocol that didn't match OpenClaw's actual ACP format.

**Fix applied (final — commit e14ff0c):**

The correct fix was architectural: make the relay a transparent passthrough and parse ACP natively on mobile.

1. **`apps/relay/src/bridge/relay-bridge.ts`** — Removed translation layer; `onUpdate` now forwards `session/update` params unchanged to mobile (one line)
2. **`apps/mobile/lib/services/relay/acp_update_parser.dart`** (new) — `AcpUpdateParser` sealed class hierarchy parses real ACP wire format; returns `AcpTextDelta` for content, `AcpNonContentUpdate` for all other kinds, `null` for malformed
3. **`apps/mobile/lib/services/relay/relay_chat_service.dart`** — Uses `AcpUpdateParser` directly; handles all `sessionUpdate` kinds including `available_commands_update`, `plan`, `tool_call`, etc.
4. **`apps/relay/src/acp/types.ts`** — Updated types to match actual OpenClaw `session/update` schema
5. Added debug logging in `acp/client.ts` and `relay-bridge.ts` (raw stdout, both directions)

TDD: 26 `AcpUpdateParser` tests + 19 `RelayChatService` tests, all spec-derived payloads, all green.

**Status:** FIXED — verified ✅ (field test) + fully tested ✅

### BUG-002: Relay idle timeout shuts down ACP process (LOW)

**First seen:** 22:25:00 UTC (15:25 PDT)
**Frequency:** After 5 minutes of inactivity (default)
**Impact:** Background tasks (like "thinking" or delayed responses) are killed if the relay hits its idle timeout, as it explicitly shuts down the ACP subprocess.

**Analysis (refined post-investigation):**
The `BridgeManager` defaults to a 5-minute idle timeout. Three related gaps were discovered during deep investigation:

1. **Incoming messages don't reset timer** — `sendToRoom()` already resets `lastActivity` (so ACP streaming updates *do* keep the timer alive), but incoming mobile messages (`session/prompt`, `session/cancel`) did not. `touchRoom()` existed but was never called.
2. **No recovery if ACP dies mid-session** — if the ACP subprocess crashes, the bridge enters a zombie state (LiveKit room stays connected, mobile messages arrive, but `sessionPrompt()` throws). Only the idle timer could clean this up.
3. **No `participant_left` cleanup** — when the user disconnects, the ACP subprocess and LiveKit connection stayed alive until idle timeout fired (up to 5 min wasted per session).

**Root cause:** BUG-002 was three distinct session lifecycle gaps, all traceable to the initial room lifecycle design (room-lifecycle) which only considered the happy path.

**Fix applied (commits 38b7166, 9c48516, d0a1f73):**

1. **touch-on-incoming** (`38b7166`) — `touchRoom()` added at top of `handleMobileMessage()` in `relay-bridge.ts`. Incoming mobile messages now reset the idle timer.

2. **participant-left-webhook** (`9c48516`) — `participant_left` webhook handler added in `webhook.ts`. When a human participant disconnects, the bridge tears down immediately (ACP subprocess killed, LiveKit room left). Relay and agent departures are filtered. Returns 200 even on teardown failure (avoids LiveKit retries).

3. **lazy-acp-reinit** (`d0a1f73`) — Lazy ACP re-init on subprocess death. `AcpClient` now exposes `onExit()` callback; `RelayBridge` sets `needsReinit = true` on unexpected exit; next `session/prompt` triggers `ensureAcp()` which re-initializes and creates a fresh session. Concurrent re-init calls are coalesced. Also increased default idle timeout from 5 min → 30 min (`RELAY_IDLE_TIMEOUT_MS=1800000`).

**Status:** FIXED ✅ — all three gaps addressed, 78 tests green, type-check clean.

### BUG-005: TUI Ctrl+C doesn't kill manually-restarted relay (MEDIUM)

**First seen:** Session 2 wrap-up
**Frequency:** Any time relay process is restarted outside TUI
**Impact:** After `Ctrl+C` on TUI, relay continues running — occupies port 7890 and keeps ACP subprocesses alive

**Analysis:**
`services.ts` spawns the relay via `spawn()` and pushes it to the `children[]` array. `cleanup()` (SIGINT handler) calls `child.kill()` on every entry. There are two failure modes:

1. **Relay already running when TUI starts** — `startRelay()` detects the port is occupied, logs "Relay already running", and returns without adding anything to `children`. The pre-existing process is invisible to `cleanup()`.

2. **Relay restarted manually after TUI starts** — TUI has a stale `Subprocess` reference (old PID) in `children`. When `cleanup()` runs, it kills the dead old process; the new relay (started with `nohup`) has a different PID and is not tracked.

The current session hit case 2: I killed PID 2011858 and started PID 2044026 with `nohup`, replacing the TUI-tracked child.

**Proposed fix:** On `SIGINT`/`SIGTERM`, additionally query `/health` on the relay port and — if responding — send a `POST /shutdown` or kill the PID from the pidfile. Alternatively, have the relay write a pidfile on startup and have `cleanup()` read it.

**Related:** `tasks/` — no existing task; needs one.

**Status:** FIXED ✅ — pidfile-based relay tracking implemented (2026-03-12)

### BUG-009: ACP subprocess leak — `proc.kill()` fails silently (HIGH)

**First seen:** 22:30–22:56 PDT (session 4, relay PID 2316925)
**Frequency:** Every `shutdown()` call — 100% reproduction
**Impact:** Orphaned `openclaw` + `openclaw-acp` process pairs accumulate indefinitely. Each pair uses ~62MB RSS. 5 pairs (10 processes, ~310MB) leaked in a single 26-minute session.

**Evidence:**
Relay PID 2316925 spawned 5 `openclaw` subprocesses between 22:30 and 22:52. The relay logged `acp_shutdown` events for 4 of them, but ALL 5 are still alive:
```
PID 2317634 (spawned 22:30, shutdown 22:31): ALIVE — 62MB RSS, sleeping
PID 2319495 (spawned 22:32, shutdown 22:37): ALIVE — 62MB RSS, sleeping
PID 2324207 (spawned 22:38, shutdown 22:48): ALIVE — 62MB RSS, sleeping
PID 2331858 (spawned 22:48, no shutdown):   ALIVE — 62MB RSS, sleeping
PID 2335268 (spawned 22:52, current active): ALIVE — 62MB RSS, sleeping
```
Each `openclaw` also has a child `openclaw-acp` process (10 total). Relay `/health` reports `acpProcesses: 1` — it only tracks the latest.

**Root cause:** `AcpClient.shutdown()` in `apps/relay/src/acp/client.ts:130-152` calls `proc.kill()` which sends SIGTERM, but the `openclaw` process ignores SIGTERM and stays alive. The relay doesn't verify the process actually exited.

**Code path:**
```typescript
// client.ts:142-148
const proc = this.proc;
this.proc = null;        // ← reference cleared
try {
  proc.kill();           // ← SIGTERM sent, but openclaw ignores it
} catch { }              // ← no verification that process exited
```

**Proposed fix:**
1. After `proc.kill()`, wait briefly for `proc.exited`, then escalate to SIGKILL
2. Also kill the process group (`-pid`) to catch `openclaw-acp` children
3. Add a `verifyDead()` check after shutdown

**Status:** NEW

### BUG-010: Chat prompt lost — ACP never responded, endless spinner (HIGH)

**First seen:** 22:56:27 PDT
**Frequency:** Once observed (but likely reproducible under same conditions)
**Impact:** User sends message, sees endless spinner, never gets a response. Cancel button sends `session/cancel` but has no visible effect.

**Timeline:**
```
22:56:00.255  prompt completed (previous message) — stopReason: end_turn ✅
22:56:27.138  ← mobile: new message received (correlationId: 7118ade5)
22:56:27.139  → acp: session/prompt sent (sessionId: f5098261)
              [42 seconds of silence — no acp_raw_stdout, no response]
22:57:09.311  ← mobile: session_cancel (user hit cancel)
22:57:14.541  ← mobile: session_cancel (user hit cancel again)
              [end of log — no further activity]
```

**Analysis:**
The relay sent `session/prompt` to the active ACP subprocess (PID 2335268) at 22:56:27. ACP never responded — no `acp raw stdout` lines appear after this point. The relay has no timeout on ACP responses, so it waits forever.

Possible causes:
1. **OpenClaw gateway hung** — the gateway never returned a response to this prompt
2. **ACP stdin write silently failed** — the message was written to the pipe but the subprocess didn't receive it (unlikely — Bun would throw on broken pipe)
3. **ACP subprocess is in a bad state** — possibly related to BUG-009 (5 orphaned processes competing for the same OpenClaw session?)

**Contributing factors:**
- No relay-side timeout on ACP responses
- `session/cancel` is forwarded to ACP as a JSON-RPC notification but there's no guarantee the subprocess acts on it
- The relay doesn't detect or recover from an unresponsive ACP subprocess

**Proposed fix:**
1. Add a configurable timeout to `AcpClient.request()` (e.g., 60s default)
2. On timeout, mark the subprocess as unhealthy, trigger `needsReinit`
3. Return a user-visible error to mobile ("Response timed out, please try again")

**Status:** NEW
