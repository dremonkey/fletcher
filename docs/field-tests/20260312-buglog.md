### BUG-001: Chat message returns empty response — ACP schema mismatch (HIGH)

**First seen:** 21:07:19 UTC
**Frequency:** Every chat message
**Impact:** User sends message in chat mode, mobile UI shows empty response box — content is flowing end-to-end but not rendered

**Log excerpt (relay debug, showing OpenClaw's actual format):**
```
"params": {
  "sessionId": "31ed0f39-...",
  "update": {
    "sessionUpdate": "agent_message_chunk",
    "content": { "type": "text", "text": "Copy that. I" }
  }
}
```

**Analysis:**
The relay was written expecting `params.updates[]` (array with `kind: "content_chunk"` items), but OpenClaw's ACP protocol sends `params.update` (singular object with `sessionUpdate: "agent_message_chunk"`). The mobile client calls `params['updates']` → gets null → defaults to `[]` → no content rendered. OpenClaw was responding correctly throughout; the mismatch was in our relay and mobile assumptions.

**Root cause:** Schema mismatch on our side — the relay + mobile were written against a designed protocol that didn't match OpenClaw's actual ACP format.

**Fix applied (final — commit e14ff0c):**

The correct fix was architectural: make the relay a transparent passthrough and parse ACP natively on mobile.

1. **`apps/relay/src/bridge/relay-bridge.ts`** — Removed translation layer; `onUpdate` now forwards `session/update` params unchanged to mobile (one line)
2. **`apps/mobile/lib/services/relay/acp_update_parser.dart`** (new) — `AcpUpdateParser` sealed class hierarchy parses real ACP wire format; returns `AcpTextDelta` for content, `AcpNonContentUpdate` for all other kinds, `null` for malformed
3. **`apps/mobile/lib/services/relay/relay_chat_service.dart`** — Uses `AcpUpdateParser` directly; handles all `sessionUpdate` kinds including `available_commands_update`, `plan`, `tool_call`, etc.
4. **`apps/relay/src/acp/types.ts`** — Updated types to match actual OpenClaw `session/update` schema
5. Added debug logging in `acp/client.ts` and `relay-bridge.ts` (raw stdout, both directions)

TDD: 26 `AcpUpdateParser` tests + 19 `RelayChatService` tests, all spec-derived payloads, all green.

**Status:** FIXED — verified ✅ (field test) + fully tested ✅

### BUG-002: Relay idle timeout shuts down ACP process (LOW)

**First seen:** 22:25:00 UTC (15:25 PDT)
**Frequency:** After 5 minutes of inactivity (default)
**Impact:** Background tasks (like "thinking" or delayed responses) are killed if the relay hits its idle timeout, as it explicitly shuts down the ACP subprocess.

**Analysis (refined post-investigation):**
The `BridgeManager` defaults to a 5-minute idle timeout. Three related gaps were discovered during deep investigation:

1. **Incoming messages don't reset timer** — `sendToRoom()` already resets `lastActivity` (so ACP streaming updates *do* keep the timer alive), but incoming mobile messages (`session/prompt`, `session/cancel`) did not. `touchRoom()` existed but was never called.
2. **No recovery if ACP dies mid-session** — if the ACP subprocess crashes, the bridge enters a zombie state (LiveKit room stays connected, mobile messages arrive, but `sessionPrompt()` throws). Only the idle timer could clean this up.
3. **No `participant_left` cleanup** — when the user disconnects, the ACP subprocess and LiveKit connection stayed alive until idle timeout fired (up to 5 min wasted per session).

**Root cause:** BUG-002 was three distinct session lifecycle gaps, all traceable to the initial room lifecycle design (R-004) which only considered the happy path.

**Fix applied (commits 38b7166, 9c48516, d0a1f73):**

1. **R-008** (`38b7166`) — `touchRoom()` added at top of `handleMobileMessage()` in `relay-bridge.ts`. Incoming mobile messages now reset the idle timer.

2. **R-007** (`9c48516`) — `participant_left` webhook handler added in `webhook.ts`. When a human participant disconnects, the bridge tears down immediately (ACP subprocess killed, LiveKit room left). Relay and agent departures are filtered. Returns 200 even on teardown failure (avoids LiveKit retries).

3. **R-006** (`d0a1f73`) — Lazy ACP re-init on subprocess death. `AcpClient` now exposes `onExit()` callback; `RelayBridge` sets `needsReinit = true` on unexpected exit; next `session/prompt` triggers `ensureAcp()` which re-initializes and creates a fresh session. Concurrent re-init calls are coalesced. Also increased default idle timeout from 5 min → 30 min (`RELAY_IDLE_TIMEOUT_MS=1800000`).

**Status:** FIXED ✅ — all three gaps addressed, 78 tests green, type-check clean.

### BUG-005: TUI Ctrl+C doesn't kill manually-restarted relay (MEDIUM)

**First seen:** Session 2 wrap-up
**Frequency:** Any time relay process is restarted outside TUI
**Impact:** After `Ctrl+C` on TUI, relay continues running — occupies port 7890 and keeps ACP subprocesses alive

**Analysis:**
`services.ts` spawns the relay via `spawn()` and pushes it to the `children[]` array. `cleanup()` (SIGINT handler) calls `child.kill()` on every entry. There are two failure modes:

1. **Relay already running when TUI starts** — `startRelay()` detects the port is occupied, logs "Relay already running", and returns without adding anything to `children`. The pre-existing process is invisible to `cleanup()`.

2. **Relay restarted manually after TUI starts** — TUI has a stale `Subprocess` reference (old PID) in `children`. When `cleanup()` runs, it kills the dead old process; the new relay (started with `nohup`) has a different PID and is not tracked.

The current session hit case 2: I killed PID 2011858 and started PID 2044026 with `nohup`, replacing the TUI-tracked child.

**Proposed fix:** On `SIGINT`/`SIGTERM`, additionally query `/health` on the relay port and — if responding — send a `POST /shutdown` or kill the PID from the pidfile. Alternatively, have the relay write a pidfile on startup and have `cleanup()` read it.

**Related:** `tasks/` — no existing task; needs one.

**Status:** FIXED ✅ — pidfile-based relay tracking implemented (2026-03-12)

### BUG-006: Chat response truncated — missing final 3 words (MEDIUM)

**First seen:** 18:51:50 local (relay-2026-03-13.log, relay PID 2152717)
**Frequency:** Once observed
**Impact:** User sees partial response; message ends mid-sentence

**Relay log excerpt:**
```
[18:51:44.846] mobile_prompt_received  correlationId: edad9bd1-...
[18:51:44.846] sending prompt          sessionId: a6e3ef87-...
[18:51:49.814] prompt completed        stopReason: end_turn
[18:51:49.815] mobile_prompt_responded stopReason: end_turn
```

**New evidence:** OpenClaw logs confirm full text "Ready for your next move!" was produced. Truncation is downstream of OpenClaw.

**Root cause (confirmed):** Race between the last `session/update` chunk and the `session/prompt` result — both forwarded via concurrent, un-awaited `publishData` calls.

**Exact sequence:**

In `acp/client.ts` `readLoop`, the last chunk and the prompt response arrive as consecutive stdout lines and are processed in the same loop iteration:

1. Last chunk line → `handleLine` → `onUpdate` → `relay-bridge.ts` `forwardToMobile(chunk)` → `sendToRoom(chunk)` **started, NOT awaited**
2. Result line → `handleLine` → `pending.resolve(result)` → schedules `.then(() => forwardToMobile(result))` microtask → after loop: `sendToRoom(result)` **also started, NOT awaited**

Two concurrent `publishData` calls in-flight with no ordering guarantee. If the result packet wins the race and arrives at mobile first:

- `_handlePromptResult()` → `_activeStream?.close()` → **`_activeStream = null`**
- Last chunk arrives → `_handleSessionUpdate()` → `_activeStream?.add(...)` → **`_activeStream` is null — chunk silently dropped**

`forwardToMobile` swallows all errors (`.catch(() => {})`), making the drop invisible on the relay side.

**Fix:** Serialize `forwardToMobile` calls via a Promise chain in `relay-bridge.ts`. Each `sendToRoom` must await the previous one to complete before the next begins. This guarantees chunks always arrive before the result.

**Status:** ROOT CAUSE CONFIRMED — fix needed in `apps/relay/src/bridge/relay-bridge.ts`
