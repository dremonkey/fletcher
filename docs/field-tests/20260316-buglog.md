## Detailed Entries

### BUG-042: App still reconnecting every 30m in background/screen-off (HIGH)

**First seen:** ~09:18 PDT (Overnight report)
**Frequency:** Every 30 minutes
**Impact:** Battery drain and unnecessary session noise; contradicts BUG-034 fix.

**Analysis:** Despite the BUG-034 refactor (TASK-074), the app stays connected when the screen is locked because the `isScreenLocked` guard in `onAppBackgrounded()` returns early *before* the chat-mode check. This was designed for voice mode (earbuds use case) but incorrectly applies to chat mode too. With the room alive and no messages flowing, the relay's 30-minute idle timeout fires, disconnects the relay, LiveKit redispatches it ~27s later, and the cycle repeats.

Logcat evidence (PID 24312, 07:16–08:18 PDT): relay participant `relay-amber-elm-arjk` disconnected and reconnected 3 times at exactly 31-minute intervals. Zero app-side disconnect/reconnect events — only the relay was cycling.

**Root cause:** `isScreenLocked` guard positioned before `_voiceModeActive` check in `livekit_service.dart:2263`. Chat mode should disconnect immediately regardless of screen lock state.

**Task:** [`tasks/22-dual-mode/084-screen-lock-chat-disconnect.md`](../../tasks/22-dual-mode/084-screen-lock-chat-disconnect.md)

**Client logs:** `20260316-client-0130-0917.txt` (723 lines, filtered PID 24312)

**Status:** FIXED — Commit `TASK-084` (reorder guards in `onAppBackgrounded`)

### BUG-043: Relay crashes on session_cancel after ACP subprocess exits (HIGH)

**First seen:** 08:21 UTC (line 33872)
**Frequency:** 2 crashes observed (08:21 UTC and 20:30 UTC)
**Impact:** Relay process exits with FATAL unhandled rejection, killing all active sessions. Requires manual restart.

**Analysis:** When the mobile app sends a `session_cancel` data channel message after the ACP subprocess has already been killed (race condition between room cleanup and message delivery), `AcpClient.sendNotification()` calls `send()` which throws `"ACP subprocess not running"`. The error was not caught, becoming an unhandled promise rejection that terminates the relay process.

The sequence: participant leaves room → relay SIGKILL's ACP subprocess (see below) → mobile sends a late `session_cancel` → `sessionCancel()` → `sendNotification()` → `send()` throws → FATAL unhandled rejection → relay dies.

Secondary issue: every ACP subprocess shutdown requires SIGKILL escalation because the ACP process ignores SIGTERM (3s timeout, then exit code 137). This happened 30+ times across the session. Not the crash itself but likely contributes to the race window.

**Root cause:** `sendNotification()` in `acp-client/src/client.ts:346` did not catch the throw from `send()` when the subprocess was already dead.

**Fix:** Added try-catch in `sendNotification()` that silently ignores `"ACP subprocess not running"` for `session/cancel` notifications. Other methods still throw.

**Relay logs:** `relay-2026-03-16.log` (lines 33872, 44382)

**Status:** FIXED — Commit `1a443dc` (acp-client try-catch for dead subprocess on session/cancel)

### BUG-044: Token fetch fails on resume from background — all hosts fail (HIGH)

**First seen:** 21:49 PDT (04:49 UTC+1)
**Frequency:** Observed once; likely reproducible whenever phone deep-sleeps >5min on WiFi
**Impact:** App stuck in error state after unlocking phone. User must manually retry or restart.

**Symptoms:** Tester unlocked phone after Fletcher was backgrounded ~11 minutes (chat mode). App attempted to reconnect and showed "Connection failed: All token hosts failed". WiFi was working fine minutes before.

**Analysis:** When the app resumes from background (chat mode), `onAppResumed()` calls `connectWithDynamicRoom()` exactly once with no retry. On Android, the WiFi radio needs several seconds to re-associate after deep sleep. The URL resolver races 3 hosts with a 3-second TCP timeout — all fail. Then the token service races 3 hosts with a 5-second HTTP timeout — all fail. Total: 8 seconds from resume to failure. The user is stranded in error state.

The existing reconnect infrastructure (`tryReconnect()` / `ReconnectScheduler`) is never engaged because `tryReconnect()` fires synchronously before the async `connectWithDynamicRoom()` has time to fail and set error status.

Logcat evidence (PID 5507):
```
21:49:39.843  resumed → Resuming after background disconnect
21:49:39.844  Session stale (659s > 120s) → new room amber-elm-cbud
21:49:39.844  UrlResolver: racing 3 URLs
21:49:42.850  UrlResolver: ALL unreachable → fallback to first
21:49:42.851  TokenService: racing 3 hosts
21:49:47.866  All 3 timed out → "All token hosts failed"
```

**Root cause:** `onAppResumed()` in `livekit_service.dart:2313` calls `connectWithDynamicRoom()` fire-and-forget with no retry. WiFi not ready after deep sleep causes all hosts to time out.
