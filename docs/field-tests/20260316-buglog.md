### BUG-048: Session restoration shows current timestamp instead of original (LOW)

**First seen:** 23:30 PDT (06:30 UTC+1)
**Frequency:** Consistent during session restoration
**Impact:** Confusing UI; loses historical context of when messages were actually sent.

**Symptoms:** When a session is restored, all historical messages display the current "loaded" time in the UI instead of their original timestamps.

**Analysis:** The ACP `session/load` response or the individual `user_message` / `agent_message` updates likely do not include a `timestamp` field that the Flutter app's `Message` model can ingest. Alternatively, the app might be defaulting to `DateTime.now()` during the re-parsing of historical chunks.

**Status:** OPEN ‚Äî Recorded as reported.

### BUG-045: Outbound relay silence while "Connected" ‚Äî ghost data channel (HIGH)

**First seen:** 22:19 PDT (05:19 UTC+1)
**Frequency:** Observed during stability testing; likely reproducible after any network handoff or relay restart
**Impact:** Total communication blackout despite UI showing "Connected". User sees no response.

**Symptoms:** Tester reports "I'm not getting a response from you" even though the UI indicates the relay is connected.

**Analysis:** Four interacting failures produce the ghost connection:

1. **SDK reconnect never re-validates relay binding** (PRIMARY). `RoomReconnectedEvent` handler (`livekit_service.dart:574-626`) restores status to idle/muted without checking whether the relay participant is still present or the data channel is functional. `_sessionBound` stays `true` from the pre-reconnect state, so no new `session/bind` is sent. If the relay also disconnected and rejoined (new participant, fresh pending-bind state), it waits 30s for a bind that never arrives, then leaves the room.

2. **Relay bridge stays "started" after ACP death**. When the ACP subprocess dies, the relay bridge sets `needsReinit = true` but stays started with data handlers registered (`relay-bridge.ts:222-228`). If no new message arrives from mobile (because the data channel is also degraded), the bridge is a zombie ‚Äî occupies the room, appears healthy, processes nothing.

3. **No heartbeat on the data channel**. Neither side implements liveness probes on the "relay" topic. `RelayChatService.sendPrompt()` (`relay_chat_service.dart:128-154`) has no timeout ‚Äî the stream stays open indefinitely waiting for a response that never arrives. Once stuck, `isBusy` returns `true` blocking all subsequent sends.

4. **Relay participant replacement without re-binding**. When the relay rejoins with the same identity, LiveKit may coalesce as a participant update rather than disconnect+connect events. Mobile sees no `ParticipantDisconnectedEvent` (so `_sessionBound` stays true) and no `ParticipantConnectedEvent` (so no new bind). The relay's new bridge waits for a bind that never comes.

**Contributing factors:**
- `publishData` failures silently caught on mobile side (`livekit_service.dart:1686-1691`)
- `_sendSessionBind` is fire-and-forget with no retry or timeout (`livekit_service.dart:1696-1712`)
- Relay `forwardToMobile` logs "forward_path_dead" after 3 failures but takes no corrective action (`relay-bridge.ts:694-742`)

**Root cause:** `RoomReconnectedEvent` handler has no relay-awareness ‚Äî it was written for voice-mode reconnection (BUG-027 audio buffer) and never updated when the relay (Epic 22) was added. Combined with no prompt timeout and no data channel heartbeat.

**Fix plan:**
1. Reset `_sessionBound = false` and re-send `session/bind` in `RoomReconnectedEvent` handler
2. Add configurable timeout (30s) to `RelayChatService.sendPrompt()`
3. Add `session/bind` retry with timeout (10s, 3 attempts)
4. Relay: notify mobile when ACP dies via `session/update` error notification
5. Relay: stop bridge on dead forward path (`forwardFailures >= MAX`)

**Status:** OPEN ‚Äî RCA complete, fix plan ready

### BUG-047: Session history load inconsistent ‚Äî shows only last message or nothing (MED)

**First seen:** ~22:10 PDT (06:10 UTC+1)
**Frequency:** Frequent ‚Äî most reconnects show incomplete or missing history
**Impact:** User loses conversation context when reconnecting to an existing session. Must remember what was discussed.

**Symptoms:** When connecting to a new room within the same session (same `sessionKey`, different room name), the session history load is unreliable:
1. **Sometimes only the last message pair loads** ‚Äî not the full conversation history
2. **Sometimes nothing loads at all** ‚Äî blank transcript despite confirmed same session key across rooms

**Root cause 1 (blank transcript): `_needsSessionLoad` is never `true` when creating a new room for a continuing session.** The flag is set at `livekit_service.dart:238`: `_needsSessionLoad = recentRoom != null`. When `SessionStorage.getRecentRoom()` returns null (room stale after >120s, which always happens in chat mode since the app disconnects immediately on background), a new room is generated but `_needsSessionLoad = false`. The session key is unchanged, the ACP backend has full history, but `_loadSessionHistory()` is never called. Same issue in `_connectToNewRoom()` (line 328-376) ‚Äî it never sets the flag.

**Root cause 2 (partial history ‚Äî only last message): `user_message_chunk` kind not handled in parser.** OpenClaw's `session/load` emits user turns as `user_message_chunk` (documented in TASK-075 spike results at `tasks/25-session-resumption/075-spike-results.md:48`), but `AcpUpdateParser.parse()` at `acp_update_parser.dart:236` only handles `user_message`. The `user_message_chunk` falls through to `AcpNonContentUpdate`, which is silently dropped by `RelayChatService._handleSessionUpdate()`. Without user message delimiters, `_loadSessionHistory()` never calls `finalizeAgentTurn()` between turns ‚Äî all agent chunks from ALL turns accumulate into one giant concatenated entry. Result: one merged agent message, zero user messages.

**Contributing factors:**
- `_needsSessionLoad` consumed once (set to `false` on first successful bind) and never re-set on reconnect
- `_loadSessionHistory()` called fire-and-forget (line 962) ‚Äî errors uncaught, `_isReplaying` may stay `true`
- No validation that `session/load` returned both user and agent entries
- Silently dropped `AcpNonContentUpdate` ‚Äî no logging for unrecognized kinds during replay
- Room staleness threshold (120s) too aggressive ‚Äî always triggers "new room" for chat-mode sessions

**Fix plan:**
1. Add `user_message_chunk` handling to `AcpUpdateParser` (same `content` structure as `agent_message_chunk`)
2. Set `_needsSessionLoad = true` in `_connectToNewRoom()` when session key exists
3. Decouple session history load from room staleness ‚Äî always load if session key pre-existed
4. Add error handling / `.catchError()` to `_loadSessionHistory()` call
5. Add logging for dropped ACP update kinds during session load

**Status:** FIXED ‚Äî Added `user_message_chunk` parsing, decoupled history load from room staleness, added error handling for load failures, added replay logging for dropped update kinds

### BUG-046: Network switch grabs mic in chat mode + room stuck after disconnect (HIGH)

**First seen:** ~22:10 PDT (06:10 UTC+1)
**Frequency:** Reproducible on network switches (WiFi ‚Üî cellular)
**Impact:** Two compounding issues make network transitions unusable without force-quitting the app.

**Symptoms:**
1. **Mic grab in chat mode:** When a network switch occurs while in chat (text) mode, the app activates the microphone even though the user is not in voice mode.
2. **Room stuck after disconnect:** After the network-triggered disconnect, the app cannot reconnect or connect to a new room. User must force-quit.

**Root cause (Issue 1 ‚Äî Mic grab):** `RoomReconnectingEvent` handler (`livekit_service.dart:550-553`) unconditionally creates a `PreConnectAudioBuffer` and calls `startRecording()`, which invokes `LocalAudioTrack.create()` ‚Üí `getUserMedia` ‚Üí opens the device microphone. There is **no guard** checking `_isMuted`, `_voiceModeActive`, or input mode. This code was added for BUG-027 (voice-mode audio buffering during reconnect) and never gated for chat mode.

Other audio paths are safe: initial `connect()` guards with `if (!_isMuted)` (line 485), `_refreshAudioTrack()` bails when track is null (line 1220), `_connectToNewRoom()` routes through guarded `connect()`.

**Root cause (Issue 2 ‚Äî Stuck room):** Three compounding failures:

1. **No retry after `_connectToNewRoom()` failure.** When reconnect budget is exhausted (`livekit_service.dart:2175-2184`), the exhausted branch calls `_connectToNewRoom()`. If that fails (catch at line 369-375 sets `ConversationStatus.error`), the method returns ‚Äî no further retry, system stops trying.

2. **No user-accessible retry in chat mode.** The error banner (`conversation_screen.dart:114-137`) is a plain `TuiCard` with no tap handler. Recovery triggers exist only via app lifecycle resume (stays foreground during network switch ‚Äî never fires) or mic toggle (not the primary interaction in chat mode).

3. **ConnectivityService blind to interface switches.** `_update()` at `connectivity_service.dart:67` only emits when the boolean online state changes. WiFi‚Üícellular produces `[wifi]‚Üí[mobile]` ‚Äî both "online" ‚Äî so no event fires. `_waitForNetworkRestore()` never triggers. The reconnect loop burns its entire budget against broken TCP connections without waiting for the new interface to stabilize.

**Contributing factors:**
- SDK internal retry (~40s) runs before Fletcher's app-layer reconnect, consuming budget
- Token never refreshed during retry loop (`_doReconnectAttempt` reuses cached `_token!` at line 2222)

**Fix plan:**
1. Guard `PreConnectAudioBuffer` with `if (_voiceModeActive && !_isMuted)` (1-line fix)
2. Add tap-to-retry on error banner (`conversation_screen.dart`)
3. Retry `_connectToNewRoom()` on failure with delay before giving up
4. Detect network interface changes in `ConnectivityService` (emit event on WiFi‚Üícellular even if "online" throughout)

**Status:** FIXED ‚Äî guard PreConnectAudioBuffer for voice mode, add tap-to-retry banner, retry new room on failure, detect interface switches in ConnectivityService

### BUG-042: App still reconnecting every 30m in background/screen-off (HIGH)

**Frequency:** Every 30 minutes
**Impact:** Battery drain and unnecessary session noise; contradicts BUG-034 fix.

**Analysis:** Despite the BUG-034 refactor (TASK-074), the app stays connected when the screen is locked because the `isScreenLocked` guard in `onAppBackgrounded()` returns early *before* the chat-mode check. This was designed for voice mode (earbuds use case) but incorrectly applies to chat mode too. With the room alive and no messages flowing, the relay's 30-minute idle timeout fires, disconnects the relay, LiveKit redispatches it ~27s later, and the cycle repeats.

**Root cause:** `isScreenLocked` guard positioned before `_voiceModeActive` check in `livekit_service.dart:2263`. Chat mode should disconnect immediately regardless of screen lock state.

### BUG-043: Relay crashes on session_cancel after ACP subprocess exits (HIGH)

**First seen:** 08:21 UTC (line 33872)
**Frequency:** 2 crashes observed (08:21 UTC and 20:30 UTC)
**Impact:** Relay process exits with FATAL unhandled rejection, killing all active sessions. Requires manual restart.

**Analysis:** When the mobile app sends a `session_cancel` data channel message after the ACP subprocess has already been killed (race condition between room cleanup and message delivery), `AcpClient.sendNotification()` calls `send()` which throws `"ACP subprocess not running"`. The error was not caught, becoming an unhandled promise rejection that terminates the relay process.

The sequence: participant leaves room ‚Üí relay SIGKILL's ACP subprocess (see below) ‚Üí mobile sends a late `session_cancel` ‚Üí `sessionCancel()` ‚Üí `sendNotification()` ‚Üí `send()` throws ‚Üí FATAL unhandled rejection ‚Üí relay dies.

Secondary issue: every ACP subprocess shutdown requires SIGKILL escalation because the ACP process ignores SIGTERM (3s timeout, then exit code 137). This happened 30+ times across the session. Not the crash itself but likely contributes to the race window.

**Root cause:** `sendNotification()` in `acp-client/src/client.ts:346` did not catch the throw from `send()` when the subprocess was already dead.

**Fix:** Added try-catch in `sendNotification()` that silently ignores `"ACP subprocess not running"` for `session/cancel` notifications. Other methods still throw.

**Relay logs:** `relay-2026-03-16.log` (lines 33872, 44382)

**Status:** FIXED ‚Äî Commit `1a443dc` (acp-client try-catch for dead subprocess on session/cancel)

### BUG-044: Token fetch fails on resume from background ‚Äî all hosts fail (HIGH)

**First seen:** 21:49 PDT (04:49 UTC+1)
**Frequency:** Observed once; likely reproducible whenever phone deep-sleeps >5min on WiFi
**Impact:** App stuck in error state after unlocking phone. User must manually retry or restart.

**Symptoms:** Tester unlocked phone after Fletcher was backgrounded ~11 minutes (chat mode). App attempted to reconnect and showed "Connection failed: All token hosts failed". WiFi was working fine minutes before.

**Analysis:** When the app resumes from background (chat mode), `onAppResumed()` calls `connectWithDynamicRoom()` exactly once with no retry. On Android, the WiFi radio needs several seconds to re-associate after deep sleep. The URL resolver races 3 hosts with a 3-second TCP timeout ‚Äî all fail. Then the token service races 3 hosts with a 5-second HTTP timeout ‚Äî all fail. Total: 8 seconds from resume to failure. The user is stranded in error state.

The existing reconnect infrastructure (`tryReconnect()` / `ReconnectScheduler`) is never engaged because `tryReconnect()` fires synchronously before the async `connectWithDynamicRoom()` has time to fail and set error status.

Logcat evidence (PID 5507):
```

21:49:39.844  Session stale (659s > 120s) ‚Üí new room amber-elm-cbud
21:49:39.844  UrlResolver: racing 3 URLs
21:49:42.850  UrlResolver: ALL unreachable ‚Üí fallback to first
21:49:42.851  TokenService: racing 3 hosts
21:49:47.866  All 3 timed out ‚Üí "All token hosts failed"
```

**Root cause:** `onAppResumed()` in `livekit_service.dart:2313` calls `connectWithDynamicRoom()` fire-and-forget with no retry. WiFi not ready after deep sleep causes all hosts to time out.
