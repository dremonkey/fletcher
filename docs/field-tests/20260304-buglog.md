### BUG-020 (Recurrence): Transcript Stopped on Client (07:07 PST)

**User Observation:** "One thing I would like to log, though, is that it looks like the transcript has stopped. Coming through for some reason."

**Server-side evidence:** The agent was actively receiving and processing STT at this time. The LLM context shows the agent responding:

**Analysis:** This is a one-way data channel failure. The WebRTC audio path (client→server STT) remained functional, but the data channel path (server→client transcript) was broken. This is consistent with a partial connection degradation during a network transition — the UDP audio stream recovers faster than the SCTP data channels.

### BUG-024 (Recurrence): Gemini TTS 429 Per-Minute Rate Limit Cascade (06:53–07:16 PST)

**First occurrence:** 14:53:18 UTC (Session 2)
**Frequency:** 35 warnings across sessions 2–3
**Impact:** Complete loss of voice output. Text responses continued via data channel. 7 Pipeline errors. 10 unhandled error events.

**What happened:**

The Gemini 2.5 Flash TTS per-minute quota (10 requests/minute/model) was exhausted. Unlike the BUG-024 from 03-03 (which was the 100 req/day limit), this was the per-minute rate limit being hit during normal conversation:

```
14:53:18 WARN: failed to generate TTS completion, retrying in 0.1ms
    error: 429 RESOURCE_EXHAUSTED
    "Quota exceeded for metric: generativelanguage.googleapis.com/generate_requests_per_model,
     limit: 10, model: gemini-2.5-flash-tts"
    "Please retry in 41.095601987s."
```

The retry logic (0.1ms → 2000ms → 2000ms) was far too aggressive — Google requested a 41-second backoff but the agent retried within milliseconds, generating a storm of 429s. Each failed TTS synthesis for each LLM text chunk emitted a separate error, quickly overwhelming the error threshold:

```
14:53:18–14:54:15  35 TTS retry warnings (429)
14:53:23–14:54:15  7 Pipeline errors (from TTS exhaustion)
14:53:27           "Queue is closed" cascade → ERR_UNHANDLED_ERROR
```

**Cascade mechanism:**
1. LLM response streams text in chunks (sentences)
2. Each chunk triggers a parallel TTS API call
3. ALL calls hit 429 simultaneously
4. Retry logic retries within ms instead of respecting the 41s backoff
5. Each retry hits 429 again → error multiplication
6. `maxUnrecoverableErrors` threshold exceeded → Pipeline errors → session degradation

**Key difference from 03-03:** The BUG-024 fix from `658182b` (which prevents AgentSession death on TTS 429) IS deployed and working — the session survived the 429 cascade. However, the user experience was severely degraded (no voice output for minutes).

**Fix path:** TASK-015 (Tiered TTS) — implement local TTS fallback when cloud rate limits are hit. Also: respect Google's `retryDelay` header instead of using fixed retry intervals.

### BUG-029 (NEW): EOU Detection AbortError — Turn Detector Inference Cancelled (All Sessions)

**Frequency:** 22 occurrences across all 5 sessions
**Impact:** MEDIUM — the error is logged but does not kill the session. Turn detection appears to recover on next utterance.

**What happened:**

The end-of-utterance (EOU) turn detector model's inference is aborted mid-computation, producing `AbortError: The operation was aborted` from `audio_recognition.js:302`:

```
15:10:06 ERROR: Error in EOU detection task:
    err: DOMException: "The operation was aborted."
    at cancel@.../utils.js:327:26
    at runEOUDetection@.../audio_recognition.js:302:51
```

This happens in two scenarios:
1. **FINAL_TRANSCRIPT arrives during EOU inference** — the STT produces a final transcript while the turn detector is still evaluating the previous interim, causing the previous inference to be cancelled via AbortController.
2. **VAD START_OF_SPEECH during EOU wait** — the user starts speaking again before the EOU timeout resolves, aborting the pending detection via `createVadTask`.

**Analysis:** These AbortErrors appear to be a normal consequence of the LiveKit agents SDK's cancellation-via-AbortController pattern. The errors are noisy but likely non-impactful — the turn detector runs again on the next speech event. However, 22 occurrences in 90 minutes suggests the error logging level should be reduced from ERROR to DEBUG in the SDK, or the abort case should be handled without throwing.

**Status:** Likely an SDK-level issue. Non-blocking but noisy.

### BUG-022: Total Downstream Failure Caused by OpenClaw Lane Lock (CRITICAL)

**User Observation (07:28 PST):** Cannot see text or hear TTS, but STT (client→hub) is still working.

**Root Cause: OpenClaw session lane lock after network drop.**

BUG-022 (the user-visible symptom) and BUG-030 (the gateway-side cause) are the same bug. A network disruption at 15:08:43 UTC interrupted an in-flight LLM request, leaving the OpenClaw session lane permanently locked. Every subsequent LLM request to the same session key returned **0 chunks**.


**The lock persisted across 3 agent sessions** (sessions 3, 4, and 5) because they all shared the same OpenClaw session key (`guest_user-1772610061397`). The gateway never released the lane.

**What the user experienced:**

A triple failure — all three downstream paths broken simultaneously:
1. **No transcript** — data channel broken (BUG-020, partial WebRTC failure)
2. **No TTS audio** — Gemini daily quota exhausted (BUG-024, "retry in 8h51m")
3. **No LLM content** — 0 chunks from every request (BUG-030, lane lock)

The user kept trying: "Are we still there, Clutch?", "Hello?", "Alright.", "I think I lost you now. For reals." — the agent received all of this via STT but had nothing to send back. The LLM was returning empty streams, TTS was 429'd, and the data channel was down.

**Gateway diagnostic (08:14 PST):**
```
[diagnostic] lane wait exceeded: lane=session:agent:main:openai-user:guest_user-1772610061397
  waitedMs=10805 queueAhead=0
```

**Root cause:** The voice agent does not use AbortController to cancel in-flight LLM requests when the WebRTC connection drops. The `departure_timeout` (120s) keeps the agent alive while the HTTP request to OpenClaw hangs. OpenClaw's lane serialization system keeps the session lane locked waiting for the stale turn to resolve. New turns queue behind the lock and time out with 0 chunks. The lock survives across agent session restarts because it's tied to the session key, not the agent process.

**Fix path (TASK-016):**
1. When participant disconnects, abort any in-flight `fetch()` to the OpenClaw gateway via AbortController
2. The gateway detects the aborted SSE stream (client TCP RST) and releases the lane lock
3. On reconnect, the agent starts fresh without a stale lock

**Fix applied:** `0a18fbd` — Thread `LLMStream.abortController.signal` through to `OpenClawClient.chat()` and `NanoclawClient.chat()` via `AbortSignal.any()`. When the SDK calls `LLMStream.close()` on disconnect, the in-flight `fetch()` is immediately aborted (TCP RST), releasing the OpenClaw lane. External-signal aborts return cleanly instead of rethrowing.

**Related:** TASK-016 (lane management), BUG-020 (transcript loss), BUG-024 (TTS rate limit)

### BUG-030 (NEW): Piper TTS Fallback Returns HTTP 404 (HIGH)

**First seen:** 06:02:41 UTC (22:02 PST)
**Frequency:** Every TTS fallback attempt
**Impact:** Complete loss of voice output — both Gemini (daily quota exhausted) AND Piper (wrong endpoint) fail simultaneously.

**What happened:**

When Gemini TTS hit its 100 req/day limit, the FallbackAdapter switched to the Piper TTS sidecar. But Piper returned HTTP 404 on every request:

```
INFO:werkzeug:127.0.0.1 - POST /synthesize HTTP/1.1 404
```

**Root cause:** The `piper-tts.ts` implementation had two mismatches with the Piper HTTP server API:
1. **Wrong endpoint:** Sent `POST /synthesize` — Piper only has a route at `/` (root)
2. **Wrong body format:** Sent `Content-Type: application/json` with `{"text":"..."}` — Piper expects `Content-Type: text/plain` with raw text body

The Piper server (`waveoffire/piper-tts-server`) uses a simple Flask app with a single route at `/` that reads `request.data.decode("utf-8")`. The `voice` parameter was also never supported per-request (Piper loads the voice model at startup via `--model` CLI arg).

**Fix applied:** Changed `piper-tts.ts` to `POST /` with `text/plain` body. Removed unused per-request `voice` parameter. Tests updated. Verified with curl: `HTTP 200, 54KB WAV`.

### BUG-031 (NEW): WiFi→Cellular Transition — Client Uses LAN IP (HIGH)

**First seen:** 06:02:46 UTC (22:02 PST)
**Frequency:** Every WiFi→cellular transition
**Impact:** Complete inability to reconnect until back on WiFi. User was disconnected for ~10 minutes.

**Tester context:** "Walked out of the house, basically as soon as I got a little away from the WiFi it stopped working. Unable to reconnect to room until I got back to the house about 10 or 15 minutes later."

**Client log evidence:**

```
22:01:44 [Fletcher] Connecting to ws://192.168.87.59:7880    ← LAN IP
22:01:45 [Fletcher] Connected to room
22:02:44 [Fletcher] SDK reconnect attempt 1/10
22:02:46 [Fletcher] Connection failed: Network is unreachable, address = 192.168.87.59
                    network=cellular
22:07:53 [Fletcher] Connecting to ws://192.168.87.59:7880    ← still trying LAN IP on cellular
22:09:34 [Fletcher] Connection failed: Software caused connection abort
22:10:06 [Fletcher] Connecting to ws://192.168.87.59:7880    ← app restarted, still LAN IP
22:12:26 [Fletcher] Connection failed: Connection timed out (2 min timeout!)
22:12:28 [Fletcher] Connecting to ws://192.168.87.59:7880    ← back on WiFi
22:12:36 [Fletcher] Connected to room                        ← finally works
```

**Root cause:** The client is configured with the server's LAN IP (`192.168.87.59`), not the Tailscale IP. When the phone leaves WiFi range, the LAN IP becomes unreachable from cellular. Even though Tailscale is running on both the phone and server, the app never uses the Tailscale address.

**The reconnect loop is also wasteful:** 5 fast reconnect attempts + SDK retries, all to an unreachable address. The client burns through its reconnect budget before even having a chance of success.

**Fix path:** The client should connect via the Tailscale IP/hostname instead of the LAN IP when on cellular. This could be:
1. Always use Tailscale IP (works from both WiFi and cellular, slightly higher latency on LAN)
2. Detect network type and switch URL (WiFi → LAN IP, cellular → Tailscale IP)
3. Try both in parallel and use whichever connects first

**Related:** BUG-028 (session instability), tasks/09-connectivity/

1. **Lane lock is the #1 issue (BUG-022)** — a single network drop at 07:08 AM permanently killed LLM responses for the rest of the morning. 12 of 14 subsequent streams returned 0 chunks across 3 different agent sessions. The lock persists via the shared session key.
2. **Driving scenario is brutal for connectivity** — cellular handoffs between towers create frequent brief gaps that the Tailscale+WebRTC stack cannot handle gracefully.

6. **EOU AbortErrors are noisy but harmless** — 22 errors that don't impact functionality. SDK should handle this more gracefully.
