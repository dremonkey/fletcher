### BUG-021: WiFi‚ÜíCellular Transition Causes 55s Audio Track Publish Timeout + Audio Route Disruption (HIGH)

**First seen:** 16:55:05 UTC
**Frequency:** Every WiFi‚Üícellular transition while walking
**Impact:** Bluetooth audio drops to speaker for ~55 seconds. User must toggle Bluetooth to recover audio routing.

**What happened:**

Walking away from WiFi range, the phone switched to cellular. Instead of an ICE migration, the Flutter LiveKit SDK created a new session (DUPLICATE_IDENTITY), killing the old one:

```
16:55:05 removing duplicate participant  user-1772517644087  reason: DUPLICATE_IDENTITY
16:55:05 WARN  dtls timeout on all 4 data channels (old session)
16:55:05 starting RTC session  network: "cellular"
16:55:07 participant active  connectTime: 1.9s  (vs 503ms on WiFi)
16:55:38 ERROR  supervisor error: "publish time out"  trackID: TR_AMLh7DykUngWDV
16:56:00 mediaTrack published  user microphone  (55 SECONDS after session start)
```

The 55-second audio track publish timeout likely disrupted OS-level audio routing ‚Äî when the audio session was torn down and re-established, Android switched from BT SCO to speaker. The user had to manually toggle Bluetooth to restore the correct audio route.

A second DUPLICATE_IDENTITY churn happened at 16:58:43 with another DTLS timeout cascade, plus a CLIENT_REQUEST_LEAVE firing simultaneously.

**Root cause:** Tailscale WireGuard tunnel re-establishment on cellular takes 30-55s. SDK creates new participant (DUPLICATE_IDENTITY), signaling reconnects in 1.9s, but DTLS handshake for media transport fails and retries until tunnel is ready. Confirmed by comparing transition #1 (55s, tunnel cold) vs #2 (3s, tunnel warm).

**Related:** BUG-015 (WiFi‚Üí5G session kill), BUG-017 (BT audio route changes)

### BUG-022: Android Background Microphone Silencing

**Severity:** HIGH
**Status:** RCA COMPLETE
**Description:** When Fletcher goes to background (screen lock, accidental swipe, task switch), Android 14+ silences the microphone via `AppOps` policy within 5 seconds. The app lacks a foreground service with `FOREGROUND_SERVICE_MICROPHONE` type.
**Root cause:** Missing `FOREGROUND_SERVICE_MICROPHONE` permission and service declaration. Android enforces while-in-use restrictions on `RECORD_AUDIO` ‚Äî silences mic within 5s of backgrounding.

### BUG-023: Empty System Message Causes LLM Backend Hang ‚Äî Zero Responses (CRITICAL)

**Severity:** CRITICAL
**First seen:** 2026-03-04 00:03:40 UTC (post-fix deployment session)
**Frequency:** Every request ‚Äî 100% failure rate
**Impact:** Agent connects, hears user, sends request to backend, but never receives any response tokens. Pondering indicators cycle indefinitely. Complete voice session failure.

**What happened:**

After deploying fixes for BUG-019 through BUG-022, the voice agent accepted connections and processed speech normally (STT, VAD, EOU all functional). However, every LLM request returned 0 content chunks despite the backend responding 200 OK:

```
00:03:40 POST /v1/chat/completions  200 OK (fetchLatency=3ms)
00:03:40 pondering: "Considering..." streamId=s_1
... (80+ seconds of rotating pondering messages) ...
00:05:03 stream complete, 0 chunks in 94860ms
```

Four consecutive streams (s_1 through s_4) all produced 0 chunks. The backend accepted each request instantly (2-3ms) but the SSE body never contained any `data:` lines.

**Investigation:**

Direct curl to the same gateway endpoint with the same session key returned tokens immediately:
```
$ curl http://localhost:18789/v1/chat/completions -d '{"messages":[{"role":"user","content":"hello"}],"stream":true,"user":"guest_user-1772581291121"}'
data: {"choices":[{"delta":{"content":"Hello! Looks like the connection is solid again"}}]}
```

The difference: the voice agent sends `instructions: ''` (empty string), which the LiveKit SDK converts to a ChatMessage with `role=system` and empty `textContent`. In `llm.ts`, the conversion `item.textContent || undefined` evaluates `"" || undefined` ‚Üí `undefined`, which JSON.stringify omits entirely. The resulting payload:

```json
{"messages": [{"role": "system"}, {"role": "user", "content": "Inkleach."}], "stream": true}
```

A system message with **no content field** causes the OpenClaw gateway (and its upstream LLM) to hang indefinitely ‚Äî it opens the SSE stream but never produces tokens.

Confirmed with targeted curl tests:
- `{"role":"system"}` (no content) ‚Üí **hangs** (timeout)
- `{"role":"system","content":""}` (empty string) ‚Üí **hangs** (timeout)
- `{"role":"system","content":" "}` (space) ‚Üí **works** (immediate response)
- No system message at all ‚Üí **works** (immediate response)

**Root cause:** `packages/livekit-agent-ganglia/src/llm.ts:232` ‚Äî `content: item.textContent || undefined` drops empty string content from the system message. The gateway/upstream LLM cannot handle a system message with no content field.

**Fix:** Added guard at line 230: `if (item.role === 'system' && !item.textContent) continue;` ‚Äî skip empty system messages entirely rather than sending a malformed one.

**Commit:** `ae94d70` fix(ganglia): skip empty system messages that cause LLM backend hang

### BUG-023: Empty System Message Causes LLM Backend Hang

**Severity:** CRITICAL
**Status:** FIXED (`ae94d70`)
**Description:** Voice agent sends `instructions: ''` which creates a system message with no content field (`{"role":"system"}`). The OpenClaw gateway/upstream LLM hangs indefinitely on this malformed message ‚Äî accepts with 200 OK but produces zero SSE tokens.
**Root cause:** `llm.ts` line 232: `content: item.textContent || undefined` drops empty string. JSON serialization omits `undefined`, producing a content-less system message.
**Fix:** Skip empty system messages: `if (item.role === 'system' && !item.textContent) continue;`
