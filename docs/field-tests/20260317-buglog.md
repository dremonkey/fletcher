## Detailed Entries

### BUG-051: Tool-call status not surfaced during relay chat (MED)

**First seen:** 09:39 PDT (16:39 UTC)
**Frequency:** Consistent during Chat Mode (text-only) tool use
**Impact:** "Black hole" UX during long processing; directly causes BUG-050 timeouts because the user/client has no visibility into progress.

**Symptoms:** When in text-mode (Chat Mode), tool-calls (reading, editing, etc.) do not emit status updates to the mobile UI like they do in Voice Mode. The user sees a blank state until the final response is delivered, triggering a timeout if the task is long.

**Analysis:** Voice Mode uses the `StatusBar` and `DiagnosticsBar` wired to data channel events from the voice-agent. Chat Mode routes through the Relay via JSON-RPC, and the Relay may not be forwarding interim `tool_call` or `status` notifications from the ACP subprocess to the mobile data channel. If we surface these, we can reset the client-side timeout on every "pulse" of activity.

**Root cause:** Two issues: (1) `negotiateSessionConfig()` in `relay-bridge.ts` looked up desired config values by `option.category`, but OpenClaw's `verbose_level` config option has no `category` field — so tool verbosity was never negotiated and remained `"off"`, meaning OpenClaw never emitted `tool_call`/`tool_call_update` events; (2) `acp-session-config.json` had no `verbose_level` entry. The `_meta: { verbose: true }` in `session/new` is opaque metadata and does not control OpenClaw's tool event emission. The mobile-side parser (`AcpUpdateParser`), UI (`ToolCallCard`), and relay forwarding were all already wired — the events simply never arrived.

**Fix:**
1. `relay-bridge.ts`: `negotiateSessionConfig()` falls back to `option.id` when `option.category` is absent (`const key = option.category ?? option.id`).
2. `acp-session-config.json`: Added `"verbose_level": "on"` under `"openclaw"`.
3. `relay_chat_service.dart`: Added `_resetPromptTimer()` — resets the 30s timeout on every valid incoming event (also fixes BUG-050).
4. `livekit_service.dart`: Passes `activeToolCalls: const []` at prompt start/complete/error to clear stale tool-call UI state.

**Status:** FIXED

### BUG-050: "Prompt timed out" UX error during long ACP tasks (MED)

**First seen:** 09:35 PDT (16:35 UTC)
**Frequency:** Observed during complex/long-running ACP tasks
**Impact:** Confusing UX; indicates failure when work is actually continuing in the background.

**Symptoms:** The mobile app displays a "Prompt timed out" error message even though the backend (Relay/ACP) is still processing the request.

**Analysis:** This is likely a client-side timeout in `RelayChatService.sendPrompt()` or a relay-side timeout that fires before the ACP subprocess finishes a heavy task (like Claude Code doing a repo-wide review). We recently added a 30s timeout to `sendPrompt()` in BUG-045 to fix ghost data channels, but that might be too aggressive for the "Chat Mode" use case where sub-agents can take minutes. 

**Root cause:** The 30s `promptTimeout` in `RelayChatService.sendPrompt()` was a flat wall-clock timer set once at prompt send. Tool-call and text-delta events arriving during processing never reset it, so multi-tool ACP tasks exceeding 30s total always triggered a false timeout — even though events were streaming in.

**Fix:** Resolved by BUG-051 fix. `_resetPromptTimer()` in `RelayChatService` converts the timeout to an *inactivity* timer: every valid incoming event (`tool_call`, `text_delta`, `thinking_delta`, `usage_update`, `user_message`) resets the 30s clock. The timeout only fires after 30s of silence.

**Status:** FIXED — addressed by BUG-051 fix.

### BUG-049: No network error on cold start; manual reconnect failed (HIGH)

**First seen:** 07:49 PDT (14:49 UTC)
**Frequency:** Observed once on cold start
**Impact:** App unusable on first launch. Manual reconnect button unresponsive.

**Symptoms:** User opened the app and immediately saw a "no network" error. Tapping the reconnect button did nothing. After a force-quit and relaunch, the app connected successfully.

**Task:** [`tasks/09-connectivity/096-cold-start-connection-failure.md`](../../tasks/09-connectivity/096-cold-start-connection-failure.md)

**Root cause:** Two compounding issues: (1) `connectWithDynamicRoom()` starts TCP-racing URLs ~1s after Dart VM boot, before Android's network routes are functional — all connections time out; (2) `tryReconnect()` requires cached `_url`/`_token` from `connect()`, but `connect()` was never reached, so tap-to-retry silently fails via null-credential guard in `_reconnectRoom()`.

**Status:** RCA COMPLETE — task 096 filed with fix plan.
