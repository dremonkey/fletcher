### BUG-014: NO_REPLY — Gateway TTS Generation Blocks Voice Pipeline (MEDIUM)

**First seen:** 23:00 UTC
**Frequency:** Intermittent — triggered by upstream LLM API limit exhaustion
**Impact:** Agent receives STT transcripts but never replies. User hears nothing.
**Status:** Resolved

**Symptoms:**
- STT working correctly — user speech transcribed and sent to OpenClaw gateway
- Gateway responds `200 OK` in 1-4ms (suspiciously fast) with 0 content chunks
- LLM stream completes with `chunks=0`, `ttftMs: -1`, `tokensPerSecond: 0`
- Agent transitions thinking → listening without ever speaking

**Root cause:** OpenClaw gateway started generating TTS server-side through the `/v1/chat/completions` endpoint, which it should never do — Fletcher handles its own TTS pipeline via Google TTS. The gateway's TTS generation consumed the response, resulting in 0 text chunks being streamed back to Fletcher.

**Resolution:** Asked OpenClaw to stop generating TTS on the completions endpoint. Once disabled, text streaming resumed normally and the voice pipeline worked again.

### BUG-015: Agent Binds to Stale Participant Identity on Rejoin (HIGH)

**First seen:** 00:07 UTC (2026-03-08)
**Frequency:** Reproducible — occurs whenever a user drops and rejoins the same room with a new participant identity
**Impact:** Agent becomes unresponsive. User's audio is silently ignored; agent shows as "disconnected" in the app.
**Status:** Resolved

**Symptoms:**
- App reconnects to the same room after a drop (e.g., app relaunch, network blip)
- LiveKit assigns a new participant identity (e.g., `user-...457115` → `user-...470907`)
- Agent remains bound to the old identity — new user's audio is silently dropped
- App diagnostics show `AGENT: disconnected`, `SYS: RECONNECTING`
- Voice-agent logs show `Participant disconnected — waiting for reconnect (departure_timeout=120s)` for the old identity

**Root cause:** `@livekit/agents` `RoomIO` binds audio I/O to a specific participant identity at session start. When a participant disconnects, the agent enters a reconnect-wait state expecting the **same identity** to return. If the same user rejoins with a **new identity** (which LiveKit assigns by default), the agent treats them as a stranger and ignores their audio tracks (`onTrackSubscribed` short-circuits on identity mismatch).

**Relevant code path:**
- `RoomIO.setParticipant(identity)` → `ParticipantAudioInputStream.setParticipant(identity)` — binds mic input to one identity
- `_input.js:onTrackSubscribed` — `this.participantIdentity !== participant.identity → return false`
- `room_io.js:onParticipantDisconnected` — resets `participantAvailableFuture` but agent waits for same identity

**Resolution:** Replaced ephemeral `user-<timestamp>` identity with a stable hardware device ID via `device_info_plus`. `SessionStorage.getDeviceId()` returns `device-<ANDROID_ID>` (Android) or `device-<identifierForVendor>` (iOS), ensuring the same physical device always reconnects with the same participant identity. The agent's `RoomIO` now recognizes the returning participant and rebinds correctly.

### BUG-028: Background Auto-Close Timer Not Starting on App Switch (MEDIUM)

**First seen:** 2026-03-07
**Frequency:** Consistent — timer never starts when switching away from Fletcher to another app
**Impact:** Voice session stays alive indefinitely when the user switches to another app, consuming resources and keeping the LiveKit room open. The 10-minute background timeout (task 019) is not firing.
**Status:** Open

**Expected behavior:** When the user switches from Fletcher to another app (e.g., via home button or app switcher), the 10-minute background disconnect timer should start. The foreground notification should update to show "Disconnecting in N min" and the session should auto-disconnect after 10 minutes of inactivity.

**Correct behavior preserved:** The timer should NOT start when the user locks the screen with Fletcher open, since the user may be talking via earbuds with the phone in their pocket.

**Actual behavior:** Switching to another app does not start the background timer. The session stays alive indefinitely. Lock screen behavior is correct (no timer).

**Analysis — code appears structurally intact:**
- `ConversationScreen` still mixes in `WidgetsBindingObserver` and registers via `WidgetsBinding.instance.addObserver(this)` in `initState()`
- `didChangeAppLifecycleState` still handles `AppLifecycleState.paused` → calls `ScreenStateService.isScreenLocked()` → calls `_liveKitService.onAppBackgrounded(isScreenLocked:)`
- `onAppBackgrounded()` in `LiveKitService` still has the timer logic with the `_room == null` early return and `isScreenLocked` guard
- `ScreenStateService` still uses the `com.fletcher.fletcher/screen_state` method channel → `KeyguardManager.isKeyguardLocked` on Android

**Possible root causes to investigate:**
1. **`_room` is null at pause time** — if the room reference is cleared or never set during the refactored connection flow, `onAppBackgrounded()` returns early on line 1290
2. **`isScreenLocked()` returning `true` incorrectly** — if the method channel is returning true even when the screen is not locked, the timer would be skipped (though the catch block defaults to `false`, so a channel failure would not cause this)
3. **Lifecycle event not firing** — the `WidgetsBindingObserver` may not be receiving `paused` events if something in the widget tree changed during the UI refactor (e.g., a new parent widget intercepting lifecycle)
4. **Async race** — `didChangeAppLifecycleState` is `async` and awaits `isScreenLocked()`. If the app is paused before the await completes, the callback to `onAppBackgrounded` may never execute

**Regression from:** UI refactor (branch `ui-refactor`). The feature was working prior to the refactor; task 019 was closed as complete.

**Related code:**
- `apps/mobile/lib/screens/conversation_screen.dart:57-73` — lifecycle handler
- `apps/mobile/lib/services/livekit_service.dart:1289-1316` — `onAppBackgrounded()`
- `apps/mobile/lib/services/screen_state_service.dart` — lock detection
- `apps/mobile/android/.../MainActivity.kt` — native keyguard check

**Related task:** task 019 (background-session-timeout, closed)
