### BUG-012: Artifact Cards Pool Together Instead of Anchoring to Message (MEDIUM)

**First seen:** 22:27 UTC
**Frequency:** Consistent — all artifacts collect together in the transcript
**Impact:** When multiple artifacts arrive across different messages, they all stack together at the end of the transcript rather than appearing inline with the message that produced them.

**Screenshot:** [`20260307-artifact-pooling.png`](./20260307-artifact-pooling.png)

**Expected behavior:** Each `[ARTIFACT: ...]` card should be anchored chronologically to the agent message it arrived with. If an artifact arrives during/after a specific agent turn, it should render below that message in the chat scroll.

**Actual behavior:** All 10 artifact cards are grouped together, disconnected from their originating messages. The transcript shows the latest agent text at the top, then a wall of artifact cards below.

**Analysis:** Artifacts are likely stored in a flat `ConversationState.artifacts` list and rendered separately from the message list, rather than being associated with a specific message entry. The fix requires tying each artifact to a message ID or timestamp so the transcript can render them inline.

**Related task:** TASK-023 (artifact–message association)

### BUG-014: NO_REPLY — Gateway TTS Generation Blocks Voice Pipeline (MEDIUM)

**First seen:** 23:00 UTC
**Frequency:** Intermittent — triggered by upstream LLM API limit exhaustion
**Impact:** Agent receives STT transcripts but never replies. User hears nothing.
**Status:** Resolved

**Symptoms:**
- STT working correctly — user speech transcribed and sent to OpenClaw gateway
- Gateway responds `200 OK` in 1-4ms (suspiciously fast) with 0 content chunks
- LLM stream completes with `chunks=0`, `ttftMs: -1`, `tokensPerSecond: 0`
- Agent transitions thinking → listening without ever speaking

**Root cause:** OpenClaw gateway started generating TTS server-side through the `/v1/chat/completions` endpoint, which it should never do — Fletcher handles its own TTS pipeline via Google TTS. The gateway's TTS generation consumed the response, resulting in 0 text chunks being streamed back to Fletcher.

**Resolution:** Asked OpenClaw to stop generating TTS on the completions endpoint. Once disabled, text streaming resumed normally and the voice pipeline worked again.

### BUG-015: Agent Binds to Stale Participant Identity on Rejoin (HIGH)

**First seen:** 00:07 UTC (2026-03-08)
**Frequency:** Reproducible — occurs whenever a user drops and rejoins the same room with a new participant identity
**Impact:** Agent becomes unresponsive. User's audio is silently ignored; agent shows as "disconnected" in the app.

**Symptoms:**
- App reconnects to the same room after a drop (e.g., app relaunch, network blip)
- LiveKit assigns a new participant identity (e.g., `user-...457115` → `user-...470907`)
- Agent remains bound to the old identity — new user's audio is silently dropped
- App diagnostics show `AGENT: disconnected`, `SYS: RECONNECTING`
- Voice-agent logs show `Participant disconnected — waiting for reconnect (departure_timeout=120s)` for the old identity

**Root cause:** `@livekit/agents` `RoomIO` binds audio I/O to a specific participant identity at session start. When a participant disconnects, the agent enters a reconnect-wait state expecting the **same identity** to return. If the same user rejoins with a **new identity** (which LiveKit assigns by default), the agent treats them as a stranger and ignores their audio tracks (`onTrackSubscribed` short-circuits on identity mismatch).

**Relevant code path:**
- `RoomIO.setParticipant(identity)` → `ParticipantAudioInputStream.setParticipant(identity)` — binds mic input to one identity
- `_input.js:onTrackSubscribed` — `this.participantIdentity !== participant.identity → return false`
- `room_io.js:onParticipantDisconnected` — resets `participantAvailableFuture` but agent waits for same identity

**Possible solutions (to investigate):**
- Use a stable device ID instead of LiveKit's auto-generated participant identity, so the same user always reconnects with the same identity
- Detect "sole human participant in room" and re-bind the agent to the new identity
- Have the Flutter app persist and reuse the same participant identity across reconnections

**Workaround:** Restart the voice-agent container (`docker compose restart voice-agent`) to clear stale state.
