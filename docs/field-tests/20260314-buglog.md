### BUG-024: Delivery failure for long agent responses via relay (HIGH)

**First seen:** 2026-03-14 00:15 PDT
**Frequency:** Recurring
**Symptom:** Long assistant responses (e.g., the task rundown) are appearing in the OpenClaw web UI but are not being delivered to the mobile client via the relay bridge. Short messages (like "Loud and clear") are delivered successfully.
**Impact:** Critical failure of the communication loop for complex information. User is forced to check the web UI to see the full response.
**Hypothesis:** The relay data channel or the ACP JSON-RPC bridge has a payload size limit or is timing out on larger buffers. Alternatively, the asymmetric WebRTC data channel degradation (similar to BUG-020) is triggered by the volume of data.
**Root cause:** Upstream ACP bug (openclaw#40693) — ACP subprocess returns `end_turn` with zero `session/update` notifications. The BUG-022 catch-up mechanism was triggered but its count-based dedup is broken: `forwardedChunkCount` (9) drifted above actual session history chunks (5), causing all replayed chunks to be skipped (`newChunks: -4`).
**Task:** [`tasks/22-dual-mode/067-relay-large-payload-delivery-failure.md`](../../tasks/22-dual-mode/067-relay-large-payload-delivery-failure.md)

### BUG-025: Relay crash — SIGTERM from unknown source, restart loop failure (MED)

**First seen:** 2026-03-14 00:41 PDT
**Frequency:** Observed once
**Symptom:** Relay logged "Shutting down..." and exited. User did NOT Ctrl+C. Subsequent restart attempts (via TUI/nohup) failed in a loop with `nohup: failed to run command 'bun': No such file or directory` and `libstdc++.so.6: cannot open shared object file`.

**Impact:** Relay went down and could not recover. Chat mode non-functional until manual intervention.

**Root cause (partial):** The relay received SIGTERM from an external source. Two likely causes:

1. **TUI parent died or restarted.** The TUI (`packages/tui/src/services.ts:294-328`) spawns the relay as a child process and sends SIGTERM to all children in its cleanup handler. If the TUI process crashed, was killed, or the nix shell exited, all child processes receive SIGTERM.

2. **Unhandled promise rejection (latent risk).** The relay has **zero** `process.on('unhandledRejection')` or `process.on('uncaughtException')` handlers (`apps/relay/src/index.ts`). Any unhandled rejection would crash the process — but this would NOT produce the "Shutting down..." log, so it wasn't the cause this time.

**Restart loop failure:** The TUI or an external wrapper tried to restart the relay using `nohup bun run ...`. This failed because `bun` and `libstdc++.so.6` are provided by the nix shell — once the nix environment is gone, the native LiveKit binding (`@livekit/rtc-node`) can't load. No `nohup` exists in the Fletcher codebase — the restart mechanism is external.

**Fix:** Add `unhandledRejection` and `uncaughtException` handlers to the relay to (a) log the error and (b) prevent silent crashes. This doesn't fix the external SIGTERM but closes the latent crash vector.

**Status:** RCA COMPLETE

### BUG-026: Upstream 429 rate limit produces opaque error to mobile (MED)

**First seen:** 2026-03-14 00:41 PDT
**Frequency:** Whenever upstream LLM (Gemini) hits per-minute token quota
**Symptom:** OpenClaw hit Gemini 3 Flash's 2M token/min input quota. The 429 was logged on the OpenClaw side but the error that reached the relay was a generic JSON-RPC `-32603 Internal error`. Mobile would show the opaque fallback: `"Error: JSON-RPC error -32603: Internal error"`.

**Impact:** User has no idea why their prompt failed. No retry guidance, no rate-limit indication. In this specific case the user cancelled before the error propagated, but the general flow is broken for any upstream error.

**Root cause — error detail stripping at two layers:**

1. **ACP client (`packages/acp-client/src/client.ts:384-386`):** Converts the full JSON-RPC error to a flat string: `new Error(\`JSON-RPC error ${err.code}: ${err.message}\`)`. The `data` field (which contains `details: "Gateway disconnected..."` or rate-limit info) is **discarded**.

2. **Relay bridge (`apps/relay/src/bridge/relay-bridge.ts:405-412`):** Catches the Error and forwards `err.message` (the flat string) to mobile with a hardcoded `INTERNAL_ERROR` code. The original ACP error code is lost.

3. **Mobile (`apps/mobile/lib/services/livekit_service.dart:1554-1561`):** Has a code-to-message mapping but only handles three specific codes (-32003, -32010, -32011). Everything else falls through to `"Error: $message"`.

**Error flow:**
```
Gemini 429 → OpenClaw logs it → ACP sends {code: -32603, message: "Internal error", data: {details: "..."}}
→ acp-client strips `data`, flattens to Error string
→ relay-bridge catches, re-wraps with INTERNAL_ERROR code
→ mobile shows "Error: JSON-RPC error -32603: Internal error"
```

**Fix plan:**

1. **ACP client:** Create a typed `JsonRpcError` class that preserves `code`, `message`, and `data` (instead of a flat `Error` string).
2. **Relay bridge:** Forward the original ACP error code. Detect rate-limit patterns in `data.details` (keywords: "429", "quota", "rate limit", "RESOURCE_EXHAUSTED") and use a new `RATE_LIMITED = -32029` error code with the retry delay if available.
3. **RPC errors:** Add `RATE_LIMITED` constant.
4. **Mobile:** Add `-32029` to `_relayErrorMessage()`: `"Rate limited — try again shortly"`.

**Task:** `tasks/22-dual-mode/068-relay-error-detail-forwarding.md`

**Status:** RCA COMPLETE

### BUG-027: Voice pipeline dies silently after user-away timeout (HIGH)

**First seen:** 2026-03-14 08:20 PDT (room `onyx-well`)
**Frequency:** Reproducible — happened on first session after Docker restart
**Impact:** Voice agent appears connected but produces zero STT/TTS output. User speaks, client-side VAD fires, but agent never responds. Only fix is reconnecting to a new room.

**Timeline (room `onyx-well`):**
- 15:19:55 UTC: Agent joined, bootstrap deferred (waiting for mic-on)
- 15:28:22 UTC: `onTrackSubscribed` (8min delay — device track arrived late)
- 15:28:24 UTC: Bootstrap sent, ACP responded `end_turn` (correct)
- 15:28:40 UTC: "User away timeout triggered" — 15s of silence after bootstrap
- 15:28:40–16:02 UTC: **35 minutes of dead pipeline** — only memory stats, zero VAD/STT events
- Restart to room `ridley-spur` fixed it immediately — fresh pipeline, STT/VAD worked

**Root cause — silent STT stream death with no recovery:**

1. **User-away timeout** (`@livekit/agents` agent_session.js:641-646): After 15 seconds of mutual silence, the agent sets user state to `"away"`. This is aggressive for a first interaction where the user hasn't spoken yet.

2. **STT stream can die silently** (`audio_recognition.js:419`): The STT task's `for await` loop can exit when the stream reader is released. The error is caught by `isStreamReaderReleaseError()` and silently returned — no log, no recovery.

3. **No restart mechanism**: `AudioRecognition` tasks (VAD + STT) are created once in `_startSession()` and never restarted if they fail.

4. **Dead loop**: Once STT dies → no transcripts generated → `_onUserInputTranscribed()` never fires → user can never exit "away" state → pipeline stays dead forever.

**Why it's invisible:** Agent process stays alive (memory stats every 30s). No ERROR-level logs. The stream reader release is explicitly suppressed.

**Fix plan:**
1. Add explicit logging when VAD/STT tasks exit (even normally) — make silent death visible
2. Add a heartbeat/watchdog: if VAD+STT tasks both exit, auto-restart `AudioRecognition`
3. Consider extending `userAwayTimeout` (15s is aggressive) or making it configurable
4. Log at WARN level when `isStreamReaderReleaseError` suppresses an error

**Status:** RCA COMPLETE
