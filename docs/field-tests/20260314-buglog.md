### BUG-025: Relay crash — SIGTERM from unknown source, restart loop failure (MED)

**First seen:** 2026-03-14 00:41 PDT
**Frequency:** Observed once
**Symptom:** Relay logged "Shutting down..." and exited. User did NOT Ctrl+C. Subsequent restart attempts (via TUI/nohup) failed in a loop with `nohup: failed to run command 'bun': No such file or directory` and `libstdc++.so.6: cannot open shared object file`.

**Impact:** Relay went down and could not recover. Chat mode non-functional until manual intervention.

**Root cause (partial):** The relay received SIGTERM from an external source. Two likely causes:

1. **TUI parent died or restarted.** The TUI (`packages/tui/src/services.ts:294-328`) spawns the relay as a child process and sends SIGTERM to all children in its cleanup handler. If the TUI process crashed, was killed, or the nix shell exited, all child processes receive SIGTERM.

2. **Unhandled promise rejection (latent risk).** The relay has **zero** `process.on('unhandledRejection')` or `process.on('uncaughtException')` handlers (`apps/relay/src/index.ts`). Any unhandled rejection would crash the process — but this would NOT produce the "Shutting down..." log, so it wasn't the cause this time.

**Restart loop failure:** The TUI or an external wrapper tried to restart the relay using `nohup bun run ...`. This failed because `bun` and `libstdc++.so.6` are provided by the nix shell — once the nix environment is gone, the native LiveKit binding (`@livekit/rtc-node`) can't load. No `nohup` exists in the Fletcher codebase — the restart mechanism is external.

**Fix:** Add `unhandledRejection` and `uncaughtException` handlers to the relay to (a) log the error and (b) prevent silent crashes. This doesn't fix the external SIGTERM but closes the latent crash vector.

**Status:** RCA COMPLETE

### BUG-026: Upstream 429 rate limit produces opaque error to mobile (MED)

**First seen:** 2026-03-14 00:41 PDT
**Frequency:** Whenever upstream LLM (Gemini) hits per-minute token quota
**Symptom:** OpenClaw hit Gemini 3 Flash's 2M token/min input quota. The 429 was logged on the OpenClaw side but the error that reached the relay was a generic JSON-RPC `-32603 Internal error`. Mobile would show the opaque fallback: `"Error: JSON-RPC error -32603: Internal error"`.

**Impact:** User has no idea why their prompt failed. No retry guidance, no rate-limit indication. In this specific case the user cancelled before the error propagated, but the general flow is broken for any upstream error.

**Root cause — error detail stripping at two layers:**

1. **ACP client (`packages/acp-client/src/client.ts:384-386`):** Converts the full JSON-RPC error to a flat string: `new Error(\`JSON-RPC error ${err.code}: ${err.message}\`)`. The `data` field (which contains `details: "Gateway disconnected..."` or rate-limit info) is **discarded**.

2. **Relay bridge (`apps/relay/src/bridge/relay-bridge.ts:405-412`):** Catches the Error and forwards `err.message` (the flat string) to mobile with a hardcoded `INTERNAL_ERROR` code. The original ACP error code is lost.

3. **Mobile (`apps/mobile/lib/services/livekit_service.dart:1554-1561`):** Has a code-to-message mapping but only handles three specific codes (-32003, -32010, -32011). Everything else falls through to `"Error: $message"`.

**Error flow:**
```
Gemini 429 → OpenClaw logs it → ACP sends {code: -32603, message: "Internal error", data: {details: "..."}}
→ acp-client strips `data`, flattens to Error string
→ relay-bridge catches, re-wraps with INTERNAL_ERROR code
→ mobile shows "Error: JSON-RPC error -32603: Internal error"
```

**Fix plan:**

1. **ACP client:** Create a typed `JsonRpcError` class that preserves `code`, `message`, and `data` (instead of a flat `Error` string).
2. **Relay bridge:** Forward the original ACP error code. Detect rate-limit patterns in `data.details` (keywords: "429", "quota", "rate limit", "RESOURCE_EXHAUSTED") and use a new `RATE_LIMITED = -32029` error code with the retry delay if available.
3. **RPC errors:** Add `RATE_LIMITED` constant.
4. **Mobile:** Add `-32029` to `_relayErrorMessage()`: `"Rate limited — try again shortly"`.

**Task:** `tasks/22-dual-mode/068-relay-error-detail-forwarding.md`

**Status:** RCA COMPLETE

### BUG-027: Voice pipeline dies silently after user-away timeout (HIGH)

**First seen:** 2026-03-14 08:20 PDT (room `onyx-well`)
**Frequency:** Reproducible — happened on first session after Docker restart
**Impact:** Voice agent appears connected but produces zero STT/TTS output. User speaks, client-side VAD fires, but agent never responds. Only fix is reconnecting to a new room.

**Timeline (room `onyx-well`):**
- 15:19:55 UTC: Agent joined, bootstrap deferred (waiting for mic-on)
- 15:28:22 UTC: `onTrackSubscribed` (8min delay — device track arrived late)
- 15:28:24 UTC: Bootstrap sent, ACP responded `end_turn` (correct)
- 15:28:40 UTC: "User away timeout triggered" — 15s of silence after bootstrap
- 15:28:40–16:02 UTC: **35 minutes of dead pipeline** — only memory stats, zero VAD/STT events
- Restart to room `ridley-spur` fixed it immediately — fresh pipeline, STT/VAD worked

**Root cause — silent STT stream death with no recovery:**

1. **User-away timeout** (`@livekit/agents` agent_session.js:641-646): After 15 seconds of mutual silence, the agent sets user state to `"away"`. This is aggressive for a first interaction where the user hasn't spoken yet.

2. **STT stream can die silently** (`audio_recognition.js:419`): The STT task's `for await` loop can exit when the stream reader is released. The error is caught by `isStreamReaderReleaseError()` and silently returned — no log, no recovery.

3. **No restart mechanism**: `AudioRecognition` tasks (VAD + STT) are created once in `_startSession()` and never restarted if they fail.

4. **Dead loop**: Once STT dies → no transcripts generated → `_onUserInputTranscribed()` never fires → user can never exit "away" state → pipeline stays dead forever.

**Why it's invisible:** Agent process stays alive (memory stats every 30s). No ERROR-level logs. The stream reader release is explicitly suppressed.

**Fix plan:**
1. Add explicit logging when VAD/STT tasks exit (even normally) — make silent death visible
2. Add a heartbeat/watchdog: if VAD+STT tasks both exit, auto-restart `AudioRecognition`
3. Consider extending `userAwayTimeout` (15s is aggressive) or making it configurable
4. Log at WARN level when `isStreamReaderReleaseError` suppresses an error

**Status:** RCA COMPLETE

### BUG-029: WiFi→5G network switch causes unrecoverable disconnect (HIGH)

**First seen:** 2026-03-14 09:14 PDT
**Frequency:** Reproducible on WiFi→cellular transitions (5G→WiFi at 09:05 was fine)
**Impact:** Total connection loss. User must force-quit and restart the app to reconnect. Gets a new room (loses session context).

**What happened:**
- User was in room `ridley-spur` on WiFi
- WiFi→5G switch at ~16:14 UTC broke the WebRTC ICE connection
- The mobile client did NOT automatically reconnect via ICE restart
- LiveKit server-side: `ridley-spur` lingered until departure timeout closed it at 16:18:34 ("departure timeout")
- User force-quit, reopened app → connected to new room `calm-oasis` at 16:18:09 over cellular
- LiveKit shows device connected via T-Mobile cellular IPs (172.56.xxx CGNAT, 2607:fb91 IPv6)

**Why 5G→WiFi worked but WiFi→5G didn't:**
- 5G→WiFi: Device gains a local network that can reach the server directly — ICE can find the new path quickly
- WiFi→5G: Device loses the local network and moves to CGNAT cellular — NAT traversal is harder, ICE candidates change entirely, and the Tailscale tunnel may need to re-establish

**Server-side evidence:**
- LiveKit logged `"ice reconnected or switched pair"` for the relay and agent in `calm-oasis` (server-side participants reconnected fine)
- No ICE reconnection logs for the device participant in `ridley-spur` — the device never attempted to reconnect

**Revised analysis — likely cascade from BUG-027, not a standalone bug:**

The voice pipeline was **already dead for 8+ minutes** (BUG-027 user-away timeout at 16:05:57) before the network switch at ~16:14. The session was in a zombie state — no audio flowing, no STT/VAD processing. Key evidence:

1. **Client never attempted ICE restart** — zero reconnection attempts logged server-side. LiveKit marked `isExpectedToResume: false`.
2. **But the same client reconnected fine later** — in `calm-oasis`, two brief disconnects at 09:30 and 09:35 PDT recovered via ICE restart in 7-9 seconds.
3. **The difference**: those successful reconnects were during active sessions on the same interface. The `ridley-spur` disconnect happened during a dead pipeline, possibly with the app backgrounded/idle.

**Working theory:** BUG-027 (pipeline death) → no audio activity → app goes idle/background → network switch happens → app doesn't aggressively ICE-restart because it's not in an active state → DTLS times out → connection lost. If BUG-027 is fixed (pipeline stays alive), the app would be actively processing audio during network switches, and the SDK's existing reconnect logic would likely handle it.

**Next step:** Re-test WiFi→5G transitions after BUG-027 is fixed. If the problem persists with an active pipeline, then it's a genuine SDK/connectivity issue.

**Related tasks:** `tasks/09-connectivity/` (connection resilience epic)

### BUG-030: Agent linked to relay participant instead of device — voice pipeline deaf (HIGH)

**First seen:** 2026-03-14 22:42 UTC (room `thorn-peak`)
**Frequency:** Race condition — depends on participant join order
**Impact:** Voice mode completely non-functional. STT/VAD pipeline receives zero audio because it's listening to the relay participant (which publishes no audio tracks).

**Timeline (thorn-peak):**
- 22:42:55: Agent joined, `waitForParticipant()` resolved to `relay-thorn-peak` (relay arrived first)
- 22:42:55: `setting participant audio input: relay-thorn-peak` — **wrong target**
- 22:42:58: Device audio track subscribed (`onTrackSubscribed in _input`)
- 22:43:00: Bootstrap sent (voice mode activated), ACP responded correctly
- 22:43:00–22:43:14: Zero VAD/STT events — pipeline listening to relay (no audio)
- 22:43:14–22:43:20: Tester toggled mic off/on/off trying to fix it
- 22:44:02: Session closed (user left)

**Comparison with working sessions:**
- `deep-cucco` (worked): `setting participant: device-BP3A.251105.015`
- `ridley-crane` (worked): `setting participant: device-BP3A.251105.015`
- `thorn-peak` (broken): `setting participant: relay-thorn-peak`

**Root cause:** `ctx.waitForParticipant()` at `agent.ts:683` returns the first non-agent participant to join the room. The relay participant (`relay-*`) has `kind: STANDARD` (not `AGENT`), so when it joins before the device, the agent session links its audio input to the relay — which has no audio tracks. The STT/VAD pipeline then processes silence forever.

This is a **race condition**: in most sessions the device joins first (works), but when the relay connects faster (as in `thorn-peak`), voice mode is completely broken.

**Fix options:**
1. Filter `waitForParticipant()` to match only `device-*` identities
2. Use `ctx.waitForParticipant({ identity: ownerIdentity })` if `FLETCHER_OWNER_IDENTITY` is set
3. Skip participants with `relay-` prefix in the wait logic

**Fix applied:** Added `waitForDeviceParticipant()` helper that filters out `relay-*` identities. Commit `74ac17b`.

**Status:** FIXED

### BUG-031: Barge-in during bootstrap kills first voice responses (HIGH)

**First seen:** 2026-03-14 22:54 UTC (room `sandy-orchard`)
**Frequency:** Every session where user speaks within 2s of bootstrap
**Impact:** First 1-2 voice messages get no spoken response. User hears nothing.

**Timeline:**
- 22:54:17: Bootstrap (relay_1) sent to relay/ACP
- 22:54:18: User spoke ("Hey there.") — VAD START_OF_SPEECH — **bootstrap interrupted by barge-in**
- 22:54:18: Bootstrap stream cancelled (close() called)
- 22:54:19: "Hey there." (relay_2) sent → ACP returned `end_turn` with no text (bootstrap "do not reply" still in effect in session)
- 22:54:33: "Okay." (relay_3) sent → cancelled at 22:54:34 by another barge-in
- 22:54:35: ACP DID respond with "Ready when you are..." → but stream already closed → `stream closed, ignoring message`
- 22:54:37: Fourth prompt eventually worked → "playout completed without interruption" at 22:54:43
- 22:54:47: `Error pumping input stream from MultiInputStream` (×3) — audio input pipeline broke after mic toggling

**Root cause:** Two interacting problems:
1. **Bootstrap is interruptible**: The `AgentActivity.pipelineReply` for bootstrap is treated like any other LLM turn. When the user speaks during the 2s bootstrap delay, barge-in cancels it. The subsequent user prompt lands in an ACP session where "do not reply" was the first (and possibly only processed) instruction.
2. **Cancelled streams keep pondering**: After `close()` is called on a relay stream, the pondering interval timer isn't cleared. The stream emits pondering phrases indefinitely, masking the fact that it's dead.

**Fix applied:**
1. Bootstrap now uses `generateReply({ allowInterruptions: false })` — user speech during bootstrap is queued by the SDK and processed after bootstrap completes
2. Agent emits `{ type: "bootstrap", phase: "start"|"end" }` events so the UI can show a "Connecting..." state (task 072)
3. `RelayChatStream.close()` now rejects the prompt promise immediately, terminating `run()` and clearing the pondering timer — previously the timer ran for up to 120s after barge-in

**Status:** FIXED

### BUG-032: Hold mode disconnect leaves status stuck in "reconnecting" — no redispatch (HIGH)

**First seen:** 2026-03-14 22:55 UTC (room `sandy-orchard`)
**Frequency:** Every hold mode timeout
**Impact:** After 60s idle timeout, user cannot re-engage voice mode by toggling mic. Agent never comes back.

**Root cause — race condition in event ordering:**
1. Agent sends `session_hold` event → client sets `_holdModeActive = true`
2. Agent disconnects (500ms later)
3. `TrackUnsubscribedEvent` fires BEFORE `ParticipantDisconnected` (SDK event ordering)
4. Track unsub handler checks `agentPresenceService.state` — still `agentPresent` (ParticipantDisconnected hasn't fired yet) → `isAgentAbsent = false` → sets `ConversationStatus.reconnecting`
5. `ParticipantDisconnected` fires → sets agentPresence to `agentAbsent`
6. Status stuck at `reconnecting` — never cleared
7. User unmutes → `toggleMute()` checks `_state.status != ConversationStatus.reconnecting` → FALSE → dispatch blocked ("deferring dispatch until reconnected")

**Fix applied:**
1. Track unsub handler: also skip reconnecting status when `_holdModeActive` is true
2. ParticipantDisconnected handler: when hold mode, explicitly set status to `idle` so dispatch path is unblocked

Commit `a5705f6`.

**Status:** FIXED
