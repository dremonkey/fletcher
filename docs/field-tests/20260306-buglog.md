### BUG-002: Agent DTLS timeout — dropped from room (HIGH)

**First seen:** 10:51:18 UTC
**Frequency:** Once (fatal for the session)
**Impact:** Agent silently left the room. Client remained connected to LiveKit for ~5 more hours (until 15:45) but had no agent to talk to. No error shown to user.

**Log excerpt:**
```

  participant: "agent-AJ_sXTgEV4ERXQh"
  label: "_lossy"
  error: "dtls timeout: read/write timeout: context deadline exceeded"
```

**Analysis:** The agent process was so memory-starved (2.6 GB RSS) that it couldn't maintain WebRTC connections. DTLS timeouts killed both data channels. The agent container stayed running (Docker reports 0 restarts) but the process was zombie — no further logs emitted until a new job at 16:12.

The client reconnected at 13:58 (ICE reconnect) and again at 15:45, but LiveKit showed `numParticipants: 0` — confirming the agent was gone from the room.

**Root cause:** Consequence of BUG-001 memory leak.

### BUG-004: Client can't connect without Tailscale active (MEDIUM)

**First seen:** ~17:37 UTC
**Frequency:** Every connection attempt when Tailscale is inactive on the phone
**Impact:** Client unable to reach LiveKit server despite being on the same WiFi network as the server.

**Analysis:** The URL resolver (`apps/mobile/lib/services/url_resolver.dart`) unconditionally prefers the Tailscale URL when `LIVEKIT_URL_TAILSCALE` is configured:

```dart
if (tailscaleUrl != null && tailscaleUrl.isNotEmpty) {
  return ResolvedUrl(url: tailscaleUrl);  // Always used, even if unreachable
}
```

When Tailscale is not active on the phone, the Tailscale IP (`100.87.219.109`) is unreachable. The LAN URL (`192.168.87.59:7880`) would work but is never tried.

No server-side logs — the connection attempt never reaches LiveKit. No client-side Flutter logs captured (logcat buffer had rotated).

**Root cause:** URL resolver has no reachability check or fallback. It was designed this way intentionally (see task 018 comment in the code — Android 11+ hides VPN interfaces, making Tailscale detection unreliable). But the tradeoff means the LAN path is completely broken when Tailscale is off.

**Workaround:** Enable Tailscale on the phone, or remove `LIVEKIT_URL_TAILSCALE` from `apps/mobile/.env`.

**Possible fix:** Try Tailscale URL first with a short timeout (e.g., 2-3s), fall back to LAN URL on failure.

### BUG-005: Agent not dispatched after worker restart (HIGH)

**First seen:** 18:00:04 UTC
**Frequency:** Every time the voice-agent container restarts while a client is in the room
**Impact:** Client connects to LiveKit and publishes audio, but no agent joins. User sees "agent not present" and has no STT/TTS. Persists across app restarts until LiveKit itself is restarted.

**Sequence of events:**
1. **18:00:01** — Old worker deregistered (container restart for `init: true` change)
2. **18:00:01** — Old agent left room (`CLIENT_REQUEST_LEAVE`)
3. **18:00:02** — LiveKit tried to dispatch terminate job: `"no response from servers"` — no worker available
4. **18:00:04** — New worker registered (`AW_BiY5TWdsAQfz`)
5. **18:00:58** — Client reconnected, LiveKit shows `numParticipants: 0` — no agent dispatched
6. **18:01:26** — Client reconnected again (app restart), still `numParticipants: 0`

**Log excerpt:**
```
livekit-1 | 2026-03-06T18:00:01.044Z  INFO  last worker deregistered
livekit-1 | 2026-03-06T18:00:02.047Z  INFO  failed to send job request  error="no response from servers"
livekit-1 | 2026-03-06T18:00:04.142Z  INFO  worker registered  workerID="AW_BiY5TWdsAQfz"
livekit-1 | 2026-03-06T18:00:58.652Z  INFO  starting RTC session  numParticipants=0
```

**Analysis:** LiveKit dispatches agent jobs reactively — when a participant joins. But the room `fletcher-dev` already existed with a stale job record. The new worker registered, but LiveKit's dispatch system considered the job already handled (even though it failed with "no response from servers"). Subsequent client reconnects joined the existing room, which didn't trigger a fresh dispatch.

This is a fundamental gap in LiveKit's agent dispatch model: **job dispatch is a one-shot event tied to room creation, not a continuous reconciliation loop.** If the dispatch fails or the agent crashes, there's no automatic recovery.

**Root cause:** LiveKit server does not re-dispatch agent jobs for existing rooms when a new worker registers, even if the previous dispatch failed.

**Workaround applied:** Restarted the `livekit` container to clear all room state. Client reconnected to a fresh room and agent was dispatched normally.

**Fix:** [Task 021 — Dynamic Room Names](../../tasks/09-connectivity/021-dynamic-room-names.md) (client-side workaround: create new room on budget exhaustion)

**Related:** [Task 020](../../tasks/09-connectivity/020-agent-reconnect-after-restart.md) (server-side investigation)
