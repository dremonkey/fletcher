## Detailed Entries

### BUG-034: Relay reconnects aggressively in background (LOW)

**First seen:** 08:38 UTC
**Frequency:** Consistent when app is backgrounded
**Impact:** Unnecessary battery drain and log noise while app is in chat mode

**Analysis:** Between 1:42 am and 8:16 am, repeated relay disconnect and reconnect events were observed. The relay currently attempts to maintain a connection regardless of app state. For chat mode, the relay should disconnect when the app moves to the background and only reconnect when it returns to the foreground.

**Status:** OPEN — Requires foreground sensing in the Flutter client to gate the relay connection.

### BUG-027a: session_hold lost to DTLS timeout (HIGH)

**First seen:** 07:17 UTC (room: `raw-hill`)
**Frequency:** Every BUG-027 recovery
**Impact:** Client never receives hold event, UI stays "connected", no recovery UX

**Log excerpt:**
```
[07:17:01] WARN STT watchdog: no STT activity detected — pipeline may be dead
           silenceMs: 33498, timeoutMs: 30000
```

**Analysis:** The watchdog sent `session_hold` at the 30s timeout via `disconnectRoom` callback, but the WebRTC data channels DTLS-timeout ~20s after the audio track unpublish. By the time the watchdog fires, the data channel is dead and the message is silently dropped.

**Fix applied:** Send `session_hold` early at the first 10s check interval while the transport is still alive. The actual disconnect still happens at 30s. Removed duplicate `session_hold` from `disconnectRoom` callback.

**Commit:** `62dff89`

### BUG-027b: Relay participant blocks disconnect handler (HIGH)

**Analysis:** `ParticipantDisconnectedEvent` handler counted all remote participants including the relay. When the agent left, `remaining=1` (relay still present), so the `if (remaining == 0)` guard blocked the entire hold mode flow: no `onAgentDisconnected()`, no status update, no UI event.

### BUG-027c: STT pipeline dies permanently after mic toggle / reconnect (CRITICAL)

**First seen:** 07:30 UTC (rooms: multiple, across 4 consecutive agent jobs)
**Frequency:** Every reconnect after hold timeout
**Impact:** Agent is completely deaf after reconnect — zero STT/TTS, user has no way to recover except killing and restarting the app

**Root cause: `MultiInputStream` pump error on track resubscription**

The Flutter app's mic toggle fully unpublishes the audio track (`removePublishedTrack`) to release Android's `AudioRecord` for keyboard STT (BUG-001 workaround). When the track is removed, the `@livekit/agents` SDK's `MultiInputStream` pump loop encounters a stream reader release error. The SDK silently swallows it via `isStreamReaderReleaseError()`:

```
// SDK internals — error is caught and ignored:
if (isStreamReaderReleaseError(err)) {
  // silently exits the pump loop
}
```

The `AudioRecognition` (STT + VAD) was created during `AgentSession.start()` and never restarts. Once the pump dies, the pipeline is permanently broken for this agent job. Even when the user reconnects and a new agent job is dispatched, the same failure pattern often repeats because the mobile client's initial track publish/subscribe cycle triggers the same `MultiInputStream` issue.

**Evidence across 4 jobs:**
```
Job 837: [07:34:44] ERROR MultiInputStream pump error → watchdog fires at 07:35:13
Job 1067: Zero VAD events in 17s of audio — pipeline never initialized
Job 1266: [07:38:30] ERROR MultiInputStream pump error coinciding with mic toggle
```

**Attempted fixes:**
1. **`TrackUnpublished` handler** — detect when user audio is unpublished and immediately disconnect to trigger redispatch. **Failed:** doesn't help when STT never starts in the first place (job 1067 pattern). Reverted.
2. **Watchdog Mode 2 "never started"** — detect when audio track is subscribed but no STT events arrive within 30s, then trigger recovery. **Partial mitigation:** catches the "never started" case but doesn't fix the root cause. The new agent after recovery may also fail.

**The fundamental problem:** The `@livekit/agents` SDK's audio pipeline (`AudioRecognition`, `MultiInputStream`) is not designed to survive track resubscription. It's created once during `AgentSession.start()` and there's no mechanism to reinitialize it. The `DeferredReadableStream.setSource()` method throws "Stream source already set" if called twice.

**Status:** OPEN — no reliable fix available without SDK changes or a workaround that avoids track unpublish/republish.

**Possible approaches (not yet attempted):**
- Prevent the Flutter app from fully unpublishing the track on mic toggle (mute instead of remove)
- Fork/patch the SDK to handle track resubscription gracefully
- Implement a heartbeat between mobile and agent to detect pipeline death faster

### BUG-027d: STT pipeline sometimes fails to initialize on fresh connect (HIGH)

**First seen:** 07:38 UTC (job 1067)
**Frequency:** Intermittent — seen in ~25% of agent jobs during testing
**Impact:** Agent is deaf from the very start — zero VAD events despite audio track subscribed for 17+ seconds

**Analysis:** In some agent jobs, the VAD/STT pipeline never fires a single event even though the audio track is correctly subscribed. The SDK's `AudioRecognition` appears to fail silently during initialization. No errors are logged — the pipeline simply never produces output.

This may be related to the `DeferredReadableStream` source initialization or timing issues in how the SDK sets up the audio processing chain during `AgentSession.start()`.

**Mitigation applied:** STT watchdog Mode 2 — `onAudioTrackSubscribed()` starts a timer. If no STT events arrive within 30s, the watchdog sends `session_hold` and disconnects, triggering a redispatch. This gives the system another chance to initialize correctly.

**Status:** MITIGATED by watchdog Mode 2 (not yet field-verified). Root cause unknown.

### BUG-035: Network/Room info lost on mid-session room join (LOW)

**First seen:** 15:49 PDT (UTC-7)
**Frequency:** Consistent when joining a new room without exiting the app
**Impact:** UX regression — original connection context (room/network info) disappears from the chat stream when switching rooms.

**Analysis:** When joining a new room mid-session, the new room's "Agent joined" system messages are correctly added to the stream. However, the initial system messages from the start of the app session (which displayed the original room and network metadata) are being removed or overwritten. The stream should be additive, preserving the full history of room transitions and their respective metadata.

**Status:** OPEN — Likely a state management issue in the Flutter client's message stream or a "clear on join" logic that is too aggressive.
