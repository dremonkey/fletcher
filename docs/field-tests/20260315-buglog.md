## Detailed Entries

### BUG-041: `NO_REPLY` response not displayed in UI (LOW)

**First seen:** 00:48 PDT (2026-03-16)
**Frequency:** Consistent when agent chooses `NO_REPLY`
**Impact:** UX confusion — the user sees the "thinking" spinner or simply silence/no update after sending a message that results in an internal `NO_REPLY`.

**Analysis:** When the agent logic returns `NO_REPLY`, it signals that no user-visible response is needed (e.g., acknowledging a routine task or background update). However, the mobile client currently has no visual representation for this state. From the user's perspective, it looks like the agent is still thinking or the message was lost.

**Proposed Fix:** The UI should display a subtle indication that the message was acknowledged but no reply is forthcoming (e.g., a checkmark, a "Silent Acknowledge" status, or a brief system message).

**Status:** OPEN

### BUG-037: Reconnect system messages visible on background disconnect (LOW)

**First seen:** 17:13 PDT
**Frequency:** Consistent when app is backgrounded
**Impact:** UX noise — user sees "disconnected · reconnecting..." messages in the chat stream after resuming the app, which are redundant since the background disconnect was intentional (BUG-034).

**Analysis:** BUG-034 implemented an intentional immediate disconnect when the app is backgrounded in chat mode. However, the LiveKit SDK's `RoomReconnectingEvent` and `RoomDisconnectedEvent` handlers were still emitting system messages to the chat stream. Since the `disconnect()` call is asynchronous and the SDK may briefly attempt recovery before the clean shutdown completes, these "alarming" messages were leaking into the immutable chat history. Additionally, in voice mode, network transitions or idle timeouts while in the background were also cluttering the history with similar messages.

**Fix:** Updated `LiveKitService` to set `_backgroundDisconnected = true` as soon as the app is backgrounded (previously it was only set for chat mode). Added guards to `RoomReconnectingEvent` and `RoomDisconnectedEvent` handlers to suppress system message emission when this flag is set.

**Status:** FIXED — Commit `f6b7d12` (task-082) / `87a9e1d` (suppress bg noise)

### BUG-036: Room closure grace period interfering with voice agent (HIGH)

**First seen:** 16:35 PDT
**Frequency:** N/A (Architecture/Logic issue)
**Impact:** Potential interference with voice agent initialization or session cleanup; unnecessary resource retention.

**Analysis:** The relay previously maintained a 2-minute grace period (idle timeout) for rooms after participants left. This delay meant that if a human rejoined quickly (e.g., during a mic toggle or BUG-027c recovery), they would be re-connected to the *existing* relay bridge and ACP session. If that session or the underlying STT pipeline was in a bad state, the recovery would fail.

**Fix:** Removed the deferred teardown mechanism entirely. The relay now stops the bridge and leaves the room immediately when the human participant departs. This ensures every reconnection starts with a fresh bridge instance and a clean ACP `session/new` call. Epic 25 (Session Restoration) will eventually handle state persistence across these clean restarts.

**Status:** FIXED — Commit `f3d6b71` (refactor)

### BUG-027c: STT pipeline dies permanently after mic toggle / reconnect (CRITICAL)

**First seen:** 07:30 UTC (rooms: multiple, across 4 consecutive agent jobs)
**Frequency:** Every reconnect after hold timeout
**Impact:** Agent is completely deaf after reconnect — zero STT/TTS, user has no way to recover except killing and restarting the app

**Root cause: `MultiInputStream` pump error on track resubscription**

The Flutter app's mic toggle fully unpublishes the audio track (`removePublishedTrack`) to release Android's `AudioRecord` for keyboard STT (BUG-001 workaround). When the track is removed, the `@livekit/agents` SDK's `MultiInputStream` pump loop encounters a stream reader release error. The SDK silently swallows it via `isStreamReaderReleaseError()`:

```
// SDK internals — error is caught and ignored:
if (isStreamReaderReleaseError(err)) {
  // silently exits the pump loop
}
```

The `AudioRecognition` (STT + VAD) was created during `AgentSession.start()` and never restarts. Once the pump dies, the pipeline is permanently broken for this agent job. Even when the user reconnects and a new agent job is dispatched, the same failure pattern often repeats because the mobile client's initial track publish/subscribe cycle triggers the same `MultiInputStream` issue.

**Evidence across 4 jobs:**
```
Job 837: [07:34:44] ERROR MultiInputStream pump error → watchdog fires at 07:35:13
Job 1067: Zero VAD events in 17s of audio — pipeline never initialized
Job 1266: [07:38:30] ERROR MultiInputStream pump error coinciding with mic toggle
```

**Attempted fixes:**
1. **`TrackUnpublished` handler** — detect when user audio is unpublished and immediately disconnect to trigger redispatch. **Failed:** doesn't help when STT never starts in the first place (job 1067 pattern). Reverted.
2. **Watchdog Mode 2 "never started"** — detect when audio track is subscribed but no STT events arrive within 30s, then trigger recovery. **Partial mitigation:** catches the "never started" case but doesn't fix the root cause. The new agent after recovery may also fail.

**The fundamental problem:** The `@livekit/agents` SDK's audio pipeline (`AudioRecognition`, `MultiInputStream`) is not designed to survive track resubscription. It's created once during `AgentSession.start()` and there's no mechanism to reinitialize it. The `DeferredReadableStream.setSource()` method throws "Stream source already set" if called twice.

**Status:** OPEN — no reliable fix available without SDK changes or a workaround that avoids track unpublish/republish.

**Possible approaches (not yet attempted):**
- Prevent the Flutter app from fully unpublishing the track on mic toggle (mute instead of remove)
- Fork/patch the SDK to handle track resubscription gracefully
- Implement a heartbeat between mobile and agent to detect pipeline death faster

### BUG-027d: STT pipeline sometimes fails to initialize on fresh connect (HIGH)

**First seen:** 07:38 UTC (job 1067)
**Frequency:** Intermittent — seen in ~25% of agent jobs during testing
**Impact:** Agent is deaf from the very start — zero VAD events despite audio track subscribed for 17+ seconds

**Analysis:** In some agent jobs, the VAD/STT pipeline never fires a single event even though the audio track is correctly subscribed. The SDK's `AudioRecognition` appears to fail silently during initialization. No errors are logged — the pipeline simply never produces output.

This may be related to the `DeferredReadableStream` source initialization or timing issues in how the SDK sets up the audio processing chain during `AgentSession.start()`.

**Mitigation applied:** STT watchdog Mode 2 — `onAudioTrackSubscribed()` starts a timer. If no STT events arrive within 30s, the watchdog sends `session_hold` and disconnects, triggering a redispatch. This gives the system another chance to initialize correctly.

**Status:** MITIGATED by watchdog Mode 2 (not yet field-verified). Root cause unknown.

### BUG-035: Network/Room info lost on mid-session room join (LOW)

**First seen:** 15:49 PDT (UTC-7)
**Frequency:** Consistent when joining a new room without exiting the app
**Impact:** UX regression — original connection context (room/network info) disappears from the chat stream when switching rooms.

**Analysis:** When joining a new room mid-session, the new room's "Agent joined" system messages are correctly added to the stream. However, the initial system messages from the start of the app session (which displayed the original room and network metadata) are being removed or overwritten.

**Refined Hypothesis:** The preamble metadata (room/network info) may be treated as a live reference to the current room state in the Flutter client rather than a point-in-time snapshot. When the room changes, the reference updates or the previous state is discarded, causing the "snapshot" of the original connection to disappear from the stream. The stream should be additive and immutable for past events.

**Status:** OPEN — Likely a state management issue in the Flutter client's message stream or a "clear on join" logic that is too aggressive.

### UX Strategy: Session Continuity vs. Room Transitions

**Observation (15:57 PDT):** The current room-switching behavior provides a "fresh start" which clears context (advantage for privacy/clean slate, but disadvantage for conversation flow). 

**Strategic Direction:**
- **Separate Room from Session:** We should move toward a model where "resuming a conversation" is the norm. A room change shouldn't necessarily force a session wipe.
- **Continuous Stream:** The UI should feel like one continuous stream even if the underlying infrastructure (rooms/backends) shifts.
- **Context Management:** We need to explore the UX of how/when context is explicitly cleared vs. when it is seamlessly carried over into a new room.

### BUG-038: Think-tag parser false-positives on literal `<think>` mentions in prose (HIGH)

**First seen:** 22:04 UTC
**Frequency:** Every time the agent's response text contains a literal `<think>` string (e.g., discussing the feature)
**Impact:** Response text is completely swallowed — user sees only "thinking..." spinner with no visible response. The entire 2776-char agent message disappears.

**Log excerpt:**
```
[RelayChatService] TAGS in chunk: think=true final=false
  text="Smart Interaction" layer (Slash commands and `<think>` tag parsing), but the **core engine...
[LiveKitService] FINAL relay text (2776 chars): "Epic 25: Session Resumption — Pulse Check**...
[ChatTranscript] parseAgentText: thinkingState=ThinkingState.inProgress thinking="` tag parsing)..." visible="" rawLen=2776
```

The `[ChatTranscript]` line repeats every ~100ms for 100+ frames — parser is stuck at `inProgress` forever with `visible=""`.

**Analysis:** `parseAgentText()` in `agent_text_parser.dart:61` uses `raw.indexOf('<think>')` which matches `<think>` **anywhere** in the string. When the agent discusses the `<think>` feature in prose (e.g., "Slash commands and `<think>` tag parsing"), the parser:

1. Finds `<think>` mid-text at the literal mention (position ~150 of 2776)
2. Looks for `</think>` after that position — never finds it (not a real tag)
3. Returns `thinkingState: inProgress`, `visible: ""`
4. All response content (before and after the mention) is completely swallowed

The real OpenClaw tag format puts `<think>` at position 0 of the response. The parser has no anchor check.

**Root cause:** Parser does not require `<think>` to appear at the start of the message. It uses `indexOf()` which matches anywhere, including literal mentions in prose.

**Fix:** Require `<think>` to appear at position 0 (or after only leading whitespace) to be treated as a real tag. If found later in the text, treat as literal text with `thinkingState: none`.

**File:** `apps/mobile/lib/utils/agent_text_parser.dart:61`

**Fix:** Anchored `<think>` and standalone `<final>` detection to position 0 (after optional leading whitespace). `raw.indexOf('<think>')` replaced with `trimmed.startsWith('<think>') ? raw.indexOf('<think>') : -1`. Same for standalone `<final>`. 5 new regression tests added (34 total, all passing).

**Status:** FIXED
