## Executive Summary

Phase 1 delivers a **minimal viable product** for Fletcher that prioritizes **reliability, latency, and cost efficiency** over full voice capabilities. The primary goal is to enable early field testing of the core conversation loop and sovereign pairing workflow **without the overhead and failure modes of cloud TTS**.

This spec defines the technical requirements for:

1. **Two-Step Onboarding** — QR-based Vessel Key pairing with Ed25519 device registration
2. **Text-only Mode** — Skip TTS generation and deliver agent responses as text only
3. **Managed Connection** — Use Hub address from QR to authenticate and join LiveKit room
4. **Feedback Loop Fix** — Ensure STT/Transcript feedback via LiveKit data channel (addresses BUG-030)
5. **⚠️ Reliability Redline (NEW)** — Mandatory acknowledgement chimes, disconnect alerts, and connection status UI to prevent "talking to yourself" failure mode

## 1. Two-Step Onboarding

### 1.1 Overview

The **Two-Step Onboarding** flow replaces the current manual URL entry with a secure, proximity-based pairing using a **Vessel Key**. This approach ensures:

- **Zero manual entry** — No typing of URLs, tokens, or credentials
- **Physical proximity** — Pairing requires physical access (QR scan)
- **Automatic registration** — Device keys are generated and registered transparently

### 1.2 Vessel Key Structure

The Vessel Key is a JSON payload generated by the Hub and encoded as a QR code. See [vessel-key-pairing-spec.md](./vessel-key-pairing-spec.md) for the complete payload schema.

**Required fields for Phase 1:**

```json
{
  "version": "1.0",
  "hubIdentity": {
    "name": "Toch's Heirloom Hub",
    "deviceId": "hub_a1b2c3d4e5f6"
  },
  "network": {
    "gatewayUrl": "https://heirloom.local:8443",
    "tailscaleIp": "100.64.1.42"
  },
  "auth": {
    "pairingToken": "pair_7d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a",
    "tokenExpiry": 1709850600,
    "hubPublicKey": "ed25519:A1B2C3D4E5F6..."
  },
  "services": {
    "livekit": {
      "url": "wss://heirloom.local:7880"
    }
  }
}
```

```dart
Future<VesselKey> _parseVesselKey(String rawData) async {
  try {
    final json = jsonDecode(rawData);
    
    // Version check
    if (json['version'] != '1.0') {
      throw PairingException('Unsupported Vessel Key version');
    }
    
    // Expiry check
    final expiry = json['auth']['tokenExpiry'] as int;
    if (DateTime.now().millisecondsSinceEpoch / 1000 > expiry) {
      throw PairingException('Pairing token expired (15 min limit)');
    }
    
    return VesselKey.fromJson(json);
  } catch (e) {
    throw PairingException('Invalid Vessel Key: $e');
  }
}
```

```typescript
// POST /fletcher/devices/register
// Body: { publicKey, deviceModel, os, appVersion }
// Auth: Bearer token (pairing token, single-use)

async function registerDevice(req, res) {
  const token = req.headers.authorization?.replace('Bearer ', '');
  
  // Validate pairing token (check expiry, single-use flag)
  const session = await validatePairingToken(token);
  if (!session) {
    return res.status(401).json({ error: 'Invalid or expired pairing token' });
  }
  
  const { publicKey } = req.body;
  const deviceId = `device_${randomBytes(8).toString('hex')}`;
  
  // Store device identity
  await deviceStore.create({
    deviceId,
    publicKey,
    hubId: session.hubId,
    createdAt: Date.now(),
  });
  
  // Revoke pairing token (single-use)
  await pairingTokenStore.revoke(token);
  
  return res.status(201).json({ deviceId });
}
```

## 2. Text-only Mode

### 2.1 Overview

Text-only mode **disables TTS synthesis** to eliminate:

- **TTS failures** — Cloud provider 402/429 errors, network timeouts
- **Latency** — 200-400ms TTS TTFB adds to perceived response time
- **Cost** — ElevenLabs/Google TTS API charges

Agent responses arrive as **transcription text only** via the LiveKit data channel. STT remains active (user speech is still transcribed).

### 2.2 Design Goals

1. **Lower latency** — Skip TTS TTFB and PCM frame buffering
2. **Lower cost** — Eliminate TTS API charges
3. **Higher reliability** — Remove TTS as a failure mode
4. **Preserved feedback** — User still sees real-time transcription of agent responses

### 2.3 Implementation Strategy

#### Mobile Client: TTS Toggle

The mobile app controls TTS mode via a persistent preference and sends the current state to the agent on room connect.

**UI Control:**
```dart
// In settings_screen.dart or diagnostics_panel.dart
SwitchListTile(
  title: Text('Voice Responses'),
  subtitle: Text('Enable agent speech (requires TTS quota)'),
  value: _ttsEnabled,
  onChanged: (enabled) async {
    setState(() => _ttsEnabled = enabled);
    
    // Persist preference
    final prefs = await SharedPreferences.getInstance();
    await prefs.setBool('tts_enabled', enabled);
    
    // Send to agent (if connected)
    if (_roomConnected) {
      await _liveKitService.setTtsMode(enabled ? 'on' : 'off');
    }
  },
)
```

**Data Channel Message:**
```typescript
// Sent from mobile app to voice agent
interface TtsModeEvent {
  type: 'tts-mode';
  value: 'on' | 'off';
}
```

**Send on Connect:**
```dart
// In LiveKitService.dart
Future<void> connectToRoom(String url, String token) async {
  // ... existing room connect logic ...
  
  await room.connect(url, token);
  
  // Send TTS preference immediately after connect
  final prefs = await SharedPreferences.getInstance();
  final ttsEnabled = prefs.getBool('tts_enabled') ?? true;
  await setTtsMode(ttsEnabled ? 'on' : 'off');
}

Future<void> setTtsMode(String mode) async {
  final event = jsonEncode({'type': 'tts-mode', 'value': mode});
  await room.localParticipant?.publishData(
    utf8.encode(event),
    topic: 'ganglia-events',
    options: DataPublishOptions(reliable: true),
  );
}
```

#### Voice Agent: TTS Control

The voice agent listens for `tts-mode` events and calls the LiveKit SDK's built-in audio output control:

```typescript
// apps/voice-agent/src/agent.ts

// Store current TTS state
let ttsEnabled = true;

// Listen for control events from mobile app
room.localParticipant.on('dataReceived', (payload, participant, _, topic) => {
  if (topic !== 'ganglia-events') return;
  
  try {
    const event = JSON.parse(new TextDecoder().decode(payload));
    
    if (event.type === 'tts-mode') {
      const enabled = event.value === 'on';
      logger.info(`TTS mode changed: ${enabled ? 'on' : 'off'}`);
      
      // Native LiveKit SDK method (available in @livekit/agents 1.0.48+)
      session.output.setAudioEnabled(enabled);
      ttsEnabled = enabled;
    }
  } catch (err) {
    logger.error('Failed to parse control event', err);
  }
});
```

**SDK Behavior:**
- `setAudioEnabled(true)` — Normal operation (TTS runs, audio published)
- `setAudioEnabled(false)` — **TTS skipped entirely** (no API calls, no audio track)

**Crucially:** Transcription of agent responses **still happens** via the data channel (see Section 4).

#### Acknowledgment Sound Behavior

**⚠️ RELIABILITY REDLINE UPDATE (2026-03-08):**

The acknowledgement chime is **MANDATORY** on all voice-in turns, **regardless of TTS state**. This is a critical reliability requirement to prevent silent session drops.

**Rationale:** When TTS is OFF, the acknowledgement chime is the ONLY audio feedback confirming the system received the user's input. Without it, users cannot distinguish between:
- "The system is processing my request" (correct)
- "The connection silently dropped and I'm talking to myself" (BUG-030 failure mode)

**Implementation:**

```typescript
// In agent.ts, when handling EOU (end-of-utterance)
session.on('stateChanged', (state) => {
  if (state === 'thinking') {
    // ALWAYS play ack chime, even when TTS is OFF
    // This is the user's ONLY confirmation that the session is alive
    bgAudioPlayer.play();
  }
  
  if (state === 'speaking' || state === 'listening') {
    playHandle?.stop();
  }
});
```

**UX Impact:**
- **With TTS ON:** User speaks → Ack chime → Voice response
- **With TTS OFF:** User speaks → Ack chime → Text-only response
- **No audio at all** = Connection is severed (see Disconnect Alert Sound below)

### 2.4 User Experience

**With TTS enabled (default):**
1. User speaks → STT transcribes (shown in subtitle)
2. Agent thinks → Ack chime loops
3. Agent responds → TTS plays audio + transcript appears

**With TTS disabled:**
1. User speaks → STT transcribes (shown in subtitle)
2. Agent thinks → No ack chime (silent)
3. Agent responds → **Text-only transcript** appears (no audio)

## 2.6 Disconnect Alert Sound

**⚠️ RELIABILITY REDLINE UPDATE (2026-03-08):**

### 2.6.1 Overview

A **Disconnect Alert Sound** is a high-priority audio cue that plays **immediately** when the LiveKit connection is lost. This prevents the "talking to yourself" failure mode where the user continues speaking after the session has silently dropped.

**Design Goals:**
1. **Instant feedback** — Alert plays within 100ms of disconnect detection
2. **Unmistakable** — Distinct from acknowledgement chime (e.g., error tone, buzzer)
3. **Cuts through** — Plays at high priority, interrupting any background audio
4. **Non-blocking** — Does not prevent reconnect attempts

### 2.6.2 Audio Asset

**Format:** Short (0.3-0.5s), attention-grabbing tone  
**Suggested Assets:**
- iOS System Sound: `1006` (Tock, error tone)
- Android: `notification_sound_default.ogg` (system alert)
- Custom: Three-tone descending alert (440Hz → 330Hz → 220Hz, 150ms each)

**Fallback:** If no audio asset is available, trigger device haptic feedback (strong vibration pattern).

### 2.6.3 Mobile Implementation

**Listen for Disconnect Events:**

```dart
// In LiveKitService.dart
room.addListener(() {
  final connectionState = room.connectionState;
  
  if (connectionState == ConnectionState.disconnected) {
    // Play disconnect alert immediately
    _playDisconnectAlert();
    
    // Update UI state
    _connectionBloc.add(ConnectionLost(timestamp: DateTime.now()));
    
    // Attempt reconnect (existing logic)
    _attemptReconnect();
  }
});

Future<void> _playDisconnectAlert() async {
  // iOS: Use system sound
  if (Platform.isIOS) {
    await SystemSound.play(SystemSoundType.alert);
  }
  
  // Android: Use notification sound
  if (Platform.isAndroid) {
    final player = AudioPlayer();
    await player.play(AssetSource('sounds/disconnect_alert.ogg'));
  }
  
  // Fallback: Haptic feedback
  HapticFeedback.heavyImpact();
}
```

**Dependencies:**
- `audioplayers: ^6.0.0` — For custom alert sound playback
- Built-in `HapticFeedback` — For vibration fallback

### 2.6.4 Edge Cases

| Scenario | Behavior |
|----------|----------|
| **Network blip (&lt;2s)** | Alert plays, reconnect succeeds, user sees "Reconnected" toast |
| **Extended outage (&gt;10s)** | Alert plays once (do not loop), UI shows "Connection Lost" banner |
| **User manually disconnects** | No alert (expected behavior) |
| **App backgrounded** | Alert deferred until app returns to foreground |

### 2.6.5 Testing Checklist

- [ ] Simulate network disconnect (airplane mode), verify alert plays immediately
- [ ] Verify alert does NOT play on manual disconnect (user taps "End Session")
- [ ] Test with app in background, verify alert plays on foreground return
- [ ] Verify haptic fallback works when audio is unavailable
- [ ] Measure alert latency (target: &lt;100ms from disconnect detection)

## 3. Managed Connection

### 3.1 Overview

The mobile client uses the **Hub's address** (from the Vessel Key) to:

1. **Authenticate** — Request a LiveKit room token from the Hub
2. **Join LiveKit room** — Use the token to connect to the Hub's LiveKit server

This replaces the current approach where the mobile app generates its own token or uses a hardcoded token server.

### 3.2 Authentication Flow

#### Step 1: Load Hub Credentials

```dart
// On app launch (after pairing check)
final storage = FlutterSecureStorage();
final hubUrl = await storage.read(key: 'hub_gateway_url');
final deviceId = await storage.read(key: 'device_id');
final privateKeyBase64 = await storage.read(key: 'device_private_key');
```

#### Step 2: Request Room Token

```dart
// lib/services/hub_auth_service.dart
Future<String> requestRoomToken(String roomName) async {
  // Generate Ed25519 signature for request
  final timestamp = DateTime.now().millisecondsSinceEpoch;
  final challenge = '$deviceId:$roomName:$timestamp';
  
  final algorithm = Ed25519();
  final keyPair = await algorithm.newKeyPairFromSeed(
    base64Decode(privateKeyBase64!),
  );
  final signature = await algorithm.sign(
    utf8.encode(challenge),
    keyPair: keyPair,
  );
  
  // Request token from Hub
  final response = await http.post(
    Uri.parse('$hubUrl/fletcher/rooms/join'),
    headers: {'Content-Type': 'application/json'},
    body: jsonEncode({
      'deviceId': deviceId,
      'roomName': roomName,
      'timestamp': timestamp,
      'signature': base64Encode(signature.bytes),
    }),
  );
  
  if (response.statusCode != 200) {
    throw AuthException('Token request failed: ${response.body}');
  }
  
  return jsonDecode(response.body)['token'];
}
```

**Hub-Side Endpoint:**

```typescript
// POST /fletcher/rooms/join
// Body: { deviceId, roomName, timestamp, signature }

async function joinRoom(req, res) {
  const { deviceId, roomName, timestamp, signature } = req.body;
  
  // Verify signature
  const device = await deviceStore.get(deviceId);
  if (!device) {
    return res.status(404).json({ error: 'Device not found' });
  }
  
  const challenge = `${deviceId}:${roomName}:${timestamp}`;
  const publicKey = base64ToBytes(device.publicKey);
  const signatureBytes = base64ToBytes(signature);
  
  const valid = await ed25519.verify(
    signatureBytes,
    Buffer.from(challenge),
    publicKey,
  );
  
  if (!valid) {
    return res.status(401).json({ error: 'Invalid signature' });
  }
  
  // Check timestamp (prevent replay attacks)
  const now = Date.now();
  if (Math.abs(now - timestamp) > 60_000) {
    return res.status(401).json({ error: 'Timestamp expired' });
  }
  
  // Generate LiveKit token
  const token = new AccessToken(
    process.env.LIVEKIT_API_KEY,
    process.env.LIVEKIT_API_SECRET,
    {
      identity: deviceId,
      metadata: JSON.stringify({ roomName }),
    },
  );
  
  token.addGrant({
    room: roomName,
    roomJoin: true,
    canPublish: true,
    canSubscribe: true,
  });
  
  return res.json({ token: await token.toJwt() });
}
```

**Dependencies (Hub):**
- `livekit-server-sdk` — AccessToken generation
- `@noble/ed25519` — Signature verification

#### Step 3: Connect to LiveKit Room

## 4. Feedback Loop Fix (BUG-030)

### 4.1 Problem Statement

**BUG-030 ("Unidirectional Blackout")** occurs when the user loses visual feedback of agent responses during network disruptions or TTS failures. Symptoms:

- Agent is thinking/responding (visible in logs)
- User hears no audio (TTS timeout or disabled)
- User sees no text (transcript not delivered or cleared prematurely)
- **Result:** User thinks the system is broken and abandons the conversation

### 4.2 Root Cause

The current transcript delivery pipeline relies on **LiveKit text streams** (`lk.transcription` topic), which:

1. Auto-clear after 3 seconds (by design for subtitle UX)
2. Are not buffered during reconnects
3. May not arrive if the data channel is congested

### 4.3 Solution: Dual-Channel Transcript Delivery

Deliver agent transcripts via **two channels**:

1. **LiveKit Text Streams** (`lk.transcription`) — Real-time subtitles (existing)
2. **Data Channel** (`ganglia-events`) — Persistent transcript log

The mobile app treats data channel transcripts as the **source of truth** and renders them in the main conversation view. Text streams are used only for ephemeral subtitles.

### 4.4 Implementation

#### Voice Agent: Emit Transcript Events

```typescript
// packages/livekit-agent-ganglia/src/llm.ts

// When LLM stream emits content
for await (const chunk of stream) {
  if (chunk.type === 'content') {
    // Existing: yield to TTS pipeline
    yield chunk;
    
    // NEW: Also emit as transcript event
    const transcriptEvent = {
      type: 'transcript',
      role: 'agent',
      content: chunk.delta,
      isFinal: false,
      timestamp: Date.now(),
    };
    
    await publishData(
      JSON.stringify(transcriptEvent),
      { topic: 'ganglia-events', reliable: true },
    );
  }
}

// On stream end, emit final transcript
const transcriptFinal = {
  type: 'transcript',
  role: 'agent',
  content: fullResponse,
  isFinal: true,
  timestamp: Date.now(),
};

await publishData(
  JSON.stringify(transcriptFinal),
  { topic: 'ganglia-events', reliable: true },
);
```

**Event Schema:**

```typescript
interface TranscriptEvent {
  type: 'transcript';
  role: 'user' | 'agent';
  content: string;        // Full text (for final) or delta (for interim)
  isFinal: boolean;
  timestamp: number;      // Unix timestamp (ms)
}
```

#### Mobile Client: Render Data Channel Transcripts

```dart
// In LiveKitService.dart

room.on<DataReceivedEvent>((event) {
  if (event.topic != 'ganglia-events') return;
  
  final data = utf8.decode(event.data);
  final json = jsonDecode(data);
  
  if (json['type'] == 'transcript') {
    _handleTranscriptEvent(
      role: json['role'],
      content: json['content'],
      isFinal: json['isFinal'],
      timestamp: json['timestamp'],
    );
  }
});

void _handleTranscriptEvent({
  required String role,
  required String content,
  required bool isFinal,
  required int timestamp,
}) {
  // Update conversation state (persistent log)
  conversationBloc.add(TranscriptReceived(
    role: role == 'user' ? Role.user : Role.agent,
    text: content,
    isFinal: isFinal,
    timestamp: DateTime.fromMillisecondsSinceEpoch(timestamp),
  ));
}
```

**Interim vs Final:**
- **Interim messages** (isFinal: false) — Rendered with a subtle "typing..." indicator
- **Final messages** (isFinal: true) — Rendered as solid text

### 4.5 Comparison: Text Streams vs Data Channel

| Feature | Text Streams (`lk.transcription`) | Data Channel (`ganglia-events`) |
|---------|-----------------------------------|----------------------------------|
| **Purpose** | Real-time subtitles | Persistent conversation log |
| **Delivery** | Best-effort (may drop) | Reliable (TCP-like) |
| **Lifetime** | Auto-clear after 3s | Persists in app state |
| **Reconnect** | Not buffered | Can be replayed by agent |
| **UI Binding** | Subtitle overlay | Main conversation view |

## 6. Success Criteria

Phase 1 is considered **complete** when:

1. **Onboarding:** A fresh install can pair with a Hub by scanning a QR code (no manual entry)
2. **Connection:** The app can authenticate and join a LiveKit room using Hub-issued tokens
3. **Text Mode:** The app can toggle TTS on/off, with agent responses delivered as text when TTS is off
4. **Reliability:** Agent responses appear in the conversation log **even when TTS fails or is disabled**
5. **Persistence:** Conversation transcripts survive app restarts (local SQLite storage)
6. **⚠️ RELIABILITY REDLINE (2026-03-08):**
   - **Acknowledgement chime plays on EVERY voice-in turn**, regardless of TTS state
   - **Disconnect alert sound plays within 100ms** of connection loss
   - **Status bar visually indicates disconnect** (blinking red "DISCONNECTED" badge)
   - **User can never talk to themselves for >5 seconds** without realizing the session dropped

**Non-Goals (deferred to Phase 2):**
- Voice fingerprinting (Epic 6)
- Wake word detection (Epic 5)
- Multi-device support
- Background listening
